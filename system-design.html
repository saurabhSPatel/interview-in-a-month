Fast Understanding Flows

The 7 Flows

  • Read flow: edge -> cache -> source of truth
  • Write flow: validate -> commit -> invalidate -> fan-out
  • Realtime flow: connect -> publish -> deliver -> replay
  • Search flow: write -> CDC -> index -> rank -> return
  • Media flow: ingest -> process -> store -> distribute
  • Analytics flow: event -> stream/batch -> aggregate -> serve
  • Failure flow: timeout -> retry -> circuit -> fallback

10 Questions For Any Design

  • What is the exact write path?
  • What is the exact read path?
  • What is source of truth per entity?
  • Where can data be stale?
  • Where can duplicates happen?
  • What is synchronous vs asynchronous?
  • Where will backpressure appear first?
  • How is partitioning done?
  • What is retry + idempotency strategy?
  • What metrics define health?

12 Reusable System Design Patterns

Pattern 01: Read-Heavy Caching

When to use

Read volume is high and data can tolerate bounded staleness.

Core design

  • Cache-aside with bounded TTL
  • Read replicas for source-of-truth DB
  • Hot-key mitigation with replication/local cache

Example scenario

  • Product catalog pages, user profile reads, or feed retrieval where read latency matters more than strict freshness.

Decision Flow

  • Check read/write ratio (>10:1 favors caching)
  • Assess staleness tolerance (seconds/minutes ok?)
  • Evaluate cache hit rate potential
  • Choose cache type: in-memory (Redis), CDN, or local

Trade-off Analysis

  • Pros: Dramatic latency reduction, reduced DB load
  • Cons: Cache invalidation complexity, potential stale data issues
  • Mitigations: TTL, write-through, cache-aside patterns

Trade-off

Latency drops dramatically but invalidation complexity rises.

Pattern 02: Write-Heavy Log First

When to use

Ingestion is bursty and downstream processing can be async.

Core design

  • Write-ahead event log (Kafka/Pulsar)
  • Consumer groups for fan-out workloads
  • Dead-letter + replay controls

Trade-off

Higher eventual consistency complexity, much better durability and burst absorption.

Pattern 03: Realtime Push

When to use

Client state must update instantly (chat/presence/live ops).

Core design

  • WebSocket/SSE gateway tier
  • Pub-sub backbone
  • Reconnect + offset replay

Trade-off

Connection state management becomes primary operational challenge.

Pattern 04: Global Distribution

When to use

Worldwide latency and edge delivery matter.

Core design

  • CDN + GeoDNS
  • Regional compute, local cache
  • Data residency strategy per region

Trade-off

Lower user latency, higher consistency and governance complexity.

Pattern 05: Distributed Transaction via Saga

When to use

Business operation spans multiple services and DBs.

Core design

  • Local commit per service
  • Compensating actions for rollback
  • Orchestrated or choreographed coordination

Trade-off

No global lock, but compensation correctness is critical.

Pattern 06: Outbox + CDC

When to use

Need reliable DB commit plus event publication without dual-write loss.

Core design

  • Write domain row + outbox row in same transaction
  • CDC publishes outbox rows to event bus
  • Idempotent consumers downstream

Trade-off

More moving parts, removes silent event loss class.

Pattern 07: Search-Optimized Replica

When to use

Need full-text and faceted retrieval beyond OLTP capability.

Core design

  • Primary DB for truth
  • Search index sync via CDC/events
  • Query ranking + filtering pipeline

Trade-off

Search freshness lags source slightly; retrieval speed improves massively.

Pattern 08: Geo-Spatial Matching

When to use

Proximity and dynamic locations drive matching.

Core design

  • Geohash/quadkey indexing
  • Nearby candidate fetch + score ranking
  • Frequent location updates with bounded staleness

Trade-off

Accuracy and freshness compete with update throughput.

Pattern 09: API Protection Layer

When to use

Public APIs or shared gateways need abuse resistance.

Core design

  • Token bucket/sliding window limits
  • Tier-aware quotas
  • Circuit protection and graceful backpressure

Trade-off

Higher reliability but extra state and policy surface.

Pattern 10: Ordered ID Generation

When to use

Need scalable sortable unique IDs without central bottleneck.

Core design

  • Snowflake/UUIDv7 identifiers
  • Shard-aware uniqueness
  • Clock skew handling

Trade-off

Simplifies partitioning and pagination; requires clock discipline.

Pattern 11: CQRS Read/Write Split

When to use

Read model and write model have very different shapes.

Core design

  • Command model for writes
  • Projection pipeline for read views
  • Event-driven denormalized stores

Trade-off

Read speed and flexibility improve, consistency becomes eventual by design.

Pattern 12: Resilience Envelope

When to use

Any distributed call graph with partial failure risk.

Core design

  • Timeout budget per hop
  • Retry policy with jitter and cap
  • Circuit breaker + bulkhead isolation

Trade-off

Protects system under failure but requires careful calibration to avoid retry storms.

45-Minute Interview Playbook

Execution Timeline

  • 0-5 min: scope and constraints
  • 5-10 min: rough sizing and bottleneck hypothesis
  • 10-20 min: HLD with APIs and core data model
  • 20-35 min: deep dive hardest component
  • 35-45 min: failure modes, trade-offs, evolution path

Staff-Level Signals

  • Name assumptions early and validate them
  • Separate correctness from latency concerns
  • State consistency boundaries explicitly
  • Mention idempotency before retries
  • Close with observability and rollout strategy
