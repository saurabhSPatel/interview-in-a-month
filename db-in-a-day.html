01

B-Tree vs LSM-Tree

The choice between B-tree and LSM-tree storage engines is not about preference — it's about your workload's read/write ratio and latency requirements.

B-Tree (PostgreSQL, MySQL, InnoDB): self-balancing, O(log n) reads and writes. Pages are typically 8KB, sized to match disk block size. In-place updates make it efficient for read-heavy workloads where data is frequently read back immediately after writes. Write amplification comes from updating internal tree nodes on every write.

LSM-Tree (Cassandra, RocksDB, LevelDB): writes go to an in-memory MemTable first — sequential, O(1) cost. MemTable is periodically flushed to an immutable SSTable on disk. Background compaction merges SSTables, removes tombstones, maintains sort order. Bloom filters per SSTable allow skipping irrelevant SSTables on read. The result: dramatically better write throughput at the cost of read amplification (multiple SSTables may need to be checked) and compaction I/O overhead.

Decision Rule

Write-heavy (>70% writes), time-series, audit logs, IoT telemetry: LSM-tree. Read-heavy, relational data, OLTP with complex queries, joins: B-tree. Most production systems use both — PostgreSQL for relational OLTP, Cassandra or RocksDB for high-throughput append-heavy workloads.

02

MVCC and WAL — How PostgreSQL Actually Works

MVCC (Multi-Version Concurrency Control): every write creates a new row version with a transaction timestamp rather than updating in-place. Readers see a snapshot of the database as of their transaction's start time. This means readers and writers never block each other — a long-running SELECT doesn't hold locks that block INSERTs.

The cost: old row versions accumulate and must be cleaned up by the VACUUM process. A long-running read transaction prevents VACUUM from reclaiming space, causing table bloat. This is why monitoring transaction age matters in PostgreSQL production systems.

WAL (Write-Ahead Log): every change is written to the WAL sequentially before it's applied to data pages. On crash, the database replays the WAL from the last checkpoint. WAL also enables streaming replication — replicas receive WAL segments and apply them to stay in sync. Debezium reads the WAL for CDC (change data capture), enabling the Outbox pattern.

03

Sharding and Consistent Hashing

Sharding becomes necessary when a single database instance can no longer handle write throughput or when the dataset exceeds single-node capacity. The choice of shard key is the most critical decision in the entire sharding design.

Consistent hashing distributes keys across nodes on a hash ring (0 to 2^32). Adding a node moves only k/n keys on average. Removing a node redistributes only that node's keys to the next node clockwise. Virtual nodes (vnodes) — 100-200 per physical node — ensure even distribution even with heterogeneous hardware. Used by DynamoDB, Cassandra, Redis Cluster, CDN cache routing.

Hotspot prevention: avoid sequential shard keys (auto-increment IDs put all writes on one shard). For social systems, celebrity users cause hot shards on user_id hashing — handle with separate read paths for high-follower accounts. For time-series, composite keys (time bucket + device_id) distribute writes across shards.

04

Redis Data Structures — Beyond String Cache

Most engineers use Redis as a simple key-value cache. That misses the data structures that make Redis genuinely powerful for specific access patterns.

Sorted Set (ZSET): elements scored by a float. O(log n) insert and range query. Perfect for leaderboards, feed ordering, delayed job queues (score = execution timestamp). ZRANGEBYSCORE retrieves elements in score range — this is how you implement a sliding window rate limiter without scanning all elements.

HyperLogLog: probabilistic cardinality estimation. Counts unique items using ~12KB regardless of how many billions of items you've seen. Error rate ~0.81%. Use for unique visitor counts, unique IP tracking, distinct user counts in analytics — anywhere you need scale without exact precision.

Streams: durable append-only log with consumer groups. Lighter-weight Kafka alternative. Each entry has a unique ID (timestamp-sequence). Consumer groups allow multiple consumers to process entries in parallel. On crash, a consumer can re-read from its last acknowledged ID.

05

Flash Cards

LSM-tree vs B-tree: when does LSM win?
Write-heavy workloads where write throughput matters more than read latency. Time-series, IoT telemetry, audit logs. The sequential MemTable write and compaction amortise over time. B-tree wins for read-heavy OLTP with complex queries and joins.
Why does MVCC enable non-blocking reads?
Each transaction sees a snapshot of the database at its start time. Writers create new row versions rather than updating in-place. Readers never need to wait for writers to finish — they see the version that was current when their transaction started.
w=2, r=2, n=3 — strongly consistent?
Yes. w + r = 4 > n = 3, so at least one node in the write quorum overlaps with the read quorum. Any read is guaranteed to see the latest write. w=1, r=1, n=3 is not strongly consistent — only eventual consistency.
What is the N+1 query problem?
One query fetches a list of N items. Then N more queries fetch related data for each item. Fix: JOIN the related data in the original query, or batch-load related data by IDs. ORM lazy-loading is the most common cause — always check generated SQL.
