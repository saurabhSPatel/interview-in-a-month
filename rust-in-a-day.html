01Key Concepts

Ownership Model — Deep

  • Move semantics: value has one owner, move = transfer
  • Copy types: stack-only, no destructor (i32, bool, char, tuples of Copy)
  • Clone: explicit deep copy (expensive — allocation)
  • Drop: deterministic destructor (RAII), called at scope end
  • Drop order: reverse declaration order, fields before struct

Borrow Checker — Invariants

  • At any time: ONE mutable ref OR N immutable refs
  • Refs must not outlive referent (lifetime check)
  • No aliased mutable state (XOR mutability)
  • This prevents: data races, use-after-free, dangling pointers
  • Enforced at compile time: zero runtime overhead

Lifetimes — Not Magic

  • Lifetime = scope for which reference is valid
  • Elision: compiler infers in simple cases (single input ref)
  • 'a on struct: struct cannot outlive ref it holds
  • Static lifetime: entire program duration (global data)
  • Higher-ranked: for<'a> Fn(&'a str) — any lifetime

Async Rust — Conceptual

  • Future: lazy, zero-cost state machine at compile time
  • async fn: desugars to impl Future
  • Executor runs futures: Tokio, async-std
  • await: yields control when I/O pending
  • No implicit heap allocation (unlike Go goroutines)

Trait System

  • Static dispatch: monomorphization (zero-cost, code bloat)
  • Dynamic dispatch: dyn Trait = vtable pointer
  • Object safety: required to use as dyn Trait
  • Blanket impl: impl<T: Display> ToString for T
  • Orphan rule: impl own trait OR for own type

Error Handling Patterns

  • thiserror: derive Error for library crates
  • anyhow: ergonomic errors for application code
  • ? operator: early return on Err, converts with From
  • Never panic in library code — return Result
  • Error enum variants with context (not just strings)

02Must-Know Deep Dives

🔥 Async/Await Internals — What Tokio Actually Does

Every async fn compiles to a state machine enum. Each .await point becomes a state. When the future is polled, it resumes from the last yielded state. No heap allocation for the state machine itself — it lives on the stack of the spawned task.

Pinning: Self-referential structs (futures holding references to their own fields) cannot be moved after creation. Pin<T> prevents moving the value. Unpin marker trait: most types implement it. Futures that self-reference must be !Unpin.

// async fn desugars to this: async fn fetch_data(url: &str) -> Result<String> { ... } // Becomes roughly: enum FetchDataFuture<'a> { State0 { url: &'a str }, // Before first .await State1 { connection: TcpStream }, // After connect, before read State2 { data: Vec<u8> }, // After read, before decode Done, } impl Future for FetchDataFuture<'_> { type Output = Result<String>; fn poll(self: Pin<&mut Self>, cx: &mut Context) -> Poll<Self::Output> { match self.get_mut() { Self::State0 { url } => { // Try to connect. If pending, register waker and return Pending. // When socket ready, OS notifies → Waker::wake() called → re-poll } } } }

Tokio runtime: Work-stealing thread pool. Each worker thread has a local run queue + can steal from others. Single-threaded runtime (Tokio current_thread) for your UTXO service that didn't need parallelism.

🔥 Unsafe Rust — When and How

Unsafe doesn't disable the borrow checker. It only unlocks 5 additional capabilities: raw pointer dereference, calling unsafe functions, implementing unsafe traits, mutating statics, accessing union fields.

// Pattern: safe abstraction over unsafe internals pub struct GapBuffer { data: Vec<u8>, gap_start: usize, gap_end: usize, } impl GapBuffer { pub fn insert(&mut self, ch: u8) { if self.gap_start == self.gap_end { self.expand_gap(); } unsafe { // SAFETY: gap_start is always < data.len() by invariant *self.data.get_unchecked_mut(self.gap_start) = ch; } self.gap_start += 1; } } // Your UTXO service: likely needed unsafe for crypto operations // interfacing with C libraries (libsecp256k1) extern "C" { fn secp256k1_ecdsa_sign( ctx: *const Secp256k1Context, sig: *mut Signature, msg: *const u8, ) -> i32; }

🔥 Memory Layout — Zero-Cost Abstractions

Rust structs: fields laid out in order with alignment padding (unlike C++ guaranteed order). #[repr(C)] forces C layout for FFI. #[repr(packed)] removes padding (unaligned reads = UB on some architectures).

Enum layout optimization: Option<&T> is the same size as &T — null pointer optimization. Option<NonZeroU32> = 4 bytes. Rust's type system enables this without any runtime overhead.

// Enum memory — discriminant + largest variant enum Message { Quit, // 0 bytes payload Move { x: i32, y: i32 }, // 8 bytes payload Write(String), // 24 bytes payload (ptr+len+cap) } // Size: 24 + 8 (discriminant+padding) = 32 bytes // Niche optimization: Option<&T> uses null as None use std::mem::size_of; assert_eq!(size_of::<Option<&i32>>(), size_of::<&i32>()); // true!

🔥 Rust vs Go — The Real Answer

🦀 USE RUST WHEN

  • Predictable latency (no GC pauses) — your UTXO service
  • Interfacing with C/system libraries (FFI)
  • Memory-constrained environments (embedded, WASM)
  • Crypto/security code (no memory bugs)
  • High-throughput data processing (zero-copy)

🐹 USE GO WHEN

  • Network services with concurrency (goroutines win)
  • Team velocity matters (simpler, faster to write)
  • Microservices (your CBDC bridge)
  • GC pauses acceptable (most web services: yes)
  • Ops tooling, CLIs, backends

04Quick Revision

RUST EXPERT FLASH CARDS

What does Pin guarantee?
The pointed-to value will not be moved in memory. Required for self-referential structs (futures). Pin<Box<T>> heap-pins, Pin<&mut T> stack-pins.
Send vs Sync?
Send: safe to transfer ownership to another thread. Sync: safe to share reference across threads. &T: Send iff T: Sync. Rc: !Send+!Sync. Arc: Send+Sync if T: Send+Sync.
Monomorphization cost?
Each generic instantiation = separate compiled function. Fast at runtime (no dispatch), larger binary. Use dyn Trait for type-erased dynamic dispatch when code size matters.
When is unsafe needed?
Raw pointer deref, unsafe fn call, unsafe trait impl, static mut access, union field access. NOT for "working around the borrow checker" — that's always wrong.
Option<&T> size?
Same as &T — null pointer optimization. None = null, Some(ptr) = non-null. Compiler exploits niche (invalid bit patterns) to avoid discriminant overhead.
thiserror vs anyhow?
thiserror: derive Error macro, type-safe, for library crates. anyhow: type-erased errors with context, for application code. Use both together: library returns thiserror, app wraps with anyhow.
