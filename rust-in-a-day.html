Why Rust Exists

70% of CVEs in C/C++ codebases are memory safety bugs: use-after-free, buffer overflows, data races, null pointer dereferences. Garbage-collected languages (Go, Java, Python) prevent these but pay a latency penalty for GC pauses and a throughput penalty for heap pressure. Rust eliminates memory safety bugs at compile time with zero runtime cost — no GC, no reference counting, no runtime checks.

The mechanism: the compiler statically verifies a set of ownership rules. If your code violates them, it does not compile. This is the borrow checker. It feels like fighting the compiler until you internalize the ownership model — then it feels like having a collaborator who catches every memory bug before the code runs.

Safety vs performance — where Rust sits
                     Memory Safe?
                     YES          NO
                  +-----------+-----------+
GC / runtime  YES | Java, Go  |     -     |
                  | Python    |           |
Performance   +-----------+-----------+
(no GC)       NO  |  RUST     | C, C++    |
                  | (zero GC) | (unsafe)  |
                  +-----------+-----------+

Rust's guarantee: the compiler proves memory safety
  No use-after-free (ownership: value has exactly one owner)
  No buffer overflow (bounds checks, slice types)
  No null pointer (Option instead of null)
  No data races (Send/Sync traits enforce thread safety)
  No dangling references (lifetimes prove reference validity)

Ownership — The Core Model

Rust's ownership system has three rules enforced at compile time: (1) each value has exactly one owner, (2) when the owner goes out of scope, the value is dropped (memory freed), (3) there can be at most one mutable reference OR any number of immutable references, but not both simultaneously.

Ownership, move, and drop
OWNERSHIP RULES:
  1. Every value has exactly ONE owner
  2. When owner goes out of scope: value is Dropped (free)
  3. Ownership can be MOVED (not copied)

fn main() {
    let s1 = String::from("hello");  // s1 OWNS the String
    //        ^-- allocates on heap
    
    let s2 = s1;                     // MOVE: s1 -> s2
    //        ownership transferred
    //        s1 is now INVALID (compiler error if used)
    
    println!("{}", s2);              // ok: s2 is the owner
    // println!("{}", s1);           // ERROR: value moved
    
}   // s2 goes out of scope: String is DROPPED (heap freed)
//  No GC needed: compiler knows exactly when to free

COPY vs MOVE:
  Types that implement Copy: i32, f64, bool, char, tuples of Copy types
  -> assignment copies the bits, original still valid
  
  Types that do NOT implement Copy: String, Vec, Box, most heap types
  -> assignment MOVES ownership, original invalidated

CLONE: explicit deep copy (expensive, always heap allocation)
  let s2 = s1.clone();  // both s1 and s2 valid, different heap allocs
rust — ownership and drop
fn takes_ownership(s: String) {
    println!("{s}");
}   // s dropped here -- heap freed

fn makes_copy(x: i32) {
    println!("{x}");
}   // x dropped here -- stack pop (trivial)

fn gives_ownership() -> String {
    let s = String::from("hello");
    s   // moved to caller -- NOT dropped here
}

fn main() {
    let s1 = gives_ownership();         // s1 owns the String
    let s2 = String::from("world");
    takes_ownership(s2);                // s2 moved into function
    // println!("{s2}");               // ERROR: s2 was moved
    makes_copy(5);                     // i32 is Copy, 5 still valid
    
    // RAII: Drop runs in reverse declaration order
    let _a = MutexGuard::new();        // lock acquired
    let _b = SomeResource::new();      // resource opened
    // out of scope: _b dropped first, _a dropped second
}   // s1 dropped -- heap freed

// Custom Drop implementation
struct FileHandle { fd: i32 }
impl Drop for FileHandle {
    fn drop(&mut self) {
        unsafe { libc::close(self.fd); }  // runs automatically
    }
}

Borrowing and Lifetimes

Instead of moving ownership, you can borrow a reference to a value. Borrowing has rules: you can have any number of immutable references (&T), OR exactly one mutable reference (&mut T), but never both at the same time. Lifetimes ensure references never outlive the data they point to.

Borrow checker rules — visualised
IMMUTABLE BORROWS: any number, simultaneously
  let s = String::from("hello");
  let r1 = &s;      // immutable borrow
  let r2 = &s;      // another immutable borrow -- ok
  println!("{r1} {r2}");  // both valid here
  // r1, r2 dropped here (end of last use = Non-Lexical Lifetimes)

MUTABLE BORROW: exactly one, no immutable borrows simultaneously
  let mut s = String::from("hello");
  let r1 = &s;      // immutable borrow starts
  let r2 = &mut s;  // ERROR: cannot borrow mutably while immutable borrow exists
  //              ^-- borrow checker rejects this

  // Correct pattern: borrow ends before mutable borrow starts
  let r1 = &s;
  println!("{r1}");  // r1 last use here -- borrow ENDS (NLL)
  let r2 = &mut s;   // ok: r1 is done
  r2.push_str(" world");

WHY this rule exists:
  Two simultaneous mutable references = DATA RACE in safe code
  One mut + one immut = iterator invalidation (classic C++ bug)
  Rust's compiler prevents BOTH at compile time, not at runtime

Lifetimes — Proving References Don't Dangle

Lifetime annotations — what they mean
// DANGLING REFERENCE (compile error):
fn dangle() -> &String {
    let s = String::from("hello");
    &s   // ERROR: s is dropped when function returns
         // returning a reference to dead memory
}

// LIFETIME ANNOTATION: 'a means "reference lives at least as long as 'a"
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
    // compiler needs 'a to know: return value lifetime
    // is tied to the SHORTER of x and y's lifetimes
}

// This would fail:
let result;
{
    let string2 = String::from("xyz");
    result = longest("long string", &string2);
}
println!("{result}");  // ERROR: string2 dropped above
// Compiler proves result can't outlive string2

// STRUCT with lifetime:
struct Important<'a> {
    text: &'a str,   // struct can't outlive the str it references
}

// Lifetime elision rules (compiler infers these):
// 1. Each reference param gets its own lifetime
// 2. If one ref param: output lifetime = input lifetime
// 3. If self param: output lifetime = self's lifetime
fn first_word(s: &str) -> &str {  // elided: fn first_word<'a>(s: &'a str) -> &'a str
    let bytes = s.as_bytes();
    for (i, &byte) in bytes.iter().enumerate() {
        if byte == b' ' { return &s[0..i]; }
    }
    &s[..]
}

Type System — Option, Result, Enums

Rust has no null. Instead of a null pointer dereference at runtime, you get a compile error if you try to use a potentially-absent value without handling the absent case. This is enforced by Option<T> and exhaustive pattern matching.

Option and Result — replacing null and exceptions
// NULL: billion dollar mistake. Rust has none.
// Instead:
enum Option {
    Some(T),   // value is present
    None,      // value is absent
}

enum Result {
    Ok(T),     // success with value
    Err(E),    // failure with error
}

// You MUST handle both cases (exhaustive match):
let maybe_num: Option = Some(42);
let doubled = match maybe_num {
    Some(n) => n * 2,
    None    => 0,     // must handle None
};

// ? operator: propagate errors up (early return on Err)
fn parse_and_double(s: &str) -> Result {
    let n = s.parse::()?;  // returns Err early if parse fails
    Ok(n * 2)
}

// Combinators (functional style):
let result = some_option
    .map(|x| x * 2)          // transform if Some
    .filter(|x| x > 0)       // keep if predicate
    .unwrap_or(0);            // default if None

// unwrap() panics on None/Err -- only use in tests/prototypes
// expect("msg") panics with message -- better than unwrap in prod
// if let: concise single-branch matching
if let Some(value) = maybe_num {
    println!("got {value}");
}

Enums with Data — Algebraic Data Types

rust — enums, pattern matching, generics
// Enum variants can hold different data types
enum Message {
    Quit,                          // unit variant
    Move { x: i32, y: i32 },      // struct variant
    Write(String),                 // tuple variant
    ChangeColor(i32, i32, i32),   // tuple variant
}

// Exhaustive pattern matching (compiler ensures all covered)
fn process(msg: Message) {
    match msg {
        Message::Quit => println!("quit"),
        Message::Move { x, y } => println!("move to {x},{y}"),
        Message::Write(text) => println!("write: {text}"),
        Message::ChangeColor(r, g, b) => println!("color {r},{g},{b}"),
    }
}

// GENERICS: zero-cost -- monomorphised at compile time
struct Stack {
    elements: Vec,
}
impl Stack {
    fn push(&mut self, item: T) { self.elements.push(item); }
    fn pop(&mut self) -> Option { self.elements.pop() }
}

// No runtime overhead vs concrete type
// Stack and Stack are DIFFERENT types at compile time
// Each gets its own optimised machine code (monomorphisation)

Traits — Rust's Interfaces

Traits define shared behaviour (like interfaces in Go/Java). Rust's standard library traits are the foundation of everything: Display, Debug, Clone, Iterator, From/Into, Send, Sync. Trait bounds constrain generic types at compile time — the optimizer sees through them and inlines everything.

Trait dispatch — static vs dynamic
STATIC DISPATCH (monomorphisation -- zero runtime cost):
  fn print_area(shape: &T) {
      println!("{}", shape.area());
  }
  // Compiler generates separate fn for each concrete type:
  // fn print_area_circle(shape: &Circle) { ... }
  // fn print_area_square(shape: &Square) { ... }
  // No virtual dispatch overhead. Inlineable.

DYNAMIC DISPATCH (trait objects -- runtime vtable):
  fn print_area(shape: &dyn Shape) {
      println!("{}", shape.area());
  }
  // shape is a FAT POINTER: (data ptr, vtable ptr)
  // vtable: array of function pointers for this concrete type
  // Runtime overhead: one indirect function call per method
  // Required when: multiple different types in same Vec, return type unknown

  let shapes: Vec> = vec![
      Box::new(Circle { r: 3.0 }),
      Box::new(Square { side: 4.0 }),   // different types, same Vec
  ];

OBJECT SAFETY: a trait can be used as dyn Trait only if:
  - No generic methods
  - No methods that take/return Self (by value)
  - No associated functions without self
rust — traits, blanket impls, operator overloading
use std::fmt;

trait Shape {
    fn area(&self) -> f64;
    fn perimeter(&self) -> f64;
    // Default implementation:
    fn describe(&self) -> String {
        format!("area={:.2} perimeter={:.2}", self.area(), self.perimeter())
    }
}

struct Circle { radius: f64 }
impl Shape for Circle {
    fn area(&self) -> f64 { std::f64::consts::PI * self.radius * self.radius }
    fn perimeter(&self) -> f64 { 2.0 * std::f64::consts::PI * self.radius }
}

// Display trait -- custom fmt::Display
impl fmt::Display for Circle {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "Circle(r={})", self.radius)
    }
}

// From/Into -- type conversion idiom
struct Meters(f64);
struct Feet(f64);
impl From for Meters {
    fn from(f: Feet) -> Self { Meters(f.0 * 0.3048) }
}
let m: Meters = Feet(10.0).into();  // Into is auto-derived from From

// Iterator trait -- the core of Rust's functional style
struct Counter { count: u32, max: u32 }
impl Iterator for Counter {
    type Item = u32;
    fn next(&mut self) -> Option {
        if self.count < self.max {
            self.count += 1;
            Some(self.count)
        } else {
            None
        }
    }
}
// Now ALL iterator adapters are available:
let sum: u32 = Counter { count: 0, max: 5 }
    .filter(|x| x % 2 == 0)
    .map(|x| x * x)
    .sum();   // all zero-cost: compiled to a tight loop

Concurrency — Fearless by Design

Rust's ownership model makes data races impossible in safe code. The Send trait marks types safe to send to another thread. The Sync trait marks types safe to share via reference between threads. The compiler uses these to reject code that would cause data races at compile time — not at runtime.

Send and Sync — thread safety at compile time
Send: type can be TRANSFERRED to another thread
  (its ownership can be moved across thread boundaries)
  Most types: Send. Exceptions: Rc (use Arc instead),
              raw pointers *mut T, MutexGuard

Sync: type can be SHARED by reference across threads
  (& T can be sent to another thread)
  Most types: Sync. Exceptions: Cell, RefCell,
              Rc, UnsafeCell

Compiler enforces at CALL SITES:
  thread::spawn(|| {
      // closure must be Send (captures are Send)
  });

  // Rc is NOT Send -- compile error:
  let rc = Rc::new(5);
  thread::spawn(move || println!("{rc}"));  // ERROR
  // Use Arc (atomic ref count) instead:
  let arc = Arc::new(5);
  thread::spawn(move || println!("{arc}"));  // ok

Common thread-safe patterns:
  Arc         -- shared ownership across threads (atomic refcount)
  Arc>  -- shared mutable state
  Arc> -- shared state, many readers OR one writer
  channel        -- message passing (mpsc::channel)
  Atomic types   -- lock-free shared integers (AtomicUsize, etc.)
rust — threads, Arc+Mutex, channels
use std::sync::{Arc, Mutex};
use std::thread;

// SHARED MUTABLE STATE: Arc>
fn shared_counter() {
    let counter = Arc::new(Mutex::new(0u64));
    let mut handles = vec![];

    for _ in 0..10 {
        let c = Arc::clone(&counter);   // clone the Arc (increments refcount)
        let h = thread::spawn(move || {
            let mut lock = c.lock().unwrap();  // blocks until acquired
            *lock += 1;
            // lock released here when MutexGuard drops (RAII)
        });
        handles.push(h);
    }
    for h in handles { h.join().unwrap(); }
    println!("final: {}", counter.lock().unwrap());
}

// MESSAGE PASSING: mpsc channel (multiple producer, single consumer)
fn channel_example() {
    let (tx, rx) = std::sync::mpsc::channel::();

    // Multiple senders
    for i in 0..3 {
        let tx = tx.clone();
        thread::spawn(move || {
            tx.send(format!("message {i}")).unwrap();
        });
    }
    drop(tx); // drop original sender so rx knows when all senders are done

    for msg in rx {  // rx is an Iterator -- blocks until message or all tx dropped
        println!("received: {msg}");
    }
}

// RAYON: data parallelism with no data races (work-stealing thread pool)
use rayon::prelude::*;
fn parallel_sum(data: &[i64]) -> i64 {
    data.par_iter()    // parallel iterator -- safe by Send/Sync
        .map(|x| x * x)
        .sum()
}

Async / Await — Futures and Executors

Rust's async/await is zero-cost: an async function compiles to a state machine with no heap allocation per await point (unlike Go goroutines which allocate a stack). The runtime (Tokio, async-std) provides the executor that polls futures. Understanding the polling model is essential for debugging async Rust.

Async execution model — state machine + executor
FUTURE: a value that represents a computation not yet complete
  trait Future {
      type Output;
      fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll;
  }
  Poll::Ready(value)  -- computation done, return value
  Poll::Pending       -- not done yet, will call cx.waker() when ready

ASYNC FUNCTION -> STATE MACHINE (compiler transforms this):
  async fn fetch(url: &str) -> String {
      let resp = client.get(url).await;    // await point 1
      let body = resp.text().await;        // await point 2
      body
  }

  Compiles to an enum (state machine):
  enum FetchFuture {
      Start,
      WaitingForResp { /* captured vars */ },
      WaitingForBody { /* captured vars */ },
      Done,
  }
  impl Future for FetchFuture {
      fn poll(mut self: Pin<&mut Self>, cx: &mut Context) -> Poll {
          loop { match &mut *self {
              Start => { /* start get(), transition to WaitingForResp */ }
              WaitingForResp { .. } => { /* poll inner, maybe transition */ }
              WaitingForBody { .. } => { /* poll inner, maybe return Ready */ }
              Done => panic!("polled after completion"),
          }}
      }
  }

EXECUTOR (Tokio runtime):
  - Thread pool (N threads, typically N = CPU cores)
  - Task queue (each spawned task = Future)
  - Poll loop: take task, poll its Future
    - Poll::Ready -> task done, remove
    - Poll::Pending -> park task, register waker
  - I/O reactor (epoll/kqueue) wakes tasks when I/O ready
  - Work stealing: idle threads steal from busy threads' queues
rust — async/await, Tokio, Pin
use tokio::{time, task, sync::mpsc};
use std::time::Duration;

// Basic async function
async fn fetch_data(id: u64) -> Result {
    let url = format!("https://api.example.com/items/{id}");
    let body = reqwest::get(&url).await?.text().await?;
    Ok(body)
}

// Concurrent execution: join! runs futures concurrently on ONE task
async fn fetch_all() {
    let (a, b, c) = tokio::join!(
        fetch_data(1),
        fetch_data(2),
        fetch_data(3),
    );   // all three in-flight simultaneously, single thread
}

// spawn: independent tasks (scheduled across thread pool)
async fn spawn_example() {
    let handle = tokio::spawn(async {
        time::sleep(Duration::from_secs(1)).await;
        42u64
    });
    let result = handle.await.unwrap();  // JoinHandle is a Future
}

// select!: race multiple futures, take first to complete
async fn with_timeout(fut: impl std::future::Future) -> Option {
    tokio::select! {
        result = fut => Some(result),
        _ = time::sleep(Duration::from_secs(5)) => None,
    }
}

// PIN: why it exists
// Self-referential struct (future capturing a reference to itself)
// If moved in memory, internal pointer becomes invalid
// Pin<&mut Self> guarantees the value won't move in memory
// Box::pin(future) -- heap-pin a future
// pin_mut!(future) -- stack-pin a local future
Common async pitfall

Holding a MutexGuard across an .await point is a deadlock waiting to happen — the guard holds the lock, the task parks, another task tries to lock, and everything stalls. Use tokio::sync::Mutex (async-aware) instead of std::sync::Mutex in async code, or release the lock before awaiting.

Memory Layout — Stack, Heap, Smart Pointers

Memory layout — stack vs heap types
STACK (automatic, fast, fixed size known at compile time):
  let x: i32 = 42;         // 4 bytes on stack
  let y: [i32; 4] = [1,2,3,4];  // 16 bytes on stack
  // freed instantly when scope ends (pointer decrement)

HEAP (dynamic, GC-free via RAII in Rust):
  let s = String::from("hello");
  // Stack: [ptr | len | capacity]  <- 24 bytes on stack
  //           |
  //           +-> Heap: [h][e][l][l][o]  <- 5 bytes on heap
  // When s drops: Box::drop -> dealloc (free())

SMART POINTERS:

  Box         Single owner, heap allocated
  [ptr] -> [T on heap]
  Use: recursive types, large values, dyn Trait

  Rc          Multiple owners, SINGLE THREAD only
  [ptr] -> [strong:2, weak:0, T]  <- refcount on heap
  Clone increments count. Drop decrements. Free when 0.
  NOT Send -- cannot cross thread boundary.

  Arc         Multiple owners, THREAD SAFE
  [ptr] -> [strong:2(atomic), weak:0(atomic), T]
  Atomic refcount = slower than Rc but safe across threads.

  RefCell     Interior mutability, SINGLE THREAD
  Borrow rules checked at RUNTIME (panics on violation).
  borrow() -> Ref    (like &T, can have many)
  borrow_mut() -> RefMut  (like &mut T, only one)

  Mutex       Interior mutability, THREAD SAFE
  lock() -> MutexGuard  (blocks until acquired)
  Poisoned if thread panics while holding lock.

  Cell        Interior mutability for Copy types
  get() copies out, set() replaces -- no borrows

Common patterns:
  Arc>  -- shared mutable state across threads
  Rc> -- shared mutable state on one thread
  Weak        -- non-owning reference (breaks Rc cycles)

Unsafe Rust — When You Know More Than the Compiler

Unsafe Rust does not disable safety — it allows five additional operations: dereference raw pointers, call unsafe functions, implement unsafe traits, access mutable statics, and access union fields. All other safety guarantees (memory layout, type system, lifetimes) still apply. The goal: contain unsafety to small, auditable blocks with a safe API around them.

rust — unsafe, raw pointers, FFI
// Raw pointers: *const T and *mut T
// Creating them is safe. Dereferencing is unsafe.
fn raw_pointer_example() {
    let x = 42i32;
    let raw = &x as *const i32;      // create raw pointer (safe)
    unsafe {
        println!("{}", *raw);         // dereference (unsafe)
    }
}

// Unsafe function: caller must uphold invariants
unsafe fn dangerous(ptr: *mut u8, len: usize) {
    // We assume ptr is valid and points to len bytes
    std::slice::from_raw_parts_mut(ptr, len);
}

// Safe abstraction over unsafe code (the correct pattern)
pub struct MyVec {
    ptr: *mut T,
    len: usize,
    cap: usize,
}
impl MyVec {
    pub fn get(&self, index: usize) -> Option<&T> {
        if index < self.len {
            Some(unsafe { &*self.ptr.add(index) })  // unsafe contained here
        } else {
            None  // bounds check makes the unsafe above safe
        }
    }
}

// FFI: calling C from Rust
extern "C" {
    fn abs(input: i32) -> i32;       // declare C function
    fn strlen(s: *const u8) -> usize;
}
fn call_c() -> i32 {
    unsafe { abs(-42) }              // unsafe: C has no safety guarantees
}

// Implementing an unsafe trait (Send/Sync)
// Only do this if you've manually verified thread safety
struct MyRawPointer(*mut u8);
unsafe impl Send for MyRawPointer {}  // I've verified this is safe to send

Performance Patterns

Zero-cost abstractions — what the compiler produces
HIGH LEVEL RUST CODE:
  let sum: i64 = (0..1_000_000)
      .filter(|x| x % 2 == 0)
      .map(|x| x * x)
      .sum();

WHAT THE COMPILER GENERATES (LLVM IR -> machine code):
  ; A single tight loop with no allocations
  ; No Vec created, no closures allocated on heap
  ; filter, map, sum all INLINED and FUSED
  xor eax, eax        ; sum = 0
  xor ecx, ecx        ; i = 0
  loop:
    test ecx, 1       ; if i % 2 != 0, skip
    jnz next
    mov rdx, rcx      ; rdx = i
    imul rdx, rdx     ; rdx = i * i
    add rax, rdx      ; sum += rdx
  next:
    inc ecx
    cmp ecx, 1000000
    jl loop

ARENA ALLOCATION: avoid heap fragmentation for short-lived objects
  use bumpalo::Bump;
  let arena = Bump::new();
  let vec = arena.alloc_slice_fill_copy(1000, 0u8);
  // all allocated from one contiguous slab
  // freed all at once when arena drops

AVOID CLONES: prefer borrows and slices
  // Bad: fn process(s: String)  -- takes ownership, caller must clone
  // Good: fn process(s: &str)   -- borrows a string slice
  // Good: fn process(data: &[u8]) -- borrows any slice

Flash Cards

What are Rust's three ownership rules?
(1) Each value has exactly one owner. (2) When the owner goes out of scope, the value is dropped (memory freed). (3) There can be multiple immutable references OR exactly one mutable reference, but never both simultaneously. These three rules are enforced at compile time by the borrow checker and make both use-after-free and data races impossible in safe Rust.
What is the difference between Clone and Copy in Rust?
Copy is for types where a bitwise copy is safe and cheap (i32, bool, f64, char, small structs). Assignment copies the bits — original remains valid. Clone is an explicit deep copy — you call .clone() when you want a copy of a heap-allocated type (String, Vec). Assignment of non-Copy types MOVES ownership. You cannot accidentally clone an expensive type without writing .clone().
What is a lifetime annotation and when do you need one?
A lifetime annotation (like 'a) tells the compiler how long a reference lives relative to another. You need them when a function takes multiple references and returns a reference — the compiler needs to know which input the output's lifetime is tied to. They don't change how long references live; they communicate the relationship so the compiler can verify no dangling references are created.
What is the difference between Rc and Arc?
Both are reference-counted smart pointers for shared ownership. Rc uses non-atomic (fast) reference counting — single-thread only, not Send. Arc uses atomic reference counting — thread-safe, implements Send. Use Rc for single-threaded shared ownership (with RefCell for mutation). Use Arc for multi-threaded shared ownership (with Mutex for mutation). Prefer Rc when you don't need thread safety — it's faster.
What does Pin do and why does async/await need it?
Pin guarantees a value won't be moved in memory after being pinned. Async state machines are self-referential — they contain references to fields within the same struct. If the struct moves, those internal references become dangling pointers. Pin prevents movement, making self-referential structs safe. Box::pin allocates on the heap and pins. You usually don't write Pin code directly — async/await generates it.
Static dispatch vs dynamic dispatch — when to use each?
Static dispatch (generics: fn foo(x: &T)): compiler generates a specialised version for each concrete type — monomorphisation. Zero runtime overhead, inlineable. Code size grows. Dynamic dispatch (trait objects: fn foo(x: &dyn Trait)): fat pointer (data ptr + vtable ptr). One virtual call per method. Use when: different concrete types in one Vec, return types unknown at compile time. Use static dispatch by default; use dyn only when you need type erasure.
What is the Send trait? What types are NOT Send?
Send marks types safe to transfer to another thread (their ownership can be moved across thread boundaries). The compiler checks Send automatically at spawn sites. NOT Send: Rc<T> (use Arc<T>), raw pointers *mut T, MutexGuard (don't send a lock across threads), types containing non-Send fields. If you try to spawn a closure that captures a !Send type, you get a compile error.
Why should you not hold std::sync::Mutex across an .await?
std::sync::Mutex blocks the OS thread while locked. In async code, .await parks the task — but the task still holds the MutexGuard (the lock). The runtime can't run other tasks on that thread while the lock is held. If another task on the same thread tries to lock the same mutex, deadlock. Solution: use tokio::sync::Mutex (async-aware, yields the thread while waiting) or release the lock before any .await point.
