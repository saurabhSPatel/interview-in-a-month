The Universal 45-Minute Framework

Every system design interview has the same structure underneath the specific problem. The candidate who controls the room doesn't answer questions — they drive a structure that makes their thinking legible. Here's the framework, minute by minute.

TimePhaseWhat to do
0–5 minClarify requirementsAsk before drawing. Users, scale (DAU, RPS), read:write ratio, latency SLA, consistency requirement, global or single region. You drive this — don't wait to be asked.
5–10 minCapacity estimation"100M users, 10% DAU = 10M/day, 1 post each = 115 writes/sec, 20× read:write = 2.3K reads/sec." Then: CPU-bound, memory-bound, I/O-bound? That determines the architecture.
10–20 minHLD — APIs + core flowDefine 2–3 key API endpoints first. Draw the 5-layer diagram. Identify which of the 12 patterns this maps to. Stay at one abstraction level — no class fields yet.
20–35 minDeep dive — hardest componentAsk "Where should I go deep?" or pick it yourself. Show internals: why Cassandra (LSM-tree, write-optimized), why Redis ZSET (O(log n), sorted by score), why Kafka (durable, replayable). This is where senior separates from mid.
35–45 minTrade-offs + wrap-upSurface limitations proactively. "One bottleneck is celebrity fan-out — I'd handle it with hybrid push/pull." End with: "What aspect would you like me to go deeper on?"
Staff-level signal

Start with what can go wrong, not only the happy path. Name the operational knobs: TTL, partition count, batch size, retry budget, timeout budget. End with observability — metrics, logs, traces for the critical path. Raising failure modes unprompted is the single clearest signal of seniority.

Pagination — the choice that matters at scale

Offset: ?page=2&limit=20 — simple but inconsistent when records are inserted during pagination. O(n) table scan grows with page number.

Cursor: ?cursor=eyJpZCI6NDJ9&limit=20 — stable regardless of concurrent writes. O(1) lookup. Use for any feed that receives new items.

Keyset: ?after_id=42&limit=20 — fastest for indexed columns. No skip cost. Default choice for high-traffic APIs with a clear sort key.

Week 1 — Foundations: Scaling, Caching, CAP

Day 01–02
Scaling — Vertical, Horizontal, the Scale Cube

Don't say "add more servers." Say: "I'd first profile to identify whether we're CPU, memory, or I/O bound. For the stateless API tier, horizontal scaling behind an ALB. For the database, start with read replicas and connection pooling via PgBouncer — escalate to sharding only when reads on replicas saturate."

The Scale Cube (AKF): X-axis = clone (horizontal scale). Y-axis = functional decomposition (microservices). Z-axis = data partition (sharding by customer ID or geography). Most systems do X first, then Y when the monolith becomes a bottleneck, then Z when a single dataset outgrows a node.

Day 03–04
CAP Theorem + PACELC — the More Useful Model

CAP: partition tolerance is not optional in distributed systems. The real choice is C vs A during a network partition. CP: HBase, ZooKeeper, etcd. AP: Cassandra, DynamoDB.

PACELC extends CAP: even without a partition, distributed systems trade latency vs consistency. DynamoDB: PA/EL (available, low latency). Spanner: PC/EC (consistent, higher latency). Using PACELC signals deeper thinking — use it when the interviewer asks about consistency trade-offs.

Day 05–07
Caching — Cache Stampede, Redis Internals, Invalidation

Cache stampede (thundering herd): many requests miss simultaneously, all hammer the DB. Fix: mutex lock (only one request rebuilds), probabilistic early expiry (PER algorithm), or request coalescing via singleflight. This is the failure mode interviewers want you to raise unprompted.

Write-through doubles write latency but keeps cache consistent. Write-behind is fast but risks data loss on crash. Cache-aside (lazy) is the safe default — every new key needs a DB round-trip first, but stale reads are bounded by TTL.

Week 1 Checkpoint
Five Questions to Answer Before Moving On
  • Draw the 5-layer request flow with latency at each hop — from memory
  • Design a distributed rate limiter at 50K RPS — which algorithm and why?
  • Payment system: which CAP trade-off? What's the UX of making the wrong choice?
  • gRPC vs REST: articulate the decision without hedging
  • What is the thundering herd problem and what are the three ways to fix it?

Week 2 — Storage Deep Dives

Day 08–10
Database Internals — B-Tree vs LSM, MVCC, WAL

B-Tree (PostgreSQL): self-balancing, in-place updates, good for read-heavy OLTP. Write amplification on random writes. LSM-Tree (Cassandra, RocksDB): MemTable → SSTable → compaction. O(1) sequential writes, Bloom filters per SSTable. Use for write-heavy workloads where you can tolerate compaction overhead.

MVCC: each write creates a new row version. Readers see a snapshot at transaction start — no reader/writer blocking. WAL: every change written to log before data pages. Enables crash recovery, streaming replication, and CDC (Debezium).

Day 11–14
Sharding — Consistent Hashing, Hotspot Prevention

Consistent hashing: place nodes on hash ring (0 to 2³²). Adding/removing a node moves only k/n keys. Virtual nodes (100–200 per physical node) ensure even distribution. Used by DynamoDB, Cassandra, Redis Cluster.

Shard key selection is the most critical decision. Bad key = hotspot. For social networks, celebrities cause hot shards on user_id hash. For time-series, time-based shard key means current-time shard gets all writes. Rule: high cardinality AND even access distribution.

Week 2 Checkpoint
Storage Fundamentals
  • When would you choose LSM over B-Tree? Articulate write amplification and compaction
  • Design a sharding strategy for a payment system — shard key choice and reasoning
  • w=2, r=2, n=3: strongly consistent? What about w=1, r=1, n=3?
  • Explain MVCC: why does it enable non-blocking reads with concurrent writers?
W3

Week 3 — Messaging, Events, Real-Time

Day 15–17
Kafka Internals — Partitions, Consumer Groups, EOS

Kafka guarantees ordering per partition only. For strict ordering of all events per user: partition by user_id. Cross-user ordering not guaranteed. For globally ordered events (rare): single partition, which sacrifices parallelism entirely.

Exactly-once: idempotent producer + transactional API. In practice: idempotent consumers (dedup on unique constraint in DB) are simpler and more resilient than Kafka EOS transactions. Partition count = parallelism ceiling — you can't have more consumers in a group than partitions.

Day 18–21
Saga vs 2PC, Outbox Pattern — Distributed Transactions

2PC is a blocking protocol. If coordinator crashes after PREPARE, participants hold locks indefinitely. Never use in microservices — Saga instead. Choreography is simpler but harder to debug. Orchestration has a coordinator (single debug point) but introduces coupling.

Outbox pattern solves dual-write: write to DB + Outbox table in one transaction. Debezium reads WAL → publishes to Kafka. Atomicity in DB, delivery guaranteed by CDC. This is the production answer whenever asked "how do you keep DB and message bus in sync?"

Week 3 Checkpoint
Messaging and Real-Time
  • Explain the Outbox pattern to a junior engineer — why is it needed, how does Debezium fit in?
  • Design a presence system for 10M concurrent chat users — heartbeat, fan-out, reconnect
  • When does Kafka guarantee ordering and when does it not?
  • Saga choreography vs orchestration: what's the debugging trade-off?

Week 4 — Full System Designs

Day 22 · Full Design
Design WhatsApp — Messaging at Scale

Storage: Cassandra, partitioned by chat_id, clustered by timestamp DESC. Write-optimized LSM-tree handles high message throughput. Message IDs: Snowflake — globally unique, time-sortable.

Delivery: WebSocket server per region. Connection server stateful — consistent hash routes user to same server. On send: write to Kafka → Delivery Service pushes to recipient's WebSocket → write to Cassandra. If offline: store + push notification via FCM/APNs.

Group fan-out: Small groups (<100): fan-out on write. Large groups: fan-out on read. Hard group size limit caps write amplification. Group membership cached in Redis, backed by PostgreSQL.

Day 26 · Full Design
Design Twitter Feed — The Fan-Out Decision

Fan-out on write: on tweet, push tweet_id to all followers' Redis ZSETs. O(followers) work at write. Celebrity problem: 50M followers = 50M Redis writes per tweet.

Fan-out on read: on feed load, fetch recent tweets from all followed users, merge-sort. Heavy reads, no write amplification. Slow for users following 5000+ accounts.

Hybrid (production answer): regular users (<1M followers) use fan-out on write. Celebrity users (>1M followers): merge their recent tweets with the user's pre-built feed at read time. Mentioning this unprompted signals production trade-off awareness.

Day 27 · LLD Focus
LLD — Concurrency as the Differentiator

The gap between senior and mid-level in LLD is concurrency awareness. Don't just draw classes — identify the race conditions.

Parking lot slot booking: availability check and reservation write are two operations — TOCTOU race condition. Two users check simultaneously, both see slot available, both book. Fix: optimistic locking — version field on Slot entity. UPDATE only succeeds if version matches what was read. On conflict, retry. Proactively naming this is the staff-level signal.

Week 5 (Bonus) — AI Agent System Design

AI agent design appears in senior interviews at companies building AI products. All prior weeks remain fully intact — this is additive. The core insight: AI agents are distributed systems with non-deterministic components. Every reliability and observability pattern from Weeks 1–4 applies.

Day 31–32
Agent Patterns — ReAct vs Plan-Execute

ReAct (Reason + Act): LLM alternates Thought → Action (tool call) → Observation → repeat. Flexible, discovers information dynamically. Risk: errors compound over long chains. Add hard max_steps limit.

Plan-and-Execute: Planner LLM generates full task list upfront. Executor (smaller, cheaper model) runs each step. Dramatically reduces cost. More auditable — the plan is a visible artifact. Brittle if plan is wrong at step 1.

Day 34–36 · Full Design
Design a Production RAG System

Chunking: semantic boundaries beat fixed-size. Parent-child: retrieve small chunk, return larger parent for context. Incremental indexing with file-watcher, not full re-index.

Hybrid search: ANN (semantic) + BM25 (keyword) combined with RRF scoring. Best precision for most domains — neither alone is consistently better.

Eval pipeline: RAGAS metrics (faithfulness, answer relevance, context recall) on a golden dataset. Run on every deploy. Most teams skip this and regret it.

AI Agent Production Failure Modes

Compounding hallucination: wrong tool call → wrong observation → cascades. Fix: validate tool output schema. Cost explosion: agent loops without progress. Fix: max_steps + cost budget hard cap + loop detection. No audit trail: impossible to debug. Fix: persistent trace log from day one.

Flash Cards — Final Review

What's the first thing you do in a system design interview?
Clarify requirements before drawing anything. Ask: Who are the users? What's the scale (DAU, RPS)? Read:write ratio? Latency SLA? Consistency requirement? Global or single region? You drive this — don't wait to be asked.
Cache-aside vs write-through — when does each win?
Cache-aside (lazy): safe default, no wasted writes for infrequently accessed data. Cache misses on first read. Write-through: no cache miss penalty on read, but every write hits both cache and DB — doubles write latency. Use write-through when read-after-write consistency is critical.
When does Kafka guarantee message ordering?
Within a single partition only. Cross-partition ordering is not guaranteed. For strict per-user ordering: partition by user_id — all that user's events land in one partition in order. Cross-user events may arrive out of order from each other.
Why does the Outbox pattern exist — what problem does it solve?
Dual-write problem: you can't atomically write to DB AND publish to Kafka. If you write to DB then publish and crash between the two, the event is lost. Outbox: write DB + Outbox table in one DB transaction (atomic). Debezium reads WAL → publishes to Kafka. Atomicity is in the DB; delivery is by CDC.
What's the difference between CP and AP in CAP theorem — examples?
CP: during a partition, system refuses writes to maintain consistency — never returns stale data. Examples: ZooKeeper, etcd, HBase. AP: during a partition, system continues serving — may return stale data. Examples: Cassandra, DynamoDB, CouchDB. In practice: you choose what "wrong" looks like — stale data (AP) or unavailability (CP).
Fan-out on write vs read — when does each make sense?
Fan-out on write: precompute each user's feed at write time. Fast reads (O(1)), expensive writes for users with many followers. Fan-out on read: compute feed at read time by merging followed users' timelines. Handles celebrities well but slow for users following many accounts. Production answer: hybrid — write fan-out for regular users, read fan-out for celebrities (>1M followers).
Why is jitter non-optional in retry backoff?
Without jitter, 1000 services that all failed at t=0 will all retry at exactly t=100ms, t=200ms — recreating the original load spike that caused the failure. Full jitter (random 0 to backoff) desynchronizes retries across all clients, spreading the load over the backoff window instead of concentrating it.
What is the TOCTOU race condition and how do you fix it in a booking system?
Time-of-check-time-of-use: check availability (read), then book (write) are two separate operations. Two users check simultaneously, both see available, both book. Fixes: optimistic locking (version field, UPDATE fails if version changed), pessimistic locking (SELECT FOR UPDATE), or a Redis SET NX atomic claim before the DB write.
