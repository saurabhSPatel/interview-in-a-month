The Universal 45-Minute Framework

Every system design interview has the same structure underneath the specific problem. The candidate who controls the room doesn't answer questions — they drive through a structure that makes their thinking legible. Here's the framework, minute by minute.

TimePhaseWhat to do
0–5 minClarify requirementsAsk before drawing anything. Users, scale (DAU, RPS), read:write ratio, latency SLA, consistency requirements, global or single region. You drive this — don't wait to be asked.
5–10 minCapacity estimationBack-of-envelope math. "100M users, 10% DAU = 10M/day, 1 post each = 115 writes/sec, 20× read:write = 2.3K reads/sec." Then: CPU-bound, memory-bound, I/O-bound, or network-bound? That determines the architecture.
10–20 minHLD — APIs + core flowDefine 2–3 key API endpoints first. Draw the 5-layer diagram. Identify which of the 12 patterns this maps to. Stay at one abstraction level — no class fields yet. State what you're skipping: "I'll skip auth for now and focus on the write path."
20–35 minDeep dive — hardest componentAsk "Where should I go deep?" or pick it yourself. Show internals: why Cassandra for messages (LSM-tree, write-optimized), why Redis ZSET for feeds (O(log n) insert, sorted by score), why Kafka for fan-out (durable, replayable). This is where you differentiate from mid-level.
35–45 minTrade-offs + wrap-upSurface limitations proactively. "One bottleneck is celebrity fan-out — I'd handle it with hybrid push/pull." End with: "What aspect would you like me to go deeper on?" — shows you're comfortable driving the conversation.
Staff-level signal

Start with what can go wrong, not only the happy path. Name the operational knobs: TTL, partition count, batch size, retry budget, timeout budget. End with observability — metrics, logs, traces for the critical path. Raising failure modes unprompted is the single clearest signal of seniority.

The 12 Universal Patterns

Every system design question maps to one or more of these 12 patterns. Identify the pattern first — the specific trade-offs follow automatically. The decision tree at the bottom is what you run through in the first 2 minutes after clarifying requirements.

01 · Read Heavy
Cache-Aside + Read Replicas
Cache-Aside + Read Replicas
App checks Redis first → miss → read DB → populate cache. Add read replicas to distribute query load. Critical question: what's the invalidation strategy? TTL vs event-driven.
Social feeds · Product pages · YouTube view counts
02 · Write Heavy
Write-Ahead Queue + Async
Queue + Async Workers
Writes go to Kafka/SQS first, workers process async. Decouples ingestion from processing. Backpressure = queue depth. Key: exactly-once semantics? Idempotency key on each message.
Analytics events · IoT telemetry · Email queue · Logs
03 · Real-Time
WebSocket + Pub-Sub
WebSocket + Pub-Sub Backend
Persistent WebSocket connections, backend pub-sub (Redis/Kafka) fans out to all connection servers. Key: reconnect + replay from last received sequence number. Presence via heartbeat TTL.
Chat · Live scores · Collaborative editing · Dashboards
04 · Global Scale
CDN + GeoDNS
CDN + GeoDNS + Edge Cache
Static assets from CDN edge. GeoDNS routes to nearest PoP. Origin shield collapses cache misses. Key: invalidation at edge, data residency laws, cache-control header strategy.
Netflix · YouTube delivery · Global APIs · Static assets
05 · Distributed Tx
Saga Pattern
Saga (Choreography / Orchestration)
Break multi-service transaction into steps. Each step publishes an event. On failure: compensating transactions roll back in reverse order. Key: idempotent compensations. Orchestration = easier to debug.
E-commerce checkout · Flight booking · Bank transfer
06 · Dual Write
Outbox + CDC
Outbox Pattern + CDC (Debezium)
Write to DB + Outbox table in the same transaction. Debezium reads WAL → publishes to Kafka. Atomicity in DB, eventual delivery to bus. Eliminates the dual-write failure mode entirely.
Any service writing to DB AND publishing events
07 · Search
Inverted Index + ES
Inverted Index + ES Sync via CDC
Primary DB = source of truth. Elasticsearch = read-optimized search replica, synced via Kafka CDC. Hybrid search: ANN (semantic) + BM25 (keyword) + RRF fusion for best precision.
Airbnb search · LinkedIn · Product catalog · Legal docs
08 · Geo Proximity
Geohash + Redis GEO
Geohash Grid + Redis GEOSEARCH
Encode lat/lng as geohash string. Query neighboring cells. Redis GEOSEARCH for radius. Update moving objects via WebSocket. Key: precision vs cell size — shorter hash = larger cell = less precision.
Uber driver matching · Yelp nearby · Tinder radius
09 · Rate Limiting
Token Bucket
Token Bucket / Sliding Window
API Gateway checks Redis before forwarding. Token bucket for burst-friendly limiting. Sliding window counter for precise limits. Return 429 + Retry-After. Distributed: consistent hash user_id to Redis shard.
API platform · Payment gateway · SMS OTP
10 · Unique IDs
Snowflake ID
Snowflake ID / UUID v7
Snowflake: 64-bit = 41-bit timestamp + 10-bit machine + 12-bit seq. Sortable, ~4096 IDs/ms/node, no coordination needed. UUID v7: time-ordered + universally unique. Both enable index locality.
Tweet IDs · Order IDs · Message IDs · Media IDs
11 · CQRS
Command Query Separation
CQRS — Separate Read/Write Models
Write model: normalized DB for consistency. Read model: denormalized store (Redis, Cassandra) optimised for each query pattern. Commands produce events that update multiple read models. Eventual consistency.
E-commerce (orders write, catalog read) · Banking ledger
12 · Resilience
Circuit Breaker
Circuit Breaker + Bulkhead + Retry
Circuit breaker opens on failure threshold → fail fast. Bulkhead: isolate thread pools per dependency. Retry with exponential backoff + jitter. Timeout on every downstream call. Add this to every design.
Any service with downstream dependencies — always add this
Pattern Decision Tree

Read:write > 5:1 → Pattern 01 (cache). Write > 50K/s → Pattern 02 (queue). Server must push → Pattern 03 (WebSocket). Global <100ms → Pattern 04 (CDN). Multiple services in one tx → Pattern 05 (saga). DB write + publish event → Pattern 06 (outbox). Full-text search → Pattern 07. Location queries → Pattern 08. API abuse → Pattern 09. Unique IDs at scale → Pattern 10. Different read/write models → Pattern 11. Has downstream deps → Pattern 12 (always).

Pagination — the choice that matters at scale

Offset: ?page=2&limit=20. Simple, but inconsistent when records are inserted or deleted during pagination. Causes OFFSET table scans that get slower as page number grows — O(n) for page n.

Cursor: ?cursor=eyJpZCI6NDJ9&limit=20. Stable regardless of concurrent writes. Cursor = opaque base64-encoded value (last-seen ID or compound key). O(1) lookup. Use this for any feed or list that receives new items while being paginated.

Keyset: ?after_id=42&limit=20. Fastest for indexed columns — the database uses the index to jump directly to the right position. No skip cost. Default choice for high-traffic APIs with a clear sort key.

Week 1 — Foundations: Scaling, Caching, CAP

First principles before patterns. You can't reason about which pattern to apply until you can answer: what's the bottleneck type, what's the consistency requirement, and what breaks at 10× traffic?

Day 01 · Concept
Scaling — Vertical, Horizontal, the Scale Cube

Don't say "add more servers." Say: "I'd first profile to identify whether we're CPU, memory, or I/O bound. For the stateless API tier, horizontal scaling behind an ALB. For the database, start with read replicas and connection pooling via PgBouncer — escalate to sharding only when reads on replicas saturate."

The Scale Cube (AKF): X-axis = clone (horizontal scale). Y-axis = functional decomposition (microservices, split by domain). Z-axis = data partition (sharding by customer ID, geography). Most systems do X first, then Y when the monolith becomes a bottleneck, then Z when a single dataset outgrows a single node.

Day 03 · Concept
Caching — Strategies, Redis Internals, Invalidation

Cache-aside (lazy loading) is the default: app checks Redis, miss → read DB → populate cache. Simple, safe, but every new key requires a DB round-trip first.

Write-through: write to cache AND DB synchronously. Consistent but doubles write latency. Write-behind: write to cache only, flush async. Fast writes, risk of data loss on cache crash.

Cache stampede (thundering herd): many requests miss simultaneously, all hammer the DB. Fix: mutex lock (only one request rebuilds the cache), probabilistic early expiry (PER algorithm), or request coalescing. This is the failure mode interviewers want you to raise unprompted.

Day 04 · Concept
CAP Theorem + PACELC — the More Useful Model

CAP: partition tolerance is not optional in distributed systems. The real choice is between C and A during a network partition. CP systems: HBase, ZooKeeper, etcd. AP systems: Cassandra, DynamoDB.

PACELC extends CAP: even without a partition, distributed systems trade off latency vs consistency. DynamoDB: PA/EL (available, low latency). Spanner: PC/EC (consistent, higher latency). The PACELC framing shows deeper thinking than CAP alone — use it when the interviewer asks about consistency trade-offs.

Week 1 Checkpoint
Five Questions to Answer Before Moving On
  • Draw the full request flow (Client → CDN → LB → Gateway → Service → Cache → DB) with latency at each hop — from memory
  • Design a REST API for a ride-booking app: 5 endpoints, correct verbs, status codes, cursor pagination
  • Design a distributed rate limiter for 50K RPS — which algorithm and why?
  • Payment system: which CAP trade-off do you make and what's the user experience of the wrong choice?
  • gRPC vs REST: articulate the decision without hedging. "I'd use gRPC because..."

Week 2 — Storage Deep Dives

The most common mistake in system design interviews is choosing a datastore without justifying why. "I'd use MongoDB because it's flexible" is a red flag. The correct approach: identify the access patterns first, then choose the store that serves them.

Day 09 · Concept
Database Internals — B-Tree vs LSM, MVCC, WAL

B-Tree (PostgreSQL, MySQL): self-balancing, O(log n) reads and writes. In-place updates — good for read-heavy workloads. Random writes cause write amplification.

LSM-Tree (Cassandra, RocksDB): write to MemTable first (in-memory) → flush to immutable SSTables on disk → background compaction merges SSTables. O(1) sequential writes. Bloom filters per SSTable skip unnecessary reads. Use for write-heavy workloads where you can tolerate compaction overhead.

MVCC: each write creates a new row version with a transaction timestamp. Readers see a snapshot at their transaction start time — no reader/writer blocking. WAL: every change written to log before applied to data pages. On crash: replay WAL. Also enables streaming replication and CDC (Debezium).

Day 10 · Concept
Sharding — Consistent Hashing, Hotspot Prevention

Consistent hashing: place nodes on a hash ring (0 to 2³²). Key → hash → find nearest node clockwise. Adding/removing a node moves only ~k/n keys (where k = total keys, n = nodes). Virtual nodes (vnodes) give each physical node 100–200 points on the ring for even distribution.

Shard key selection is the most critical decision. Bad key = hotspot = one shard does all work. For social networks, user_id hash works for regular users but celebrities cause hot shards. For time-series, time-based shard key means the current-time shard gets all writes. Solution: composite key (time + device_id). Rule: high cardinality AND even distribution across your access patterns.

Week 2 Checkpoint
Storage Fundamentals
  • When would you choose LSM over B-Tree? Articulate write amplification and compaction overhead
  • Design a sharding strategy for a payment system — what's your shard key and why?
  • Explain MVCC: why does it enable non-blocking reads alongside concurrent writes?
  • w=2, r=2, n=3: strongly consistent? What about w=1, r=1, n=3?

Week 3 — Messaging, Event-Driven, Real-Time

Day 15 · Concept
Kafka Internals — Partitions, Consumer Groups, EOS

Kafka guarantees ordering per partition only — not across partitions. For strict ordering of all events per user: partition by user_id. All events for that user land in the same partition in order. Cross-user ordering is not guaranteed. For globally ordered events (rare): single partition, which sacrifices parallelism.

At-least-once by default. For exactly-once: idempotent producer (enable.idempotence=true) + transactional API. In practice: idempotent consumers (dedup on DB side using unique constraint on idempotency key) are simpler and more resilient than Kafka EOS transactions.

Partition count = parallelism ceiling. You can't have more consumers (in a group) than partitions. Setting partition count too low is a common mistake — you can add partitions later but it requires rebalancing and potentially breaks key ordering.

Day 16 · Concept
Distributed Transactions — Saga vs 2PC, Outbox Pattern

2PC is a blocking protocol. If the coordinator crashes after PREPARE, participants hold locks indefinitely. Avoid in microservices — use Saga instead.

Saga: break distributed transaction into local transactions. Each step publishes an event. On failure: compensating transactions in reverse. Choreography (event-driven, no coordinator) is simpler but harder to track overall state. Orchestration (central saga orchestrator) is easier to monitor and debug, but introduces coordinator coupling.

Outbox pattern solves the dual-write problem: you can't atomically write to DB AND publish to Kafka. Write to DB + Outbox table in one DB transaction (atomic). Debezium watches the WAL for Outbox table inserts and publishes to Kafka. Atomicity is in the DB; delivery is guaranteed by CDC. This is the production-grade answer whenever someone asks how to keep a DB and a message bus in sync.

Week 3 Checkpoint
Messaging & Real-Time
  • Explain the Outbox pattern to a junior engineer — why is it needed and exactly how does it work?
  • Design a presence system for a chat app with 10M concurrent users — heartbeat, fan-out, reconnect
  • When does Kafka guarantee ordering? When does it not?
  • Saga choreography vs orchestration — what's the debugging trade-off?

Week 4 — Full System Designs

Apply everything from the first three weeks to complete systems. The goal: HLD complete in the first 10 minutes, deep dive on the hardest component, trade-offs raised unprompted. You should be driving, not answering.

Day 22 · Full Design
Design WhatsApp — Messaging at Scale

Storage: Cassandra, partitioned by chat_id, clustered by timestamp DESC — efficient time-ordered reads per chat, write-optimized LSM-tree handles high message throughput. Message IDs: Snowflake — globally unique, time-sortable, no coordination.

Delivery: WebSocket server per region. Connection server is stateful — use consistent hashing to route a user's connections to the same server. On message send: write to Kafka → Delivery Service pushes to recipient's WebSocket → write to Cassandra. If offline: store + push notification via FCM/APNs.

Group fan-out: Small groups (<100): fan-out on write — message to each member's inbox. Large groups: fan-out on read — members pull from shared inbox on open. Hard group size limit (WhatsApp uses 1024) caps write amplification. Group membership cached in Redis, backed by PostgreSQL.

Day 26 · Full Design
Design Twitter Feed — The Fan-Out Decision

Fan-out on write (push): on tweet, push tweet_id to all followers' feed caches (Redis ZSET, score=timestamp). O(followers) work at write time. Read is O(1). Celebrity problem: 50M followers = 50M Redis writes per tweet.

Fan-out on read (pull): on feed load, fetch recent tweets from all followed users, merge-sort. Heavy reads, no write amplification for celebrities. Slow for users following 5000+ accounts.

Hybrid — the actual Twitter answer: regular users (<1M followers) use fan-out on write. Celebrity users (>1M followers): on read, merge their recent tweets with the user's pre-built feed from Redis. Mentioning this approach unprompted signals you know real production trade-offs, not just textbook patterns.

Day 27 · LLD
LLD Deep Dive — Concurrency as the Differentiator

The gap between senior and mid-level in LLD is concurrency awareness. Don't just draw classes — identify the race conditions.

Example: parking lot slot booking. The availability check and the reservation write are two separate operations — a TOCTOU (time-of-check-time-of-use) race condition exists. Two users check simultaneously, both see the slot as available, both try to book. Fix: optimistic locking — add a version field to the Slot entity. The UPDATE only succeeds if version matches what was read. On conflict, retry with fresh read. Proactively naming this race condition and its solution is the staff-level signal.

Week 5 — AI Agent System Design (Bonus)

AI agent design is now appearing in senior interviews at companies building AI products. All prior weeks remain fully intact — this is additive. The core insight: AI agents are distributed systems with non-deterministic components. Every reliability and observability pattern from Weeks 1–4 applies.

Day 31 · Concept
Agent Patterns — ReAct vs Plan-Execute

ReAct (Reason + Act): LLM alternates Thought → Action (tool call) → Observation → repeat. Flexible, handles unknown task shapes, discovers information dynamically. Risk: errors compound over long chains. Add a hard max_steps limit.

Plan-and-Execute: Planner LLM generates a full step-by-step task list upfront. Executor (smaller, cheaper model) runs each step. Dramatically reduces cost. More auditable — the plan is a visible, loggable artifact. Brittle if the plan is wrong at step 1 — add a re-planning step for unexpected failures.

Day 34 · Full Design
Design a Production RAG System

Chunking: semantic boundaries beat fixed-size. Parent-child: retrieve small chunk, return larger parent for context. For a 10M document corpus: incremental indexing with file-watcher, not full re-index on every change.

Hybrid search: ANN (semantic, via embeddings) + BM25 (keyword) combined with RRF (Reciprocal Rank Fusion) scoring. Best precision for most domains — neither semantic nor keyword alone is consistently better.

Eval pipeline: RAGAS metrics (faithfulness, answer relevance, context recall) on a golden dataset. Run on every deploy — catch regressions before they reach users. This is the part most teams skip and regret.

Production Failure Modes for AI Agents

Compounding hallucination: wrong tool call → wrong observation → wrong next step → cascades. Fix: validate tool output schema, add reflection on unexpected results. Cost explosion: agent loops without progress. Fix: max_steps + cost budget hard cap + loop detection (hash of recent states). No audit trail: impossible to debug. Fix: persistent trace log from day one — non-negotiable.
