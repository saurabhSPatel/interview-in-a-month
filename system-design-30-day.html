WEEK 01 Foundations — Scaling, Caching, Networking, CAP FIRST PRINCIPLES
DAY 01 Scaling Fundamentals — Vertical vs Horizontal Concept ▼

Core Concepts

  • Vertical scaling: limits, SPOF, no code change
  • Horizontal scaling: stateless requirement, complexity
  • Stateless vs stateful services: session handling
  • Scale cube: X (clone), Y (decompose), Z (shard)

Real-World System

Your CBDC platform: why Go microservices (stateless, horizontal) beat a monolith for Hyperledger Fabric nodes that need independent scaling per participant bank.

Trade-offs to Articulate

Vertical: fast, cheap ops, but hard ceiling ~96-core machines. Horizontal: theoretically infinite, but requires service discovery, consistent hashing, session externalization, distributed state. CAP implications change when you go horizontal.

Interview Approach

When asked "how do you scale X?": first identify the bottleneck (CPU/memory/IO/network), then choose axis. Don't say "add more servers" — say "I'd scale the stateless API tier horizontally behind a load balancer, while keeping the database vertical + read replicas initially."

DAY 02 Load Balancing — Algorithms & Layers Infra ▼

Core Concepts

  • L4 (TCP) vs L7 (HTTP) load balancing
  • Algorithms: Round Robin, Least Connections, IP Hash, Consistent Hashing
  • Health checks: active vs passive
  • Session affinity (sticky sessions) — when to avoid
  • Anycast for global load balancing

Real System: Nginx/HAProxy vs AWS ALB

  • L7 ALB: path-based routing for microservices
  • Weighted routing for canary deployments
  • Connection draining for zero-downtime deploys
  • Your Helm deployments already use K8s Services (kube-proxy = software LB)

Trade-offs

Consistent hashing LB: minimizes cache misses when scaling (same key → same server). But IP Hash breaks when clients use proxies (all appear as single IP). Least-connections beats round-robin for variable request costs (long-running jobs).

DAY 03 Caching — Strategies, Invalidation, Redis Deep Caching ▼

Caching Strategies

  • Cache-aside (Lazy): app checks cache, miss → DB, populate cache
  • Write-through: write to cache + DB synchronously
  • Write-behind: write to cache, async flush to DB
  • Read-through: cache loads from DB on miss
  • TTL vs event-based invalidation

Your Redis Experience Mapped

80% DB load reduction you achieved = cache-aside pattern. Key design: forex:rate:{USD}:{EUR}:{timestamp_bucket} — time-bucketed TTL. Stampede prevention: probabilistic early expiration or distributed lock (Redis SET NX PX).

Cache Invalidation — "Hardest Problem in CS"

Write-through: consistency but write latency doubles. Write-behind: fast writes but data loss risk on cache crash. Cache-aside: stale reads possible (use short TTL). Event-driven invalidation via Kafka (your expertise): write → publish event → consumers invalidate specific keys. Most consistent approach.

Senior-Level Points

Mention: Cache stampede (thundering herd), dog-pile effect. Solutions: mutex lock, probabilistic early refresh, request coalescing. Also discuss: hot keys, cache warming, multilevel caching (L1=local in-process, L2=Redis, L3=DB).

DAY 04 CAP Theorem + Consistency Models Theory ▼

CAP Theorem

  • Consistency: all nodes see same data simultaneously
  • Availability: every request gets a response
  • Partition Tolerance: system works despite network splits
  • Reality: P always occurs in distributed systems → choose C or A
  • PACELC extends CAP: latency vs consistency tradeoff even without partition

Consistency Models (Spectrum)

  • Strong: linearizability (single node illusion)
  • Sequential: operations ordered consistently
  • Causal: causally related ops in order
  • Eventual: all nodes converge eventually
  • Read-your-writes: session consistency

Real Mapping to Systems

Hyperledger Fabric (your work): CP system — consensus before commit, sacrifices availability during partition. Redis (your work): AP by default with async replication, eventual consistency. Your CBDC forex settlement: needed strong consistency for double-spend prevention → chose CP, accepted availability hit.

Interview Signal

Never say "I'd use CAP theorem" — say "Given the partition tolerance requirement, I need to choose between consistency and availability. For payment systems like this CBDC, I'd choose CP — a failed transaction is better than a duplicate credit." This is Staff-level thinking.

WEEK 02 Storage Deep Dives — SQL, NoSQL, Sharding, Replication DATA LAYER
DAY 08 SQL vs NoSQL — When to Choose What Storage ▼

SQL Strengths

  • ACID transactions (your CBDC payment system)
  • Complex joins, aggregations
  • Schema enforcement = data integrity
  • Mature tooling, EXPLAIN plans, indexes
  • Best for: financial systems, ERP, anything relational

NoSQL Categories

  • Document (MongoDB): flexible schema, JSON nesting
  • Key-Value (Redis/DynamoDB): O(1) access, simple
  • Wide-Column (Cassandra): time-series, write-heavy
  • Graph (Neo4j): relationship traversal

Decision Framework for Interviews

Ask yourself: (1) Do I need ACID multi-entity transactions? → SQL. (2) Do I need to query by multiple attributes with complex joins? → SQL. (3) Is write throughput >50K/s with simple access patterns? → Cassandra/DynamoDB. (4) Flexible schema, document-centric reads? → MongoDB. (5) Cache/session? → Redis. Don't say "I'd use MongoDB because it's flexible" — that's junior thinking.

DAY 09 Database Internals — B-Tree, LSM, MVCC, WAL Internals ▼

B-Tree (PostgreSQL/MySQL)

  • Balanced tree, O(log n) reads and writes
  • Good read/write mix workloads
  • Pages: typically 8KB, tuned for disk
  • Write amplification: updating internal nodes

LSM Tree (RocksDB/Cassandra)

  • Write to MemTable → flush to SSTable on disk
  • Compaction merges SSTables
  • O(1) writes (sequential), O(log n) reads
  • Write amplification during compaction
  • Bloom filters prevent unnecessary disk reads

MVCC + WAL

MVCC: Multiple readers, one writer. Each transaction sees snapshot. Old versions kept until no longer needed (vacuum). Enables non-blocking reads.

WAL (Write-Ahead Log): Write to log first, then data pages. On crash: replay WAL for recovery. Also enables replication — ship WAL to replicas. PostgreSQL uses this for streaming replication.

DAY 10 Sharding — Strategies and Hotspot Prevention Scalability ▼

Sharding Strategies

  • Hash sharding: uniform distribution, no range queries
  • Range sharding: range queries efficient, hotspots risk
  • Directory sharding: lookup table, flexible, single point
  • Geo sharding: data locality, GDPR compliance
  • Consistent hashing: minimize re-distribution on scale

Consistent Hashing

  • Virtual nodes (vnodes): better distribution
  • When node added/removed: only adjacent keys move
  • k/n keys move on average (not k total)
  • Used by: Cassandra, DynamoDB, Redis Cluster

Shard Key Selection — Critical

Bad shard key = hotspots = one shard does all work. For social network: user_id hash works until celebrities exist (hot users). Solution: celebrity detection + fan-out on write vs fan-out on read. For time-series: time-based sharding causes write hotspot on latest shard. Use time + device_id composite key.

DAY 11 Replication — Sync vs Async, Quorum Availability ▼

Replication Models

  • Single-leader: all writes to primary, fan-out to replicas
  • Multi-leader: multiple write points, conflict resolution
  • Leaderless (Dynamo): write/read quorums
  • Async: low latency writes, risk of data loss
  • Sync: durability guarantee, higher write latency

Quorum Reads/Writes

  • n = replicas, w = write quorum, r = read quorum
  • Strong consistency: w + r > n
  • Typical: n=3, w=2, r=2
  • Read-heavy: w=3, r=1 (write everywhere, read one)
  • Write-heavy: w=1, r=3 (fast writes, consistent reads)

Why This Matters for Your Background

Hyperledger Fabric uses PBFT-like consensus — a multi-party quorum system. You can connect your blockchain consensus experience directly to Dynamo-style quorum writes. This is excellent Staff-level signal in interviews.

Index Types

WEEK 03 Messaging Systems, Kafka Internals, Real-Time Architecture EVENT-DRIVEN
DAY 15 Kafka Internals — Partitions, Offsets, Consumer Groups Messaging ▼

Kafka Architecture

  • Topic → Partitions (ordered, immutable log)
  • Offset: per-partition monotonic counter
  • Consumer Group: each partition consumed by one consumer
  • Replication factor: ISR (In-Sync Replicas)
  • Leader election via ZooKeeper / KRaft
  • Retention: time-based or size-based

Performance Knobs (Your Experience)

  • Partition count: parallelism ceiling for consumers
  • acks=all: strongest durability, highest latency
  • linger.ms + batch.size: throughput vs latency
  • Compression: LZ4 for speed, zstd for ratio
  • Log compaction: keep only latest value per key

Ordering Guarantee Nuance

Kafka guarantees per-partition ordering only. Cross-partition: use a single partition (lose parallelism) or accept reordering. For your CBDC asset transfers: partition by account_id → all transfers for one account are ordered. But cross-account transactions need 2-phase approach or saga pattern.

Staff-Level: Exactly-Once Semantics

Kafka EOS: idempotent producer (sequence numbers), transactional API (atomic produce + consumer offset commit). In practice: idempotent consumers (dedup on DB side) are simpler and more resilient. Your forex gateway likely uses this — mention it.

DAY 16 Distributed Transactions — 2PC, Saga, Outbox Pattern Consistency ▼

2-Phase Commit (2PC)

  • Coordinator: PREPARE → all VOTE YES → COMMIT
  • Blocking: coordinator failure = all locked
  • Used in distributed SQL (Google Spanner uses 2PC)
  • Avoid for microservices: too much coupling

Saga Pattern (Your Kafka Expertise)

  • Choreography: services emit events, react to others
  • Orchestration: central coordinator sends commands
  • Compensating transactions: undo on failure
  • Your CBDC: orchestrated saga for cross-border settlement

Outbox Pattern — Avoid Dual Write

Problem: write to DB + publish to Kafka — what if Kafka fails? Solution: Outbox table in same DB transaction. CDC (Debezium) reads WAL → publishes to Kafka. Atomicity within DB, eventual delivery to Kafka. This is production-grade pattern every Staff eng must know.

DAY 17 Real-Time Systems — WebSockets, SSE, Long Polling Real-Time ▼

Protocols Comparison

  • Long Polling: client waits, server holds until data. Simple, high latency
  • SSE: server → client only, HTTP/1.1 compatible. Good for feeds
  • WebSocket: full-duplex, persistent TCP. Chat, gaming, forex tickers
  • WebTransport: QUIC-based, future standard

At Scale: Connection Management

  • 10K WS connections per server (Go handles well)
  • Presence service: heartbeat + timeout tracking
  • Fan-out: user in multiple rooms → multiple subscriptions
  • Message broker as pub-sub backend (Redis PubSub, Kafka)

DAY 24 Design Uber — Real-Time Location & Matching Full HLD+LLD ▼

Location Service

DAY 25 Design URL Shortener + Distributed Cache HLD ▼

URL Shortener — Simple but Deep

Hash: MD5(url) → take first 7 chars → collision check → store. Better: base62 encoding of auto-increment ID. CDN at edge for redirect (301 permanent, 302 for analytics). Scale: 100M URLs/day → Cassandra for write throughput, Redis for hot URLs.

Distributed Cache (Redis Cluster)

Consistent hashing across nodes. Hash slots: 16384 slots distributed across nodes. CRC16(key) % 16384 → slot → node. Replication: each master has N replicas. Failure: replica promotes, clients redirect. Cache-aside pattern for application layer.

DAY 26 Design Twitter Feed — Fan-out Strategies Full HLD+LLD ▼

Fan-out on Write (Push)

  • On tweet: push to all followers' feed caches
  • O(followers) work at write time
  • Read is O(1) — just fetch pre-built feed
  • Celebrity problem: 50M followers = 50M writes

Fan-out on Read (Pull)

  • On load: fetch tweets from all followed users
  • Merge, sort, paginate at read time
  • Heavy reads, but celebrities don't cause write spikes
  • Twitter uses hybrid: push for normal users, pull for celebrities

Staff-Level Answer

Hybrid approach: classify users by follower count. Regular users (< 1M followers): fan-out on write to Redis ZSET (score=timestamp). Celebrity users: on read, merge their recent tweets with pre-built feed. This is the actual Twitter approach — mention it confidently.
