Week 1 — ML Foundations

The goal of Week 1 is to understand what machine learning actually is — not algorithmically, but philosophically. ML is the process of fitting a parameterised function to data by minimising a loss. Every algorithm you'll ever learn is a variation on that sentence. Once you see it that way, nothing will surprise you.

DAY 01 Linear Regression — The Simplest Model ▼
Core concept

A linear model predicts output ŷ = w₁x₁ + w₂x₂ + ... + b. How do we find weights that make predictions good? Define "good" as a loss function, then find parameters that minimise it. Mean Squared Error: L = (1/n)Σ(yᵢ - ŷᵢ)². Squaring penalises large errors disproportionately and keeps loss positive.

Normal Equation — closed-form solution
w = (XᵀX)⁻¹ Xᵀy

X: design matrix [n_samples × n_features]
y: target vector [n_samples]
Cost: O(m³) for matrix inversion — breaks for large m (features)
For large datasets: use gradient descent instead

Key insight

Linear regression with no regularisation will memorise training data if you have enough features. The model has learned the noise, not the signal. This is overfitting — and why we need regularisation and a held-out test set.

DAY 02 Gradient Descent — How Models Actually Learn ▼
Core concept

Compute the gradient of the loss with respect to every parameter — this tells you which direction each parameter should move to increase the loss. Move in the opposite direction by a small amount (the learning rate). Repeat until the loss stops decreasing.

Weight update rule
w ← w − α · ∂L/∂w

α = learning rate (step size)
∂L/∂w = partial derivative of loss w.r.t. weight
The gradient points toward steepest increase — move opposite

Mini-batch SGD: use 32–512 samples to estimate gradient
One pass through all data = one epoch

Learning rate — the most important hyperparameter. Too large: loss oscillates or diverges. Too small: training takes forever and may get stuck. Learning rate schedule: start large, decay over time. Cosine annealing and linear warmup are standard in deep learning.

DAY 03 Bias-Variance Tradeoff and Regularisation ▼
Core concept

Bias is error from wrong assumptions — an underfitting model that misses important patterns. Variance is error from sensitivity to training data fluctuations — an overfitting model that memorised the noise. Total error = Bias² + Variance + Irreducible noise. The irreducible noise is the floor no model can beat.

L2 (Ridge): adds λΣwᵢ² to the loss. Penalises large weights — pushes toward small, distributed weights. Shrinks weights toward zero but never to exactly zero.

L1 (Lasso): adds λΣ|wᵢ|. The gradient of |w| is ±1 regardless of w's magnitude — drives some weights to exactly zero. Performs implicit feature selection. Use when you suspect most features are irrelevant.

Dropout (neural networks): during training, randomly zero out a fraction of neurons each forward pass. Forces the network to learn redundant representations. Not used at inference time (all neurons active, weights scaled). Effective regulariser for deep networks.

Validation curve: train loss always decreases with more complexity. Validation loss decreases, reaches a minimum, then increases — this is overfitting. Optimal complexity is at the minimum of the validation curve. Always requires a held-out validation set never used for training.

DAY 04 Classification — Logistic Regression and Decision Trees ▼
Logistic Regression

Linear regression followed by a sigmoid: P(y=1|x) = σ(wᵀx + b) = 1/(1 + e⁻ᶻ). Output is a probability in (0,1). Decision boundary: predict class 1 if P > 0.5.

Cross-entropy loss: L = −[y·log(ŷ) + (1−y)·log(1−ŷ)]. When the correct class probability is near 1, loss ≈ 0. When confidently wrong (probability near 0 for the correct class), loss → ∞. This strong penalisation of confident wrong predictions trains faster and more robustly than MSE for classification.

Decision Trees and Random Forests

A tree that splits data by yes/no questions about features. At each node, find the split that maximally reduces impurity (Gini or information gain). Grown to full depth: perfectly classifies training data but memorises noise. Fix: limit depth, or use ensemble methods.

Random Forest (bagging): build many trees on bootstrapped data subsets, each using a random feature subset at each split. Average predictions. Variance of the ensemble is lower than any individual tree because errors are uncorrelated.

Gradient Boosting (XGBoost, LightGBM): build trees sequentially, each one correcting the residual errors of the previous ones. Often outperforms random forests on tabular data. Standard in competitive ML on structured/tabular datasets.

DAY 05 Evaluation Metrics — What "Good" Actually Means ▼
Beyond accuracy

Accuracy is useless for imbalanced datasets. A model that always predicts "not fraud" on 99.9% non-fraud data is 99.9% accurate and completely worthless. Use metrics that account for class imbalance and your specific cost structure (cost of false positive vs false negative).

Classification metrics
Precision = TP / (TP + FP) ← of all predicted positive, how many were correct?
Recall = TP / (TP + FN) ← of all actual positive, how many did we find?
F1 = 2·(P·R)/(P+R) ← harmonic mean

AUC-ROC: area under ROC curve (TPR vs FPR at all thresholds)
AUC-PR: area under precision-recall curve — better for imbalanced data

RMSE = √(1/n·Σ(y−ŷ)²) ← same units as target, sensitive to outliers
MAE = 1/n·Σ|y−ŷ| ← robust to outliers

Precision vs Recall tradeoff: lower decision threshold → higher recall (catch more positives), lower precision (more false positives). Spam filter: high precision (don't delete good email). Cancer screening: high recall (don't miss cancer).

K-fold cross-validation: split data into K folds, train on K−1, evaluate on the held-out fold, repeat K times. Average the K scores. More reliable estimate of generalisation than a single split, especially with small datasets.

DAY 06–07 Feature Engineering and Data Preprocessing ▼
Why feature engineering beats model choice

In most real-world ML projects, the difference between mediocre and great is feature engineering, not algorithm selection. A well-engineered feature can encode months of domain knowledge and make a linear model competitive with a complex neural network.

Normalisation/Standardisation: gradient descent converges much faster when features are on the same scale. Standardisation: (x − μ)/σ. Always fit the scaler on training data only — applying training statistics to test data is data leakage.

Categorical encoding: one-hot for nominal categories (no natural order). Ordinal for ordered categories. Target encoding: replace category with mean target value — powerful but leaks label information if not done with cross-validation. Embeddings: learned dense representations in neural networks.

Data leakage — the silent killer: any information from the test set or the future leaking into training. Common forms: fitting a scaler on the whole dataset before splitting, including a feature only available after the prediction time (e.g., "was transaction reversed?" in fraud detection), normalising using test set statistics. Produces unrealistically high validation metrics that fail completely in production.

Missing values: never silently drop rows. If missingness is informative (e.g., salary missing correlates with high income), add a binary indicator salary_was_missing and impute the value separately. Missingness Not At Random (MNAR) requires explicit modelling.

Week 2 — Deep Learning

Deep learning is linear algebra + gradient descent + a non-linear activation function + a lot of compute. Stacking layers allows the network to learn hierarchical representations — early layers learn simple features (edges, frequencies), later layers combine them into complex concepts (faces, words). You do not hand-engineer these features; the network learns them from data.

DAY 08 Neural Networks — Architecture and Forward Pass ▼
What a neural network actually is

A neural network is a function composed of simpler functions. Each layer applies an affine transformation then a non-linearity: a = σ(Wx + b). Without non-linearities, any stack of linear layers collapses to a single linear layer — depth gives no benefit. The activation function is what makes depth useful.

Common activations and when to use each
ReLU: f(x) = max(0, x) — default for hidden layers
Dying ReLU: neurons stuck at 0 if inputs are always negative
GELU: f(x) = x·Φ(x) — smooth ReLU, used in GPT/BERT
SiLU: f(x) = x·σ(x) — Swish, strong empirical performance
Sigmoid: f(x) = 1/(1+e⁻ˣ) — output ∈ (0,1), for gates/probabilities
Tanh: f(x) = (eˣ−e⁻ˣ)/(eˣ+e⁻ˣ) — saturates, rarely used in hidden layers
Softmax: normalises to probability distribution — output for classification

Universal approximation theorem: a network with a single hidden layer of sufficient width can approximate any continuous function on a compact domain. In practice, depth (many layers) generalises better than width for the same parameter count — deep representations are more efficient than wide shallow ones.

DAY 09 Backpropagation — How Gradients Flow ▼
The chain rule in computational graphs

Backpropagation is the chain rule of calculus applied to a computational graph. You compute ∂L/∂w for every parameter by walking backward through the graph, multiplying local gradients at each node. Every operation has a simple local gradient formula; the global gradient is the product of all local gradients along the path from the loss to that parameter.

Chain rule + local gradients (memorise these)
∂L/∂w = (∂L/∂output) · (∂output/∂w) ← chain rule

multiply gate: grad_self = upstream_grad × other_input
grad_other = upstream_grad × self_input
add gate: grad flows through unchanged (×1 for both inputs)
ReLU gate: grad = upstream_grad × (1 if input > 0 else 0)
tanh gate: grad = upstream_grad × (1 − tanh(x)²)

Vanishing gradients: in deep networks, gradients are multiplied through many layers. Sigmoid/tanh saturate at derivatives near 0 — gradients vanish exponentially. Early layers learn nothing. Solution: ReLU activations, residual connections, proper initialisation.

Exploding gradients: gradients grow exponentially — training diverges. Fix: gradient clipping. If the gradient norm exceeds a threshold (typically 1.0), scale it down. Standard in all transformer training.

Residual connections (ResNet): output = F(x) + x. The skip connection provides a gradient highway — gradients flow directly through addition without being multiplied by layer weights. Enables 100+ layer networks. Transformers use the same trick.

DAY 10–11 Optimisers — Adam, AdamW, and Learning Rate Schedules ▼
Why SGD alone is insufficient

Vanilla SGD applies the same learning rate to all parameters. Some parameters need large updates (rarely-seen features); some need small updates (frequently updated). The loss landscape has ravines — SGD oscillates across ravines instead of following them.

Adam optimiser — know this equation cold
m ← β₁·m + (1−β₁)·∇L first moment (momentum), β₁=0.9
v ← β₂·v + (1−β₂)·(∇L)² second moment (RMSProp), β₂=0.999
m̂ = m/(1−β₁ᵗ) bias correction (t = step number)
v̂ = v/(1−β₂ᵗ)
w ← w − α·m̂/√(v̂ + ε) update (ε=1e-8 for numerical stability)

AdamW: apply weight decay directly to weights (not via gradient)
w ← w − α·m̂/√(v̂+ε) − α·λ·w ← decoupled weight decay
Standard for transformer training. Theoretically cleaner than L2 in Adam.

Learning rate warmup: start with a very small LR for the first few hundred steps, then increase to the target LR. Without warmup, large initial updates can send you into a bad region before adaptive moments have stabilised. Essential for transformer training.

Cosine annealing: after warmup, decay the LR following a cosine curve from max to near zero. Provides smooth LR reduction without manual decay schedules. Standard in LLM training.

DAY 12–13 CNNs — Convolutional Neural Networks ▼
The core idea

A convolution is a sliding filter that scans the input with shared weights. The same filter applied at every position encodes the inductive bias that a feature useful in one region is useful everywhere (translation equivariance). Early filters detect edges; deeper filters detect textures, then parts, then objects.

Pooling: reduces spatial dimensions by taking max or average over a small window. Provides translation invariance, reduces computation in deeper layers. Max pooling retains the most prominent feature in each region.

ResNet insight: before residual connections, networks deeper than ~20 layers were harder to train than shallower ones — not just slower, actually worse test accuracy. The skip connection F(x) + x fixed this. ResNet-50 and ResNet-152 dominated ImageNet. The skip connection is the most impactful architectural innovation before attention.

Batch Normalisation

Normalise each layer's output to have zero mean and unit variance across the batch. Then apply learnable scale (γ) and shift (β). Benefits: allows higher learning rates, reduces sensitivity to initialisation, provides slight regularisation. Uses a running mean/variance at inference (not batch statistics — batch stats are unavailable). Layer Norm normalises across features instead of across the batch — preferred in transformers where batch size and sequence length vary.

DAY 14 Recurrent Networks and Their Limitations ▼
Why RNNs exist and why they were replaced

An RNN processes sequences by maintaining a hidden state: hₜ = tanh(Wₕhₜ₋₁ + Wₓxₜ + b). The same weights reused at every timestep. In theory, the hidden state can remember anything. In practice, RNNs forget most things beyond ~10-20 steps due to vanishing gradients in backpropagation through time (BPTT).

LSTM: adds a cell state — a separate highway with only additive interactions. Gates (learned sigmoid layers) control what to add, forget, and output. Can carry information across hundreds of steps. Solved the vanishing gradient problem in RNNs but not the sequential computation problem.

Why transformers replaced RNNs — two reasons: (1) RNNs are sequential — token T cannot be computed until T−1 is done, making training slow on modern GPUs. Transformers compute all positions in parallel. (2) Fixed-size hidden state is an information bottleneck — the entire sequence history must be compressed into one vector. Attention directly accesses any previous position without compression.

DAY 15 Attention — The Core Mechanism ▼
The intuition

The problem with RNNs: to answer a question about a word, you retrieve information from a compressed hidden state — an information bottleneck. Attention asks: what if each word could directly look at every other word and decide how much each one matters? Instead of a bottleneck, you have direct access to all positions.

Query, Key, Value: each token produces three vectors. Query — "what am I looking for?" Key — "what do I offer?" Value — "what do I actually return?" The attention weight between token i and j = similarity between i's Query and j's Key. Output for token i = weighted sum of all Values, weighted by how much i attended to each j.

Scaled dot-product attention — memorise this
Attention(Q, K, V) = softmax(QKᵀ / √dₖ) · V

Q [seq_len × d_k]: query vectors (W_Q projects input → query space)
K [seq_len × d_k]: key vectors
V [seq_len × d_v]: value vectors

QKᵀ [seq_len × seq_len]: all-pairs dot products (attention scores)
/√dₖ: prevent softmax saturation — dot products grow with dₖ
softmax: row-wise normalisation → attention weights (sum to 1)
· V: weighted sum of values — the output for each position

Why /√dₖ? Dot products of Q and K have variance ∝ dₖ. For large dₖ, scores are large, softmax approaches one-hot, gradients through softmax vanish. Dividing by √dₖ normalises variance to ≈1 regardless of dimension.

Causal (masked) attention: for generation (GPT), position i should only attend to j ≤ i. Set scores for j > i to −∞ before softmax → 0 weight. Allows training the full sequence in parallel while preserving autoregressive generation order.

DAY 16 Multi-Head Attention and the Full Transformer Block ▼
Multi-head attention

Single-head attention captures one type of relationship between tokens. Multi-head runs h independent attention functions in parallel — each head learns different relationships (syntactic agreement, coreference, semantic similarity). Outputs are concatenated and projected back to the model dimension.

Multi-head attention + full transformer block
MultiHead(Q,K,V) = Concat(head₁,...,headₕ)·Wᴼ
headᵢ = Attention(Q·Wᵢᵠ, K·Wᵢᴷ, V·Wᵢᵛ)

GPT-2 small: d_model=768, h=12 heads → d_k=64 per head
Total attention params: 4 × d_model²

Full transformer block (Pre-LayerNorm order):
x ← x + Dropout(MultiHeadAttn(LayerNorm(x))) ← sub-layer 1
x ← x + Dropout(FFN(LayerNorm(x))) ← sub-layer 2
FFN(x) = max(0, xW₁+b₁)W₂+b₂ [4× expansion: 768→3072→768]

Why Pre-LayerNorm? Modern models apply LayerNorm before each sub-layer (Pre-LN), not after (Post-LN as in the original paper). Pre-LN trains more stably for very deep models — gradients are better-conditioned from the start of training.

Positional encoding

Attention is permutation-equivariant — shuffling input tokens identically shuffles output. No notion of order without explicit position information. Sinusoidal: deterministic position-dependent vectors added to embeddings. Learned: treat positions as tokens, learn an embedding. RoPE (Rotary Positional Embedding): used in LLaMA, Mistral — encode position by rotating Q and K before the dot product. Relative positions encoded in the rotation angle difference. Generalises to longer sequences than seen in training.

DAY 17–18 BERT vs GPT — Encoder vs Decoder, Tokenisation ▼
The two transformer families

BERT (encoder-only): bidirectional — every token attends to every other token, no causal mask. Pre-trained with Masked Language Modelling (MLM): mask 15% of tokens, predict them. Learns rich contextual representations because it sees full context. Use for classification, NER, question answering — tasks requiring full-input understanding.

GPT (decoder-only): causal — each token only attends to previous tokens. Pre-trained with next-token prediction. Naturally suited for generation. GPT-3, LLaMA, Claude, Gemini — all use this architecture. The simplicity of the objective (predict next token) scales extremely well with compute.

T5/BART (encoder-decoder): encoder processes full input (bidirectional), decoder generates output (causal + cross-attention to encoder). Used for translation, summarisation. Largely superseded by decoder-only models at scale.

Tokenisation

BPE (Byte-Pair Encoding): start with characters. Iteratively merge the most frequent adjacent pair into a new token. Repeat until vocabulary reaches target size (32K–100K). Used by GPT-2, LLaMA 1.

SentencePiece: operates on raw bytes, language-agnostic, handles any script. Used by LLaMA 2+, T5. No dependency on pre-tokenisation whitespace rules.

Why subwords, not characters or words? Characters produce sequences 4–5× longer — attention cost scales as O(n²). Words require a massive vocabulary for all forms; rare words become OOV. Subwords balance sequence length and vocabulary coverage.

Week 4 — LLMs, Fine-Tuning, and Production

Week 4 covers what happens between "we have a pre-trained transformer" and "we have a useful product." Pre-training gives you a text predictor. Alignment makes it follow instructions. Fine-tuning specialises it. RAG grounds it in your data. Evaluation tells you if any of it worked.

DAY 22 Pre-training at Scale — Scaling Laws and Emergent Capabilities ▼
Chinchilla scaling laws

Chinchilla (Hoffmann et al., 2022) showed that optimal training roughly equally splits compute between model parameters and training tokens. Practical rule: training tokens ≈ 20 × number of parameters. GPT-3 (175B params, 300B tokens) was significantly undertrained. LLaMA 3 (8B params, 15T tokens) is heavily overtrained for inference efficiency — maximise tokens per dollar of inference, not tokens per dollar of training.

Chinchilla optimal allocation
L(N, D) ≈ A/Nᵅ + B/Dᵝ + L∞

N = parameters, D = training tokens, L∞ = irreducible loss
Optimal: N_opt ∝ C^0.5, D_opt ∝ C^0.5 (split compute 50/50)
Practical rule: D ≈ 20 × N tokens for optimal single-run training

Emergent capabilities: qualitatively new abilities appear at certain scales that were absent at smaller scales — in-context learning, chain-of-thought reasoning, multi-step arithmetic. These are not predictable by extrapolating smaller model performance. One of the most important open questions in ML.

Data quality over quantity: LLaMA 3 used 15T tokens with significantly better curation than LLaMA 2's 2T tokens. Filtering, deduplication, and quality scoring produce better models than raw web data at 10× scale. Data curation is a competitive moat.

DAY 23 RLHF and DPO — Aligning LLMs to Human Preferences ▼
Why pre-training is not enough

A pre-trained LLM predicts the most likely next token given its training distribution. It will complete "How do I make a bomb?" with information — that's what's in the training data. It has no notion of helpfulness, harmlessness, or honesty. RLHF aligns the model to behave in ways humans prefer.

The three RLHF phases

Phase 1 — Supervised Fine-Tuning (SFT): fine-tune on human-written (prompt, ideal response) pairs. Teaches the format of being a helpful assistant. Quality is bottlenecked by human-written data.

Phase 2 — Reward Model (RM): given two responses to the same prompt, humans rank which is better. Train a model on these preference rankings — it outputs a scalar score predicting human preference for a (prompt, response) pair.

Phase 3 — PPO fine-tuning: treat the LLM as a policy, reward model as the reward. Use PPO to optimise the LLM to maximise reward. A KL-divergence penalty prevents drifting too far from the SFT model — prevents degenerate "reward-hacked" outputs that score highly but are low quality.

DPO — simpler alternative

Direct Preference Optimisation (Rafailov et al., 2023) achieves similar results without training a reward model or using RL. Directly optimises the LLM on preference pairs via a reparameterised objective. More stable training, simpler implementation. Now widely used for open-source fine-tuning.

DAY 24 Fine-Tuning — Full, LoRA, and QLoRA ▼
LoRA — Low-Rank Adaptation

Key insight: weight updates during fine-tuning have low intrinsic rank. Instead of updating the full weight matrix W ∈ ℝ^(m×n), decompose the update as ΔW = BA where B ∈ ℝ^(m×r) and A ∈ ℝ^(r×n), rank r ≪ min(m,n). Only train A and B; keep W frozen.

LoRA forward pass + parameter saving
h = Wx + BAx (W frozen, only BA is trained)

B: [d_out, r] — initialised to zero
A: [r, d_in] — Gaussian initialisation
α/r scaling factor applied to ΔW

At inference: merge W' = W + BA — zero latency overhead

Parameter saving for r=16, d=4096:
LoRA: 2 × 16 × 4096 = 131,072 params per matrix
Full: 4096 × 4096 = 16,777,216 params per matrix
Saving: 128× fewer trainable params

QLoRA: quantise the base model to 4-bit (NF4), then fine-tune LoRA adapters in 16-bit. A 70B model that needs ~140GB in full precision fits in ~40GB. Makes LLM fine-tuning accessible on consumer hardware.

When to use each

Full fine-tuning: large dataset, domain-critical accuracy, compute available. LoRA: moderate dataset, limited compute, maintain base model capabilities. QLoRA: extremely limited GPU memory. Prompt engineering/few-shot: when fine-tuning isn't needed and the task can be described in-context.

DAY 25 RAG — Retrieval-Augmented Generation ▼
Why RAG exists

LLMs hallucinate because they generate the most probable continuation given training data — they have no mechanism to say "I don't know." RAG grounds the model by retrieving relevant documents at query time and providing them as context. The model's task shifts from "recall" to "synthesise the provided context." Hallucination rates drop dramatically when the answer is in the context.

The retrieval pipeline

Chunking: fixed-size (simple). Semantic boundaries (better). Parent-child: small chunks (128 tokens) for precise retrieval, return larger parent chunks (512 tokens) for generation context — best of both worlds.

Embedding + vector search: encode chunks as dense vectors (text-embedding-3-large, BGE, E5). Store in vector DB (pgvector, Pinecone, Weaviate). Query: embed the question, find nearest vectors by cosine similarity.

Hybrid search: dense (ANN semantic) + sparse (BM25 keyword) combined with RRF. "GDPR Article 17" needs exact keyword matching; "rules about deleting user data" needs semantic search. Hybrid serves both.

Reranking: retrieve top-50 via hybrid search, rerank with a cross-encoder (slower but more accurate relevance scoring), return top-5 to the LLM. Cross-encoders jointly encode (query, document) — better relevance but too slow for first-pass retrieval.

RAGAS evaluation: Faithfulness (is the answer grounded in context?), Answer Relevance, Context Recall, Context Precision. Build a golden Q&A dataset of ~100 questions. Run after every pipeline change — catch regressions before they reach users.

DAY 26 LLM Inference — KV Cache, Quantisation, vLLM ▼
KV Cache — why inference is fast

During autoregressive generation, each new token needs attention with all previous tokens. Without caching: recompute K and V for all previous positions at every step — O(T²) total compute. The KV cache stores computed K and V tensors for all previous positions. Each step only computes K and V for the new token and appends. O(T) compute per step instead of O(T²).

Memory cost: 2 × layers × heads × d_k × seq_len × bytes_per_element. For LLaMA-3-70B at 128K context: ~136GB for the KV cache alone — more than the model weights.

Quantisation

INT8 (W8A8): weights and activations in 8-bit integers. ~2× speedup, ~2× memory reduction. Minimal quality loss with proper calibration.

INT4 (GPTQ, AWQ): weights in 4-bit. ~4× memory reduction. AWQ: identify and preserve the ~1% of weights that matter most (salient), quantise the rest aggressively. Good quality for 7B–70B models.

Speculative decoding: small draft model generates k candidate tokens, large model verifies all k in parallel. If draft tokens are accepted, get k tokens in the time of one large model forward pass. Typical 2–4× latency speedup with identical output quality.

vLLM and continuous batching

PagedAttention (vLLM): KV cache managed like virtual memory — allocated on demand, not pre-allocated. Eliminates fragmentation that wasted 60–80% of KV cache memory. 2–4× higher throughput than naive serving on the same hardware.

Continuous batching: when one sequence finishes, a new one fills its slot immediately — no waiting for a full batch. Dramatically improves GPU utilisation and throughput.

DAY 27–28 Evaluation, Safety, and Production Considerations ▼
Evaluation is harder than training

Evaluating open-ended text generation is genuinely hard. BLEU and ROUGE (n-gram overlap) correlate poorly with human quality judgements. LLM-as-judge (use a stronger model to score outputs) is state-of-the-art but expensive and biased toward the judge model's preferences.

Benchmarks: MMLU (57-subject multiple choice, knowledge), HumanEval (coding), GSM8K (grade school math), MATH (competition math), TruthfulQA (factuality), HellaSwag (commonsense). All have contamination risks. Treat as lower bounds on capability, not ground truth.

Production evals > benchmark evals always: task-specific golden datasets, human preference ratings via A/B testing, hallucination rate on known-answer queries, latency at p50/p95/p99, cost per 1K tokens.

Safety and production issues

Prompt injection: attacker embeds instructions in user content that override the system prompt. In RAG: a retrieved document contains "Ignore all previous instructions..." Mitigation: clear delineation between system and user content, sandboxed tool execution, output validation.

Lost in the middle: LLMs attend to beginning and end of context better than the middle. Order retrieved chunks with most relevant first and last. For very long contexts, consider hierarchical summarisation.

Observability: log every prompt and completion (with PII handling), trace token counts and latency per request, monitor for anomalous outputs, track user feedback signals. Without logging, you cannot debug, improve, or detect abuse.

Build micrograd — Autograd from Scratch

Karpathy's micrograd is a ~150-line implementation of backpropagation. Building it is more valuable than reading 10 blog posts about backprop. After you build it, PyTorch's .backward() is no longer magic — it's just the chain rule on a computational graph.

python — micrograd Value class (core autograd engine)
class Value:
  """Scalar with gradient tracking — atom of autograd."""
  def __init__(self, data, _children=(), _op=''):
    self.data = data
    self.grad = 0.0
    self._backward = lambda: None
    self._prev = set(_children)

  def __add__(self, other):
    out = Value(self.data + other.data, (self, other), '+')
    def _backward():
      self.grad  += out.grad   # ∂out/∂self = 1
      other.grad += out.grad   # ∂out/∂other = 1
    out._backward = _backward
    return out

  def __mul__(self, other):
    out = Value(self.data * other.data, (self, other), '*')
    def _backward():
      self.grad  += other.data * out.grad  # ∂out/∂self = other
      other.grad += self.data  * out.grad  # ∂out/∂other = self
    out._backward = _backward
    return out

  def tanh(self):
    t = (math.exp(2*self.data)-1)/(math.exp(2*self.data)+1)
    out = Value(t, (self,), 'tanh')
    def _backward():
      self.grad += (1 - t**2) * out.grad  # dtanh/dx = 1 - tanh²(x)
    out._backward = _backward
    return out

  def backward(self):
    # Topological sort → call _backward() in reverse order
    topo, visited = [], set()
    def build(v):
      if v not in visited:
        visited.add(v)
        for c in v._prev: build(c)
        topo.append(v)
    build(self)
    self.grad = 1.0
    for v in reversed(topo): v._backward()

# Same API as PyTorch tensors
a = Value(2.0)
b = Value(-3.0)
c = a * b + Value(1.0)
c.backward()
print(a.grad)  # -3.0
print(b.grad)  # 2.0

Build nanoGPT — Transformer from Scratch

nanoGPT is ~300 lines of PyTorch. By the end you will have trained a language model that generates Shakespeare from a blank canvas. Every line will make sense.

python — nanoGPT CausalSelfAttention (the core)
class CausalSelfAttention(nn.Module):
  def __init__(self, config):
    super().__init__()
    assert config.n_embd % config.n_head == 0
    self.n_head = config.n_head
    self.n_embd = config.n_embd
    self.c_attn = nn.Linear(config.n_embd, 3 * config.n_embd)
    self.c_proj = nn.Linear(config.n_embd, config.n_embd)
    self.attn_drop = nn.Dropout(config.attn_pdrop)
    self.resid_drop = nn.Dropout(config.resid_pdrop)
    # Causal mask: token i only attends to j <= i
    self.register_buffer("bias",
      torch.tril(torch.ones(config.block_size, config.block_size))
      .view(1, 1, config.block_size, config.block_size))

  def forward(self, x):
    B, T, C = x.size()
    q, k, v = self.c_attn(x).split(self.n_embd, dim=2)
    hs = C // self.n_head
    q = q.view(B, T, self.n_head, hs).transpose(1, 2)
    k = k.view(B, T, self.n_head, hs).transpose(1, 2)
    v = v.view(B, T, self.n_head, hs).transpose(1, 2)
    att = (q @ k.transpose(-2, -1)) * (1.0 / math.sqrt(hs))
    att = att.masked_fill(self.bias[:,:,:T,:T] == 0, float('-inf'))
    att = F.softmax(att, dim=-1)
    att = self.attn_drop(att)
    y = att @ v
    y = y.transpose(1, 2).contiguous().view(B, T, C)
    return self.resid_drop(self.c_proj(y))

The complete nanoGPT: a Block (LayerNorm → CausalSelfAttention → residual, LayerNorm → MLP → residual), a GPT class (token embed + pos embed + N blocks + final LN + lm_head), and the training loop. Build it yourself; use Karpathy's GitHub only as a reference when stuck.

MLOps — From Notebook to Production

A model that only runs in a Jupyter notebook is not a product. MLOps is the engineering practice of training, evaluating, deploying, and monitoring ML models in production systems — where your distributed systems knowledge applies directly.

Experiment Tracking

Log every experiment: hyperparameters, dataset version, model architecture, training curves, evaluation metrics, hardware used. Tools: Weights & Biases (W&B), MLflow, ClearML. Without experiment tracking, you cannot reproduce a result from last week or explain why the production model performs the way it does. Start logging from experiment one.

Data and Model Versioning

A model is a function of its data + code + hyperparameters. Use DVC (Data Version Control) to version large datasets alongside code in git. Version models in a model registry (MLflow, W&B Artifacts). Pin the exact dataset version, code commit, and hyperparameters that produced the production model. Reproducibility is a requirement, not a nice-to-have.

Training Infrastructure

Data parallelism (DDP): split the batch across GPUs, compute gradients independently, all-reduce to synchronise. Each GPU has a full model copy. Scales with batch size.

Mixed precision (BF16/FP16): store weights in FP32, compute forward/backward in BF16/FP16. Halves memory bandwidth usage, uses GPU tensor cores. BF16 preferred for training — same range as FP32 (less overflow risk), lower precision acceptable for gradients.

Model parallelism: required when the model doesn't fit on a single GPU. Pipeline parallelism splits layers across GPUs. Tensor parallelism splits individual weight matrices. DeepSpeed ZeRO Stage 3 and Megatron-LM implement this for 70B+ model training.

Monitoring in Production

Data drift: the distribution of inputs at inference time drifts from training distribution — model degrades silently. Monitor: feature distributions, embedding distances, model confidence scores.

Concept drift: the relationship between inputs and outputs changes over time. Requires periodic retraining. Harder to detect than data drift.

CI/CD for ML: every code change triggers automated training, evaluation against the test set, comparison to the production baseline. If the new model beats the baseline on all key metrics, it's promoted to staging. Same pipeline pattern as software — just with models as the artifact.

For engineers from software

MLOps is where your software engineering skills are most directly applicable. Git, CI/CD, containerisation, monitoring, alerting — all the same tools and patterns. The difference: the artifact you're testing and deploying is a model, and "correctness" is probabilistic, not deterministic. Your instinct for reliable systems is directly transferable.

Why does gradient descent use mini-batches instead of the full dataset?
Full-batch is expensive for large datasets and produces exact but slow gradients. Single-sample SGD is fast but noisy. Mini-batch (32–512) is the sweet spot: fast enough for many steps per epoch, gradient estimate good enough to converge, and naturally parallelisable on GPU hardware. The noise in mini-batch gradients also helps escape sharp local minima.
L1 vs L2 regularisation — which performs feature selection and why?
L1 (Lasso) adds λΣ|wᵢ| — the gradient of |w| is ±1 regardless of w's magnitude, so small weights are pushed to exactly zero. L2 (Ridge) adds λΣwᵢ² — gradient is proportional to w, so small weights shrink but never reach exactly zero. L1 performs implicit feature selection; L2 distributes weight across all features.
What is the vanishing gradient problem? Three solutions.
In deep networks, gradients shrink exponentially through layers as sigmoid/tanh saturate at near-zero derivatives. Early layers learn almost nothing. Solutions: (1) ReLU — gradient is 1 for positive inputs, doesn't shrink; (2) Residual connections — gradient flows directly through the skip, not multiplied by layer weights; (3) Layer/Batch Normalisation — keeps activations in a healthy range at every layer.
Scaled dot-product attention: why divide by √dₖ?
Dot products of Q and K have variance proportional to dₖ. For large dₖ, scores are large, softmax saturates (near-one-hot distribution), gradients through softmax vanish. Dividing by √dₖ normalises the variance to ≈1 regardless of dimension, maintaining healthy gradient flow through the attention mechanism.
BERT vs GPT — architecture and use cases
BERT: encoder-only, bidirectional attention (every token sees all others), pre-trained with masked language modelling. Use for classification, NER, QA — full-input understanding tasks. GPT: decoder-only, causal attention (token sees only past), pre-trained with next-token prediction. Use for generation, in-context learning, instruction following, and anything that benefits from scale.
What is LoRA and what's the parameter saving for rank-16 on a 4096-dim layer?
LoRA decomposes weight updates ΔW = BA where B is [d, r] and A is [r, d]. Only A and B are trained; W is frozen. For rank=16, d=4096: LoRA = 2×16×4096 = 131,072 parameters. Full = 4096×4096 = 16,777,216. That's a 128× reduction in trainable parameters per matrix.
What is the KV cache and why does generation need it?
During autoregressive generation, each new token needs attention with all previous tokens. Without caching: recompute K and V for all previous positions every step — O(T²) total. The KV cache stores computed K and V for all previous positions. Each step only computes K and V for the new token. O(T) compute per step. Without it, generation of a 1000-token sequence would be 1000× slower.
RLHF — three phases from memory
(1) SFT: fine-tune on human-written (prompt, ideal response) pairs — teaches the format of helpful assistant. (2) Reward Model: train a model on human preference rankings to output a scalar score for (prompt, response) pairs. (3) PPO: treat LLM as policy, RM as reward function — optimise LLM to maximise reward with KL penalty to prevent reward hacking (drifting too far from SFT model).
What is data leakage and why is it dangerous?
Information from the test set or from the future leaking into training. Produces unrealistically high validation metrics that fail in production. Common forms: fitting a scaler on the full dataset before splitting, including a feature only available after the prediction time (e.g., "was transaction reversed?" in fraud detection), normalising with test set statistics. The model appears to work but has memorised information it wouldn't have at inference time.
Why did transformers replace RNNs? Two reasons.
(1) Parallelism: RNNs are sequential — token T must finish before T+1 starts. Transformers compute all positions simultaneously via attention — training is dramatically faster on modern hardware. (2) Long-range dependencies: RNNs compress all history into a fixed hidden state (information bottleneck). Transformers directly attend to any position with O(1) path length — distant tokens are as accessible as adjacent ones.
What are Chinchilla scaling laws?
Optimal training splits compute ~50/50 between model size (N) and training tokens (D). Practical rule: D ≈ 20×N. GPT-3 (175B params, 300B tokens) was compute-suboptimal — too large, too few tokens. LLaMA-3-8B (15T tokens) is heavily overtrained for inference efficiency — you get the same quality at smaller model size, which means cheaper inference. Matters for deciding model vs data tradeoff given a compute budget.
What is RAG and why does it reduce hallucinations?
Retrieval-Augmented Generation: retrieve relevant documents at query time, provide as context to the LLM. The model's task shifts from "recall from training memory" to "synthesise the provided context." Hallucinations drop because the model doesn't need to invent information — it extracts it from documents in the prompt. Doesn't eliminate hallucinations (model can still misread context) but significantly reduces them for knowledge-intensive tasks.
