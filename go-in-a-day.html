01

GMP Scheduler Architecture

Most engineers know Go uses goroutines and that they're "cheap." What most don't know is why they're cheap, and what the runtime has to do when things get complicated — blocking syscalls, tight CPU loops, preemption. Understanding the GMP model changes how you write concurrent code and how you read scheduler traces.

The three actors

G (Goroutine) — the logical unit of work. Starts with a 2KB stack (down from 8KB in Go 1.3). Contains a program counter, stack pointer, and goroutine-local state. Goroutines are cheap to create because they start small and grow by copying.

M (Machine) — an OS thread. The actual CPU executor. An M needs a P to run goroutines. Without a P, an M can exist (to execute certain runtime operations) but cannot run user code. Go typically has a pool of Ms — their count isn't bounded by GOMAXPROCS.

P (Processor) — the scheduler context. Think of P as a permit: an M must hold a P to execute user goroutines. Each P has its own local run queue (max 256 goroutines). GOMAXPROCS controls the number of Ps, which sets the hard ceiling on true parallelism. The default is runtime.NumCPU().

Mental Model

P is a permit, not a CPU. M is a thread, not a core. G is a task, not a thread. The scheduler's job is to keep Ps busy by matching them with Ms that have runnable Gs.

Work stealing and the global queue

When a P's local queue is empty, it first checks the global run queue (accessible by all Ps), then steals half of another P's local queue. This work-stealing approach is why Go programs can efficiently utilize all cores even when work is unevenly distributed — no central lock on the hot path.

The global queue is also where goroutines land when they're newly created and no P has claimed them, and where goroutines are placed when their M releases a P (on syscall entry). The scheduler checks the global queue approximately every 61 local-queue iterations to prevent starvation.

Syscall handling — the clever part

When a goroutine makes a blocking syscall, the following happens: the M executing it can't run other goroutines while blocked. So the runtime detaches the P from that M, allowing another M (idle or newly created) to pick up the P and continue running other goroutines. When the syscall completes, the original M tries to reacquire a P. If all Ps are busy, the M parks itself and the goroutine joins the global queue.

This is how Go handles 10,000 concurrent database calls without 10,000 OS threads. Most of those goroutines are blocked in the syscall layer; the number of active Ps (and thus active OS threads doing real work) stays at GOMAXPROCS.

Preemption — the Go 1.14 change

Before Go 1.14, goroutines could only yield at cooperative preemption points: channel operations, function calls, runtime.Gosched(). A tight compute loop — something like a pure number-crunching function with no function calls — could starve other goroutines on the same P.

Go 1.14 added signal-based asynchronous preemption. The runtime sends SIGURG to a thread running a goroutine for too long. The signal handler triggers a yield at the next safe point. This means you no longer need to manually yield in compute-intensive goroutines — but it also means that tight loops now have invisible overhead in the form of signal handling. For latency-critical code, this matters.

shell — diagnose scheduler 
# Scheduler stats every 1000ms. Shows P count, idle threads,
# spinning threads, goroutines in global queue, per-P queue sizes.
GODEBUG=schedtrace=1000 ./yourservice

# Example output:
# SCHED 1000ms: gomaxprocs=8 idleprocs=2 threads=12 spinningthreads=1
# runqueue=3 [4 2 0 1 0 3 2 1]  ← per-P local queue depths

# Key signals to look for:
# runqueue growing → goroutines created faster than consumed
# spinningthreads high → M threads burning CPU looking for work
# threads >> gomaxprocs → many goroutines blocking in syscalls

GODEBUG=scheddetail=1 GODEBUG=schedtrace=100 ./yourservice

Tricolor GC and Memory Tuning

Go's garbage collector is concurrent — it runs alongside your goroutines. Understanding how it works changes three things: how you interpret latency spikes, how you write allocations-sensitive code, and how you tune GOGC and GOMEMLIMIT for production environments like Kubernetes.

The tricolor marking algorithm

The GC uses a tricolor abstraction over the heap. Every object is in one of three states: white (not yet visited — a candidate for collection), gray (reachable, but its children haven't been scanned), or black (reachable, all children scanned — safe). The algorithm starts by coloring all roots gray (globals, stack variables, goroutine stacks). It then processes grays, turning them black while graying their children. When no gray objects remain, everything still white is unreachable — it gets swept.

The hard part: this runs concurrently with goroutines that are actively mutating the heap. A goroutine could create a new pointer from a black object to a white object — the GC would miss the white object and incorrectly collect it. The write barrier prevents this: every heap pointer write during a GC cycle also notifies the GC, keeping the invariant intact.

Escape analysis — the compiler's first pass

Before the GC even runs, the compiler decides at build time where each variable lives. Stack allocation is nearly free — it's a pointer increment and decrement, no GC involvement. Heap allocation costs memory, adds GC work, and causes cache pressure. The compiler performs escape analysis to determine if a variable can stay on the stack.

A variable escapes to the heap when: its address outlives the function that created it (you returned a pointer), it's too large for the stack, or it was passed to a function accepting interface{} or any — the compiler can't know the concrete size, so it heap-allocates. That last one is subtle. fmt.Println(userID) causes userID to escape. In a tight loop processing millions of requests, this matters.

shell — inspect escape analysis 
# -m shows escape analysis decisions. -m -m shows more detail.
go build -gcflags="-m" main.go

# You'll see lines like:
# ./main.go:14:6: moved to heap: userID     ← escapes, GC pressure
# ./main.go:20:9: &user does not escape      ← stays on stack, fast

# Common escape causes to audit:
# - any interface{} / any conversion in hot path
# - returning a pointer to a local struct
# - closures capturing local variables (captured var escapes)
# - slice backing arrays that grow past initial capacity

GOGC vs GOMEMLIMIT — which to use in 2026

GOGC=100 (the default) means: trigger the next GC when live heap has grown by 100% since the last collection. If your live heap is 100MB, the next GC fires at 200MB. Lower GOGC = more frequent GC, lower peak memory, higher CPU overhead. Higher GOGC = rarer GC, more memory used, lower CPU impact.

Go 1.19 added GOMEMLIMIT — a soft memory ceiling. When heap usage approaches the limit, the runtime becomes more aggressive about collecting. This is almost always the right tool for containerized workloads: set GOMEMLIMIT to 80–90% of your container's memory limit, and the GC will automatically trade CPU for lower memory pressure near that ceiling rather than OOMing.

Production Rule

For Kubernetes deployments: set GOMEMLIMIT to 85% of resources.limits.memory. This aligns the GC's understanding of the memory envelope with the container runtime's hard limit, preventing OOM kills that look like random crashes.

Channel and Sync Internals

Channels are not magic. They're a struct with a circular buffer, a mutex, and two queues — one for waiting senders, one for waiting receivers. Understanding the implementation makes every channel behavior predictable rather than mysterious.

The hchan struct

Internally, every channel is an hchan: a circular ring buffer of elements (for buffered channels), a count of elements currently in the buffer, a mutex protecting all state, a queue of goroutines waiting to send, and a queue of goroutines waiting to receive.

For a buffered channel with space: sending copies directly into the buffer. No goroutine interaction, just a copy and a mutex. For a full buffered channel: the sender's goroutine is added to the send queue and parked — suspended by the scheduler. The goroutine won't run again until a receiver takes an element and wakes it up by copying the sender's value directly into the receiver's stack (skipping the buffer entirely for efficiency).

For an unbuffered channel: there is no buffer. A sender parks immediately unless a receiver is already waiting. When a receiver is waiting, the send is a direct copy from sender's stack to receiver's stack — a rendezvous. This is the "happens-before" guarantee you get with unbuffered channels.

go — channel and select patterns 
// Closed channel behavior — essential to know
ch := make(chan int)
close(ch)

val, ok := <-ch  // ok=false, val=0 (zero value)
val  = <-ch     // returns zero value, never blocks
                 // ranging over closed channel drains then exits

// select with default — non-blocking receive
select {
case msg := <-workCh:
    process(msg)
case <-ctx.Done():
    return ctx.Err()
default:
    // channel empty, continue without blocking
}

// When multiple cases are ready, select picks pseudo-randomly.
// This is NOT starvation-free — a busy channel can dominate.
// If fairness matters, use a priority select pattern with
// a drain-first approach for the high-priority channel.

// Context cancellation propagation — the idiomatic pattern
func worker(ctx context.Context, jobs <-chan Job) error {
    for {
        select {
        case <-ctx.Done():
            return ctx.Err()
        case j, ok := <-jobs:
            if !ok {
                return nil  // channel closed, clean exit
            }
            if err := process(ctx, j); err != nil {
                return err
            }
        }
    }
}

sync.Pool — object reuse without custom free lists

sync.Pool is a cache for temporary objects. It dramatically reduces GC pressure for allocations in hot paths. The critical guarantee to understand: pooled objects may be evicted at any GC cycle. A Pool is not a free list — you cannot rely on an object surviving across a GC. Always call Reset() (or clear fields manually) before reusing a pooled object, because the previous user may have left state in it.

go — sync.Pool for zero-allocation hot path 
var bufPool = sync.Pool{
    New: func() any {
        // Called when pool is empty. This is the only allocation path.
        return bytes.NewBuffer(make([]byte, 0, 512))
    },
}

func serializeNotification(n Notification) ([]byte, error) {
    buf := bufPool.Get().(*bytes.Buffer)
    buf.Reset() // critical: clear previous caller's data
    defer bufPool.Put(buf)

    // ... write to buf ...

    result := make([]byte, buf.Len())
    copy(result, buf.Bytes())
    return result, nil
    // Don't return buf.Bytes() directly — buf goes back to pool
}

Interfaces and Go's OOP Model

Go doesn't have classes. It has interfaces, and the decision to use structural typing (implicit satisfaction rather than explicit implements) changes everything about how you design systems. The best Go code reads like descriptions of behavior boundaries, not class hierarchies.

Interface internals — the two-word header

Every interface value is two words: a pointer to the type information (the itab, containing the interface type, the concrete type, and a table of method pointers) and a pointer to the data. A type assertion checks the itab's concrete type against the expected type. A nil interface has both words as nil. An interface holding a nil pointer has a non-nil itab word and a nil data word — this is why the nil interface trap exists.

go — the nil interface trap 
type MyError struct{ msg string }
func (e *MyError) Error() string { return e.msg }

func doWork() error {
    var err *MyError = nil  // typed nil pointer
    return err               // ← wraps in interface: (type=*MyError, value=nil)
}

func main() {
    err := doWork()
    if err != nil {       // TRUE — interface is not nil!
        panic(err)         // panics on "nil" error
    }
}

// Fix: return the untyped nil directly
func doWorkFixed() error {
    var err *MyError = nil
    if err == nil {
        return nil  // untyped nil → interface{type=nil, value=nil} → err == nil is TRUE
    }
    return err
}

// Rule: never return a typed nil pointer as an error interface.
// This bites every Go developer once. It bites middleware authors twice.

Idiomatic interface design

The Go proverb "accept interfaces, return concrete types" exists for a reason. When you return an interface from a function, you take away the caller's ability to access concrete type methods and you make mocking unnecessary indirection. When you accept an interface, you allow any type satisfying the contract to be passed — including test fakes.

Define interfaces at the consumer, not the producer. If your UserService needs to store users, define the Store interface in the userservice package — not in the postgres package. This prevents import cycles and makes the dependency direction explicit: the service defines what it needs, and any implementation satisfying that contract can plug in.

go — interface at the consumer boundary 
// In package userservice — consumer defines the contract
type Store interface {
    Save(ctx context.Context, u User) error
    ByID(ctx context.Context, id string) (User, error)
}

type Service struct { store Store }

func New(s Store) *Service { return &Service{store: s} }

func (s *Service) Register(ctx context.Context, u User) error {
    if u.Email == "" {
        return fmt.Errorf("email required")
    }
    return s.store.Save(ctx, u)
}

// In package postgres — satisfies Store without knowing about it
type Repo struct { db *sql.DB }

func (r *Repo) Save(ctx context.Context, u userservice.User) error {
    _, err := r.db.ExecContext(ctx, `INSERT INTO users ...`, u.ID, u.Email)
    return err
}

// Compile-time interface check — catch mismatches at build, not runtime
var _ userservice.Store = (*Repo)(nil)

pprof and trace — Profiling in Production

Every Go service should have pprof enabled in production behind an authenticated endpoint. Not for debugging convenience — because you will need it at 3AM and you don't want to redeploy to enable it. The data pprof provides — CPU profiles, heap snapshots, goroutine dumps, execution traces — cannot be easily reconstructed from metrics alone.

go — enable pprof in your service 
import (
    _ "net/http/pprof"  // side-effect import registers handlers
    "net/http"
)

func main() {
    // Separate server — never expose on public port
    go http.ListenAndServe("localhost:6060", nil)
    // ... rest of service startup
}

Profiling workflows for common failure modes

Symptom Profile to collect Command
CPU spiking, slow requests CPU profile — 30s sample go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30
Memory growing, suspected leak Heap profile — live objects go tool pprof http://localhost:6060/debug/pprof/heap
Goroutine count growing Goroutine dump curl localhost:6060/debug/pprof/goroutine?debug=1
Latency spikes, GC suspected Execution trace — 5s curl localhost:6060/debug/pprof/trace?seconds=5 > t.out && go tool trace t.out
Allocation hotspots Allocs profile go tool pprof http://localhost:6060/debug/pprof/allocs

Reading a heap profile

In the pprof interactive shell, top shows the functions allocating the most. web (requires Graphviz) renders a callgraph — each node is sized proportionally to its allocation share. Look for unexpectedly large nodes in hot paths. list FunctionName shows the source with per-line allocation costs annotated. The most common finding: an interface conversion in a tight loop, a fmt.Sprintf in a logging hot path, or a map lookup that could be a slice lookup.

For your notification service at 10K+ alerts/sec

Start with the CPU profile. Find the hot functions. If you see latency spikes rather than sustained CPU, collect an execution trace — GC pause events show up clearly. If goroutine count is growing linearly with traffic, you have a leak: look at the goroutine dump for goroutines blocked in the same place.

Generics — Patterns and When Not to Use Them

Go generics (added in 1.18) solve a specific problem: you want a function or data structure to work across multiple types without losing type safety or resorting to any. But they're not a replacement for interfaces, and using them where interfaces are correct creates unnecessary complexity.

go — type constraints and generic data structures 
// Type constraint with tilde — includes derived types
type Number interface {
    ~int | ~int64 | ~float64
    // ~ means: any type whose underlying type is int/int64/float64
    // So type Celsius float64 satisfies ~float64
}

func Sum[T Number](nums []T) T {
    var total T
    for _, n := range nums { total += n }
    return total
}

// Generic stack — replaces []interface{} with type safety
type Stack[T any] struct{ items []T }

func (s *Stack[T]) Push(item T)          { s.items = append(s.items, item) }
func (s *Stack[T]) Len()  int            { return len(s.items) }
func (s *Stack[T]) Pop()  (T, bool) {
    if len(s.items) == 0 { var zero T; return zero, false }
    n := len(s.items) - 1
    item := s.items[n]
    s.items = s.items[:n]
    return item, true
}

// When NOT to use generics:
// 1. The function works fine with an interface{} (just needs to call methods)
// 2. The type set is open (any user type can implement the interface)
// 3. You need runtime polymorphism (different types in the same collection)
//    → use interface, not generics
// Use generics for: closed type sets, container types, utility functions
// where the type is fixed at call site (like slices.Map, slices.Filter)

Performance Traps and How to Avoid Them

These aren't theoretical. Each of these is a class of allocation or performance mistake that shows up in pprof data in real production services.

go — the five most common allocation mistakes 
// 1. String concatenation in a loop — O(n²) allocations
var s string
for _, v := range items { s += v }  // BAD: new allocation per iteration

var b strings.Builder
b.Grow(estimatedSize())  // pre-allocate if you know the size
for _, v := range items { b.WriteString(v) }
s = b.String()             // GOOD: one allocation at the end

// 2. interface{} in a hot path — causes heap escape
fmt.Sprintf("user %d processed", userID)  // userID escapes to heap

// Use strconv for numeric formatting in tight paths
var buf [32]byte
b := strconv.AppendInt(buf[:0], userID, 10)  // zero allocation

// 3. Slice growing without pre-allocation
result := []Item{}
for _, r := range rows { result = append(result, convert(r)) }

result := make([]Item, 0, len(rows))  // pre-allocate: one allocation total
for _, r := range rows { result = append(result, convert(r)) }

// 4. Map key allocation with struct keys
// map[string]V: every key is a heap string. Consider [16]byte keys
// for fixed-size keys (UUIDs, IPs). Avoids per-lookup allocation.

// 5. Goroutine leak — goroutines are cheap but not free
// Every goroutine is at minimum 2KB of stack.
// Always give goroutines a way to exit: done channel or context.
go func() {
    for {
        select {
        case <-ctx.Done(): return   // clean exit path always present
        case work := <-queue: handle(work)
        }
    }
}()

Quick Revision — Flash Cards

Close your notes. Answer these from memory. If you can't answer one cleanly, that's the concept to revisit before your next session.

What is a P in the GMP model, and why does it matter?
P is a logical processor — a scheduler context that holds a local run queue of up to 256 goroutines. An M (OS thread) needs to hold a P to run user code. GOMAXPROCS = number of Ps = parallelism ceiling. Idle Ms work-steal from busy Ps' queues.
What happens when a goroutine makes a blocking syscall?
The M executing the goroutine releases its P. Another M (idle or newly created) picks up the P and continues running other goroutines. When the syscall returns, the original M tries to reacquire a P. If none available, it parks; the goroutine joins the global run queue.
GOGC vs GOMEMLIMIT — when do you use each?
GOGC is ratio-based: GC triggers when heap grows by GOGC% since last collection. GOMEMLIMIT (Go 1.19+) is an absolute soft ceiling — GC becomes aggressive near it. For K8s: use GOMEMLIMIT set to 85% of resources.limits.memory. Aligns GC with container runtime limits.
The nil interface trap — explain and fix it.
An interface has two words: type and value. A nil interface has both nil. A typed nil (e.g. var e *MyError = nil returned as error) has a non-nil type word → interface is not nil. Fix: return bare nil for error returns, never a typed nil pointer.
When does a goroutine stack grow, and what does that cost?
On function call, the runtime checks a stack guard. If stack is full, a new larger stack is allocated, all contents are copied, all interior pointers are updated. Expensive in deep recursion. Stacks also shrink at GC if usage is much less than allocated. Default max: 1GB.
Pointer vs value receiver — what's the rule?
Use a pointer receiver when: the method mutates state, the struct contains a mutex or sync primitive, or the struct is large enough that copying is expensive. Use value receiver when: the type is small and immutable-like. Be consistent across all methods of a type. Inconsistency breaks interface satisfaction rules.
What does sync.Pool guarantee — and what does it not guarantee?
Pool is a cache, not a free list. Pooled objects may be evicted at any GC cycle. Cannot rely on object surviving across GCs. Always Reset() before reuse. The guarantee: if an object is in the pool between GCs, you'll likely get it back — reducing allocations in hot paths. Under heavy GC pressure, pool may be drained completely.
What triggers signal-based preemption, and when was it added?
Added in Go 1.14. The runtime sends SIGURG to a thread running a goroutine for too long (determined by the sysmon goroutine). The signal handler marks the goroutine for preemption at the next safe point. Prevents tight CPU loops from starving other goroutines on the same P.
