01

Processes, Threads, and Scheduling

A process is an isolated execution environment — its own virtual address space, file descriptors, and resource limits. A thread shares the address space of its process, which makes communication cheap but introduces shared state. The OS scheduler decides which thread runs on which CPU core at any moment, switching between them at intervals or on blocking events.

Context switching — the real cost

A context switch involves saving the current thread's CPU registers (program counter, stack pointer, general-purpose registers), updating kernel bookkeeping structures, selecting the next thread, and loading its saved state. This direct overhead is 2–10 microseconds on modern hardware.

The indirect cost is more significant: the switched-out thread's data is likely warm in L1/L2 cache. The new thread's working set will cause cache misses — potentially hundreds of nanoseconds each — until its data loads. For a service doing I/O-bound work with many short-lived goroutines, this cache pressure is often the dominant performance factor, not CPU or I/O.

Go's GMP scheduler minimises OS-level context switching by keeping goroutine switching in user space. The M (OS thread) stays scheduled on its CPU core while the Go scheduler switches which G (goroutine) runs on it — no kernel trap, no register dump to kernel memory.

Scheduling policies

CFS (Completely Fair Scheduler): Linux default. Tracks virtual runtime per process/thread, always runs the task with the lowest vruntime. Implemented with a red-black tree — O(log n) insert and selection. Fair, but doesn't account for priority beyond nice values.

Real-time schedulers (SCHED_FIFO, SCHED_RR): for latency-critical tasks. FIFO runs until blocked; RR has time quanta. Used by audio/video systems, robotics, kernel drivers. If you're building a latency-SLA service, understanding when the OS preempts your process for a higher-priority task matters.

02

Virtual Memory, Paging, and TLB

Every process operates in a virtual address space — a contiguous range of addresses (0 to 2⁴⁸ on x86-64) that maps to physical RAM (and swap) via page tables. The hardware Memory Management Unit (MMU) performs this translation on every memory access. The page table is a multi-level tree (4 levels on x86-64) — each level indexed by a portion of the virtual address.

TLB — the cache for page table lookups

A full 4-level page table walk is 4 memory accesses — expensive at 100ns each. The Translation Lookaside Buffer (TLB) caches recent virtual→physical translations. A TLB hit is a few nanoseconds; a TLB miss triggers a full page table walk (or a page fault if the page isn't in RAM at all).

TLB pressure is a real performance concern for Go services. A goroutine with a large heap spread across many pages will cause more TLB misses than one with a compact working set. GODEBUG=memstats=1 shows heap fragmentation; perf stat -e dTLB-load-misses measures TLB miss rate directly.

03

I/O Models — Blocking, Non-Blocking, epoll, io_uring

The evolution from blocking I/O to io_uring is a story about syscall overhead and kernel/userspace boundary crossings.

Blocking I/O

The thread issues a read() syscall and blocks until data is available. Simple programming model, but one thread per connection — doesn't scale past tens of thousands of connections on most systems.

epoll — the Linux solution for C10K

epoll allows a single thread to monitor thousands of file descriptors. You register descriptors with epoll_ctl, then call epoll_wait which returns only the descriptors that are ready. Edge-triggered mode (EPOLLET) fires only on state transitions — more efficient but requires draining the descriptor completely on each event. Go's net package uses epoll under the hood for all network I/O; this is what makes 100K concurrent goroutines practical without 100K OS threads.

io_uring — submitting I/O without syscalls

io_uring (Linux 5.1+) uses two shared ring buffers between kernel and userspace: a submission queue (SQE) and a completion queue (CQE). You write I/O requests to the SQE in userspace, call io_uring_enter once to submit a batch (or skip the syscall entirely with IORING_SETUP_SQPOLL), and read completions from the CQE without any syscall. This eliminates per-operation syscall overhead — critical for storage-intensive services doing millions of small I/Os per second.

04

CPU Cache Hierarchy and Its Impact on Go Code

Modern CPUs have three cache levels: L1 (32–64KB per core, ~4 cycles), L2 (256KB–1MB per core, ~12 cycles), L3 (4–32MB shared, ~40 cycles), RAM (~200 cycles). The difference between an L1 hit and an L1 miss that goes to RAM is 50×. Writing cache-friendly code is one of the highest-leverage performance optimizations available.

Spatial and temporal locality

Spatial locality: access memory sequentially. CPU prefetchers anticipate sequential access patterns and load the next cache line before you ask for it. Iterating over a slice of structs sequentially is cache-friendly. Iterating via random pointer chasing (linked list traversal, map iteration) is not.

False sharing: when two goroutines write to different variables that happen to share a cache line (64 bytes on x86), the cache coherency protocol forces a line flush and reload on every write. Pad hot shared variables to cache line boundaries to eliminate false sharing: type Counter struct { val int64; _ [56]byte }.

05

Flash Cards

What happens on a TLB miss?
The MMU performs a hardware page table walk — 4 memory accesses on x86-64. If the page isn't in RAM (page fault), the OS loads it from swap, updates the page table, and retries. Cost: ~100ns for a TLB miss hitting RAM, milliseconds for a page fault hitting disk.
epoll edge-triggered vs level-triggered?
Level-triggered (default): epoll_wait returns as long as the fd is ready. Edge-triggered (EPOLLET): returns only when state transitions from not-ready to ready. ET requires draining the fd completely each time but is more efficient for high-throughput servers.
What is false sharing and how do you fix it?
Two goroutines writing different variables that share a cache line (64 bytes) cause cache invalidation on every write — the coherency protocol forces flushes between cores. Fix: pad the struct to align each hot variable to its own cache line: _ [56]byte after a int64 field.
What does io_uring eliminate that epoll doesn't?
Per-operation syscall overhead. With epoll, every read()/write() is still a syscall. io_uring batches I/O requests in a shared ring buffer — one io_uring_enter submits a batch. With IORING_SETUP_SQPOLL, the kernel polls the SQE, eliminating syscalls entirely for high-throughput I/O.
