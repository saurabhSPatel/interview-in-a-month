01Key Concepts

Linux Namespaces

  • PID: container has its own PID 1 (init), isolated process tree
  • Network: own network stack, interfaces, routing table
  • Mount: own filesystem view, pivot_root
  • UTS: own hostname and domainname
  • IPC: isolated shared memory, semaphores
  • User: root in container ≠ root on host (user ns)

cgroups v2

  • Resource limits: CPU, memory, disk I/O, network (tc)
  • CPU limit: cpu.max = 50000 100000 = 50% of one core
  • Memory limit: memory.max → OOM kill when exceeded
  • cgroup v2 unified hierarchy (vs v1 per-subsystem)
  • Kubernetes: requests = CPU shares, limits = cgroup max

OverlayFS

  • Layers: lower (read-only image layers) + upper (container rw)
  • Read: check upper first, fallback to lower layers
  • Write (CoW): copy file from lower to upper on first write
  • Delete: whiteout file created in upper layer
  • Shared layers: multiple containers share read-only layers

Docker Networking Modes

  • Bridge: default, docker0 bridge, NAT
  • Host: share host network namespace (best performance)
  • None: no network, full isolation
  • Overlay: multi-host (Docker Swarm / Kubernetes flannel)
  • Macvlan: container gets own MAC, direct L2 access

Container Runtime Hierarchy

  • Docker → containerd → runc (OCI runtime)
  • containerd: manages image pull, start, stop, networking
  • runc: creates namespaces, cgroups, calls pivot_root
  • K8s bypasses Docker: kubelet → containerd (since K8s 1.24)
  • Alternatives: podman (rootless, daemonless), cri-o

Container Security

  • Rootless containers: user namespace maps container root to unprivileged UID
  • Seccomp: syscall whitelist (Docker default blocks ~44 syscalls)
  • AppArmor/SELinux: MAC policy on file access, network
  • Capabilities: drop ALL, add only needed (NET_BIND_SERVICE)
  • Read-only rootfs: --read-only + tmpfs for /tmp

02Must-Know Deep Dives

🔥 Multi-Stage Build for Go — Production Pattern

# Multi-stage: build binary in builder, copy to minimal runtime FROM golang:1.22-alpine AS builder WORKDIR /app # Layer cache: copy go.mod/go.sum first (changes rarely) COPY go.mod go.sum ./ RUN go mod download # Cached unless deps change COPY . . RUN CGO_ENABLED=0 GOOS=linux go build \ -ldflags="-w -s" \ # Strip debug info (-60% binary size) -trimpath \ # Remove local path info -o /binary ./cmd/server # Runtime: scratch or distroless (no shell = smaller attack surface) FROM gcr.io/distroless/static-debian12 COPY --from=builder /binary /binary COPY --chown=nonroot:nonroot --from=builder /app/config /config USER nonroot # Never run as root EXPOSE 8080 ENTRYPOINT ["/binary"] # Final image size: ~8MB (vs 300MB with golang:alpine)

Your Go microservices should use this pattern. CGO_ENABLED=0 produces a static binary with no libc dependency — can run in scratch or distroless. The multi-stage pattern is what made your Hyperledger Fabric images deployable on GKE.

🔥 BuildKit — Cache Mounts for Faster CI

# Enable BuildKit DOCKER_BUILDKIT=1 docker build . # BuildKit cache mount: persist Go module cache across builds RUN --mount=type=cache,target=/root/go/pkg/mod \ --mount=type=cache,target=/root/.cache/go-build \ go build -o /binary ./cmd/server # BuildKit secret: inject secrets without baking into layers RUN --mount=type=secret,id=npm_token \ NPM_TOKEN=$(cat /run/secrets/npm_token) npm install # Parallel build stages (BuildKit builds independent stages in parallel) FROM base AS stage1 RUN ... FROM base AS stage2 # Runs in parallel with stage1 RUN ...

Cache mounts alone cut your Go build time from 3 minutes to 20 seconds on repeated builds. Module cache persists across Docker layer changes. Your Jenkins/Helm pipeline should use BuildKit with cache mounts.

04Quick Revision

DOCKER FLASH CARDS

Container vs VM?
Container: shared kernel, isolated via namespaces+cgroups. VM: separate kernel, hardware virtualization. Container: <10ms start, <10MB overhead. VM: 30s start, 100MB+ overhead.
OverlayFS CoW?
Read: check upper layer (rw) first, then lower layers (ro). Write: copy file from lower to upper on first modification. Multiple containers share read-only lower layers → efficient.
Rootless container?
User namespace maps container UID 0 → host unprivileged UID (e.g. 100000). Root in container can't do root things on host. Podman: rootless by default.
Best Go image strategy?
Multi-stage: build in golang:alpine, copy binary to distroless/static. CGO_ENABLED=0 for static binary. Final image ~5-10MB. No shell = reduced attack surface.
BuildKit cache mount?
--mount=type=cache,target=/root/go/pkg/mod persists Go module cache between builds. Layer invalidation no longer evicts module cache. 10x faster repeated builds.
cgroup CPU limit vs request?
Requests = cpu.shares (relative weight, proportional). Limits = cpu.cfs_quota_us/cpu.cfs_period_us (hard throttle). CPU throttling kills p99 latency even when average CPU is low.
