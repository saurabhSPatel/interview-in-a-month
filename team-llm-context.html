Team LLM
Context Engineering

Most teams are using LLMs the same way they used Stack Overflow in 2012 — each person alone, rebuilding context from scratch each session. This module covers what it actually looks like to treat LLM context as shared team infrastructure: versioned, observable, and consistent across Claude, GPT-4o, Gemini, and Copilot simultaneously.

4 hours
Engineering practice
Team-level, not prompt-level
01

The Real Problem — It's Not Which Model You Use

Teams typically frame the multi-LLM problem as "which model should we standardise on?" That's the wrong question. The real problem is that LLM context is not a first-class engineering artifact in most teams — it's ephemeral, siloed, and non-reproducible.

Think about what a new engineer has to do on their first day with an LLM: explain the entire system architecture, the team's conventions, the business domain, the current sprint context, and the relevant constraints — every single session. A senior engineer who has been on the project for two years does exactly the same thing. The LLM knows nothing persistent about your system.

Meanwhile, every engineer on the team is having parallel conversations with different models, getting subtly different architectural recommendations, using subtly different vocabulary to describe the same things, and none of it is captured anywhere. This is context fragmentation — and it compounds over time.

🔁
Context Rebuild Tax
Every new session starts from zero. A 5-minute re-explanation of system architecture, repeated 10 times a week, across 8 engineers, is 400+ minutes of lost time per month.
~400 min/month per team
🌀
Recommendation Drift
Engineer A asks Claude to design the notification system. Engineer B asks GPT-4o the same question. They get structurally different answers with different vocabulary. Now there's a design disagreement that's actually an LLM inconsistency.
Silent, hard to detect
🧩
Tribal Knowledge Lock-In
The engineer who wrote the auth service has the best LLM conversations about it — and all that context lives in their browser history. When they leave, the context goes with them. Not committed anywhere.
Lost on offboarding
⏱️
Onboarding Cliff
New engineers spend weeks asking LLMs questions that existing engineers have already asked and answered, in sessions that were never captured. The team's accumulated LLM knowledge is zero from the new hire's perspective.
2–4 weeks slower ramp
The reframe

LLM context is not a personal tool. It's a team infrastructure problem — the same class of problem as shared config, shared schema, shared runbooks. The solution is the same: version control it, make it collaborative, and make it reproducible.

02

The .agent Folder — Context as Code

The .agent/ folder is a versioned, committed directory in your repository that serves as the shared context layer for all LLM interactions on the project. It's LLM-agnostic — the same files load into Claude, GPT-4o, Gemini, and Copilot. It lives in git, so it has history, ownership, code review, and diffs like any other code.

The structure has two layers: stable context (architecture, conventions, decisions — changes rarely, reviewed carefully) and live context (current sprint, open questions, recent decisions — changes frequently, lower friction to update).

.agent/
│
├── CONTEXT.md      ← load this first. 1-page system summary.
│
├── stable/          ← changes require PR + review
│   ├── architecture.md ← services, data flows, dependencies
│   ├── decisions.md    ← ADRs: what was decided and why
│   ├── conventions.md  ← naming, patterns, code style
│   └── constraints.md  ← SLAs, compliance, non-negotiables
│
├── live/            ← commit directly, no PR needed
│   ├── sprint.md       ← current sprint goals + in-flight work
│   ├── open-questions.md← unresolved design questions
│   └── recent.md       ← last 2 weeks: decisions + learnings
│
└── prompts/         ← reusable prompt templates
    ├── review.md       ← code review prompt
    ├── design.md       ← system design session prompt
    └── debug.md        ← debugging session prompt

What goes in CONTEXT.md

CONTEXT.md is the first file any LLM gets. It's deliberately short — 400–600 words. It answers: what does this system do, who uses it, what's the tech stack, what are the three most important architectural decisions already made, and what does the LLM need to know to not suggest something we've already ruled out. It's the system prompt you'd write if you were hiring a contractor for a week.

markdown — CONTEXT.md template
# [Service Name] — LLM Context
## What this is
[2 sentences. What the service does and who calls it.]

## Stack
- Language: Go 1.23
- Transport: gRPC (internal) + REST (public)
- DB: PostgreSQL 15 (primary) + Redis (cache)
- Messaging: Kafka (Confluent Cloud)
- Deploy: Kubernetes on GKE, ArgoCD

## Scale
- ~50K RPS at peak, 200M requests/day
- p99 latency SLA: 150ms
- Data: 2TB primary DB, 180-day retention

## Decisions already made — do not re-suggest alternatives
1. We use Kafka for all async, not RabbitMQ. Migration done in Q1 2024.
2. We do not use ORMs. All queries are raw SQL via pgx.
3. Auth is handled by the gateway — services trust the forwarded claims.
4. We use ULID for all IDs, not UUID or auto-increment.

## Current constraints
- SOC 2 Type II: no PII in logs, all PII fields encrypted at rest
- No external HTTP calls from the hot path (latency requirement)
- Go version upgrades require platform team approval

## Team conventions — see .agent/stable/conventions.md for full list
- Error wrapping: fmt.Errorf("context: %w", err) — always wrap
- Context propagation: first argument on every function
- Tests: table-driven, no mocks for DB (use testcontainers)

What goes in decisions.md

Every significant architectural decision gets an entry in decisions.md in ADR (Architecture Decision Record) format. The format is: context (what situation forced this decision), options considered, decision made, reasoning, and consequences. The LLM doesn't need to re-derive these — but it does need to know they exist so it doesn't suggest you reverse them.

markdown — ADR entry in decisions.md
## ADR-007: Fan-out strategy for notification delivery
Date: 2024-11-12 | Status: Accepted | Author: @saurabh

### Context
Notification service needs to deliver to 50+ channel types
(push, email, SMS, Slack, webhook). Original design had a monolithic
delivery function. At 10K alerts/sec this became a bottleneck.

### Options considered
A. Single topic, channel-specific consumers (current state)
B. One Kafka topic per channel type
C. Single topic + in-consumer routing by channel

### Decision
Option B: one topic per channel type.

### Reasoning
Independent scaling per channel. Email delivery is slower than push
(SMTP vs APNs latency). Separate topics prevent email backpressure
from delaying push. Consumer group lag per channel type becomes
independently observable in Grafana.

### Consequences
+ Each channel scales independently
+ Cleaner consumer lag monitoring
- More topics to manage (currently 52 channel topics)
- Channel registry must stay in sync with topic list (see topic-registry.go)
03

The Session Context Protocol

The .agent folder is only useful if engineers actually load it at the start of every LLM session. That means the loading process must be near-zero friction. Here's the protocol that makes it habitual.

The three-file session start

Every LLM session for project work starts with three files: CONTEXT.md (the base), live/sprint.md (current state), and the most relevant stable file for today's task (architecture.md for design work, conventions.md for code review, decisions.md for planning). Three files, loaded at the top of the conversation, before any actual question. This is the "context handshake."

shell — session start script (save as bin/llm-start)
#!/bin/bash
# Usage: llm-start [context-type]
# context-type: design | code | debug | review
# Outputs the context block to copy into your LLM session

TYPE="${1:-code}"

echo "=== SESSION CONTEXT — $(date +%Y-%m-%d) ==="
echo ""
cat .agent/CONTEXT.md
echo ""
echo "--- CURRENT SPRINT ---"
cat .agent/live/sprint.md
echo ""

case "$TYPE" in
  design)  cat .agent/stable/architecture.md
           cat .agent/stable/decisions.md ;;
  code)    cat .agent/stable/conventions.md ;;
  debug)   cat .agent/stable/architecture.md
           cat .agent/live/recent.md ;;
  review)  cat .agent/stable/conventions.md
           cat .agent/prompts/review.md ;;
esac

echo ""
echo "=== END CONTEXT — ready for your question ==="

# Pipe to pbcopy (macOS) or xclip (Linux) for instant clipboard
# bin/llm-start design | pbcopy

IDE integration — VS Code and Cursor

For engineers using Cursor or VS Code with Continue/Copilot, automate context injection via workspace settings. The .cursorrules file in the repo root is loaded automatically by Cursor as the system-level context for every AI interaction in that workspace.

text — .cursorrules (auto-loaded by Cursor)
# This file is automatically loaded by Cursor as project-level context.
# Keep it under 800 words. Link to .agent/ files for detail.

You are working on [Service Name], a Go microservice handling
notification delivery at 50K RPS. Full context: .agent/CONTEXT.md

Before suggesting code:
1. Check .agent/stable/conventions.md for naming and patterns
2. Check .agent/stable/decisions.md before suggesting new dependencies
3. All DB queries use pgx, never an ORM
4. All functions take context.Context as first argument
5. Error wrapping: fmt.Errorf("functionName: %w", err)

When reviewing code:
1. Flag any PII in log statements (SOC 2 violation)
2. Flag any missing context propagation
3. Flag any error paths that don't wrap with context
4. Use .agent/prompts/review.md for the full review checklist
GitHub Copilot

Copilot reads .github/copilot-instructions.md automatically in supported editors. Add a short version of your CONTEXT.md there. It's different from .cursorrules but the same principle — inject stable project context at the tool level, not the session level.

04

Multi-Model Routing — Right Model for the Right Task

Standardising on a single model is the wrong instinct. Different models have genuinely different strengths, and using the same model for code review, creative brainstorming, formal reasoning, and quick lookups is like using the same tool for every job. The goal is a clear routing decision, not uniformity.

Task Type Best Model (2026) Why Context files needed
System design + architecture Claude Sonnet / Opus Long context (200K tokens), structured reasoning, nuanced trade-offs CONTEXT.md + architecture.md + decisions.md
Code generation (Go, Rust) Claude Sonnet / GPT-4o Both excellent. Claude is stronger on memory safety reasoning; GPT-4o on speed CONTEXT.md + conventions.md
Code review + security audit Claude Sonnet Best at finding subtle issues, race conditions, nil traps; reads full diff CONTEXT.md + conventions.md + prompts/review.md
In-editor autocomplete GitHub Copilot / Cursor Tab Low latency, file-aware context, no session management needed .cursorrules (auto-loaded)
Formal math / proofs / logic o3 / Gemini Ultra Extended reasoning chains, better on symbolic tasks Minimal — math is domain-agnostic
Document drafting / RFCs Claude Sonnet Best structured prose with technical precision, handles long output CONTEXT.md + decisions.md
Quick lookup / syntax Any / Perplexity Speed matters. Don't load context for "what's the flag for X command" None

The routing rule of thumb

Load context for anything that touches project-specific knowledge. Skip context (and often skip LLMs entirely) for anything that's general knowledge — language syntax, library docs, standard algorithms. The context overhead is worth it when the task requires knowing your system's specific constraints; it's waste when the answer doesn't depend on your system at all.

05

Preventing Context Drift

Context drift is when the .agent files fall out of sync with the actual system. An architecture.md that describes what the system looked like six months ago is worse than no architecture.md — it actively misleads both new engineers and the LLM. Preventing drift requires treating context updates as part of the definition of done, not as documentation afterthought.

The context update checklist — part of every PR

Add this to your PR template. The goal is zero extra effort for small decisions and very low friction for large ones.

markdown — .github/pull_request_template.md addition
## Context Updates (.agent/)
- [ ] No architectural changes — no update needed
- [ ] Updated decisions.md with new ADR (if a significant decision was made)
- [ ] Updated conventions.md (if a new pattern was established)
- [ ] Updated architecture.md (if a service dependency changed)
- [ ] Updated live/sprint.md (automatically, not required per PR)

## LLM session notes worth capturing
# Did an LLM session surface anything important for the team?
# Key decisions, tradeoffs explored, alternatives ruled out?
# If yes: add a brief entry to .agent/live/recent.md

Automated drift detection

For teams with stricter requirements, a lightweight CI check can flag when certain file patterns change but the corresponding .agent file was not updated. This isn't meant to be a hard gate — it's a reminder.

yaml — .github/workflows/context-check.yml
name: Context Freshness Check
on:
  pull_request:
    paths:
      - '**/*.go'
      - '**/schema/*.sql'
      - '**/proto/**'

jobs:
  context-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 2 }

      - name: Check if context needs updating
        run: |
          # If proto files changed, architecture likely changed
          PROTO_CHANGED=$(git diff --name-only HEAD~1 | grep "\.proto$" | wc -l)
          ARCH_CHANGED=$(git diff --name-only HEAD~1 | grep ".agent/stable/architecture" | wc -l)

          if [ "$PROTO_CHANGED" -gt "0" ] && [ "$ARCH_CHANGED" -eq "0" ]; then
            echo "::warning::Proto files changed — consider updating .agent/stable/architecture.md"
          fi

          # If schema changed, decisions.md likely needs an ADR
          SQL_CHANGED=$(git diff --name-only HEAD~1 | grep "schema/" | wc -l)
          DEC_CHANGED=$(git diff --name-only HEAD~1 | grep ".agent/stable/decisions" | wc -l)

          if [ "$SQL_CHANGED" -gt "3" ] && [ "$DEC_CHANGED" -eq "0" ]; then
            echo "::warning::Schema changes detected — was a migration decision documented in decisions.md?"
          fi
06

Team Workflows — Concrete Processes

Abstract principles don't change team behaviour. Concrete, low-friction processes do. These are the three workflows that matter most.

Workflow 1 — The weekly context sync (15 minutes)

01
Each engineer pastes their most useful LLM exchange of the week
Not the full conversation — the key insight or decision that came out of it. 3–5 sentences. The question that worked well and the answer that was actually right.
02
Team decides what gets committed to .agent/
Most things go into live/recent.md. Significant decisions go into stable/decisions.md as ADRs. Patterns that should be adopted broadly go into stable/conventions.md. Anything that was wrong in the existing files gets corrected.
03
One person commits the updates
Rotating responsibility. The commit goes to main directly — these are context files, not code. No PR required for the live/ directory.

Workflow 2 — Onboarding with the .agent folder

Day 1 for a new engineer: clone the repo, read .agent/CONTEXT.md, then spend 30 minutes asking an LLM questions about the system using that file as context. Every question they ask surfaces gaps in CONTEXT.md that should be filled. This is the most valuable first-day use of the folder — new engineers ask the questions that existing engineers stopped asking.

Workflow 3 — The design session transcript pattern

When an engineer does a major design session with an LLM — say, designing a new subsystem — they capture the key exchanges in a session log file. Not the full transcript, but the structured output: what alternatives were explored, what was rejected and why, what constraints shaped the final design. This becomes an ADR in decisions.md and the context for the next engineer who touches that subsystem.

markdown — session log template (.agent/live/sessions/)
# Session: [Topic] — [Date]
## Who: @engineer | Model: Claude Sonnet 4.6 | Duration: ~45min

## Question posed
How should we handle backpressure in the notification fan-out when
a downstream channel (e.g. email) is slow?

## Alternatives explored
1. Single Kafka topic, all channels — rejected: email backlog blocks push
2. In-memory queue per channel — rejected: data loss on crash
3. Per-channel Kafka topics — selected (see ADR-007)

## Key insight
LLM surfaced that Kafka consumer group lag per topic gives us independent
observability — email lag visible separately from push lag. This became
the deciding factor, not just isolation.

## What to read next
.agent/stable/decisions.md ADR-007
cmd/notification/kafka/topic-registry.go

Failure Modes — What Goes Wrong and Why

FAIL 01 Context bloat CONTEXT.md becomes a 10,000-word document

Teams over-document. Every decision, every convention, every nuance goes into CONTEXT.md. The file crosses 5,000 tokens. LLMs start to lose context from the early sections because it's buried in middle-of-context (mid-context attention degradation is real). Engineers stop maintaining it because it's too long to keep accurate.

Fix: CONTEXT.md stays under 600 words, always. Everything else goes in the specific stable/ files that are only loaded when relevant. The discipline of keeping CONTEXT.md short forces you to prioritise what actually matters for baseline orientation.

FAIL 02 Stale context The .agent files describe last year's system

Architecture changes. The auth migration happened, the monolith was split, the database was changed — but decisions.md still describes the old world. An LLM reading it confidently gives advice that's technically correct for the old system and wrong for the current one. This is worse than no context at all because it's confidently wrong.

Fix: Make context updates part of the definition of done — same as tests. The PR checklist addition from the previous section enforces this. Assign a "context owner" who reviews .agent/ files monthly for accuracy.

FAIL 03 Over-trust Treating LLM output as ground truth

Even with perfect context, LLMs make subtle mistakes. They'll confidently suggest a database schema that has an N+1 query problem, or a concurrency pattern that has a race condition, or an API design that has a pagination issue. The context doesn't prevent this — it just makes the mistakes better-informed.

Fix: LLM output is a first draft, not a final answer. Code review exists for a reason. The output should go through the same review process as code written by a junior engineer — which means the LLM is most valuable for acceleration, not for authority.

FAIL 04 Zero adoption The .agent folder exists but no one loads it

The folder gets created with great intentions. Two weeks later, half the team forgot it exists and the other half finds the loading step too much friction. It falls into disrepair, stops being updated, and the team reverts to session-by-session context rebuilding.

Fix: The llm-start script and .cursorrules remove the friction for engineers who use those tools. For the rest: make loading the context the first step in any LLM task, same as running git pull before starting work. Make it a habit, not a policy.

The One-Line Summary
Context is infrastructure. Treat it like code.
Version it in git. Review significant changes. Keep it short and accurate. Load it consistently. Update it when the system changes. Assign ownership. The team that does this well doesn't just use LLMs more efficiently — they accumulate knowledge that compounds, onboard engineers faster, and make decisions that are consistent across engineers and models.

Flash Cards

What is the fundamental problem with per-session LLM context?
It's ephemeral and siloed. Every engineer rebuilds context from scratch every session. Institutional knowledge from good LLM conversations is never captured. New engineers have zero accumulated team knowledge from the LLM's perspective. The "context tax" compounds as the team grows.
What is the .agent folder and why does it live in git?
A versioned directory of markdown files that serve as shared LLM context for the project. Lives in git so it has history, ownership, code review, and diffs. LLM-agnostic — the same files load into Claude, GPT-4o, Gemini. Structured into stable/ (changes rarely, needs PR) and live/ (changes often, commit directly).
What goes in CONTEXT.md and how long should it be?
The 1-page system primer: what the service does, tech stack, scale, decisions already made (that the LLM should not re-suggest), and current constraints. Under 600 words — always. If it grows beyond this, move content to specific stable/ files loaded only when relevant. Length discipline is the hardest part to maintain.
What is context drift and how do you prevent it?
When .agent files describe an older version of the system than currently exists. Prevents this by: adding a context update checklist to the PR template, automated CI warnings when proto/schema changes without corresponding .agent updates, assigning a "context owner," and treating context updates as part of the definition of done.
Which model for architecture design vs in-editor autocomplete — and why different?
Architecture design: Claude Sonnet/Opus — 200K context window handles full .agent folder, better at nuanced trade-offs and structured reasoning. In-editor autocomplete: Copilot/Cursor Tab — low latency matters most, file-aware context loaded automatically, no session management overhead. Wrong to standardise on one model across both tasks.
What is the session log pattern and what does it capture?
A brief markdown file (not a full transcript) capturing: the question posed, alternatives the LLM explored and rejected, the key insight that shaped the final decision, and where to read more. Goes in .agent/live/sessions/. Significant ones get promoted to ADRs in decisions.md. Ensures design thinking is preserved, not just the final answer.
What is the weekly context sync and what does it take to run it?
15 minutes. Each engineer shares their most useful LLM exchange of the week. Team decides what gets committed — recent.md for live context, decisions.md for significant decisions, conventions.md for new patterns. One person commits on a rotating basis. No PR required for live/. Compounds into a knowledge base over weeks.
What is the biggest LLM team adoption failure mode?
The .agent folder exists but no one loads it — because loading is too much friction. Fix: llm-start script (one command), .cursorrules file (auto-loaded by Cursor), .github/copilot-instructions.md (auto-loaded by Copilot). Remove the loading step from requiring deliberate action. When context loading is automatic, it gets loaded.
