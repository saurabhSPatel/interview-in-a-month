01

Why Kubernetes Exists

Before Kubernetes, deploying containers meant: SSH into a server, run docker run, SSH back in when it crashes, restart manually. At one service this is manageable. At 50 services across 100 servers it is operational chaos. Kubernetes solves the container orchestration problem: given a desired state (3 replicas of this container, always, on healthy nodes), maintain it automatically in the face of node failures, rolling deployments, and resource contention.

The key mental model: you declare what you want, Kubernetes figures out how to make it happen and keep it that way. The reconciliation loop is the heart of this — every controller in K8s observes actual state, compares to desired state, and acts to reduce the diff. This loop runs continuously and never stops.

Without K8s vs With K8s
WITHOUT Kubernetes                    WITH Kubernetes

Server A  Server B  Server C          Desired State (YAML)
+--------+ +--------+ +--------+         replicas: 3
| svc1   | | svc1   | | svc2   |         image: myapp:v2
| (ok)   | |(DEAD)  | | svc3   |              |
+--------+ +--------+ +--------+              v
     ^          ^                       Kubernetes Control Plane
  manual     manual                    +---------------------+
  restart    restart                   | Observe -> Diff     |
                                       | -> Act -> Reconcile |
Problems:                              +---------------------+
  No auto-restart                            |
  No health checks                     Ensures 3 replicas on
  No load balancing                    healthy nodes, always
  No rolling updates
  No bin-packing

etcd — The Only Source of Truth

etcd is a distributed key-value store using Raft consensus. It is the only component with persistent state. Every API object is stored here. The kube-apiserver is the only component that talks to etcd directly — all others go through the apiserver. If etcd loses quorum, the cluster cannot make state changes.

etcd Raft — 3-node cluster, quorum = 2
etcd Cluster (3 nodes, quorum = 2, tolerates 1 failure)

+----------+   heartbeat/vote   +----------+
|  etcd-1  |<------------------>|  etcd-2  |
|  LEADER  |                   | FOLLOWER |
+-----+----+                   +----------+
      |
      |  Client Write: PUT /registry/pods/default/nginx
      |  1. Leader appends to its log
      |  2. Replicates entry to followers (parallel)
      |  3. Quorum (2/3) acknowledge -> commit
      |  4. Returns success to kube-apiserver
      |
+-----+----+
|  etcd-3  |
| FOLLOWER |
+----------+

Node count  | Quorum | Tolerated failures
    3       |   2    |   1
    5       |   3    |   2
    7       |   4    |   3
    EVEN    |  N/A   |  NEVER USE EVEN NUMBERS

Deployment Rolling Update — Zero Downtime

Rolling update sequence — maxSurge:1 maxUnavailable:0
strategy: RollingUpdate
  maxSurge: 1        (at most 1 extra pod during update)
  maxUnavailable: 0  (always keep desired count available)

Step 0: Initial    RS-v1: [v1][v1][v1]           3 pods total

Step 1: New RS     RS-v1: [v1][v1][v1]
                   RS-v2: [v2]                    4 pods (surge)
                   v2 must PASS readiness probe first

Step 2: Scale down RS-v1: [v1][v1]
                   RS-v2: [v2][v2]                4 pods

Step 3: Continue   RS-v1: [v1]
                   RS-v2: [v2][v2][v2]            4 pods

Step 4: Done       RS-v1: []   (kept empty for rollback)
                   RS-v2: [v2][v2][v2]            3 pods

kubectl rollout undo deployment/app  -> scales RS-v1 back up
kubectl rollout status deployment/app -> watch progress

StatefulSet vs Deployment

StatefulSet stable identity — for databases and brokers
StatefulSet: mysql (replicas: 3)

Pod names are DETERMINISTIC:
  mysql-0   mysql-1   mysql-2
  (ordinal, always these names, created in ORDER)

DNS names are STABLE (survive restarts):
  mysql-0.mysql.default.svc.cluster.local -> 10.244.1.5
  mysql-1.mysql.default.svc.cluster.local -> 10.244.2.8
  mysql-2.mysql.default.svc.cluster.local -> 10.244.3.2

PVCs follow the pod (volumeClaimTemplates):
  data-mysql-0 -> EBS vol-abc  (stays bound to mysql-0)
  data-mysql-1 -> EBS vol-def  (stays bound to mysql-1)
  data-mysql-2 -> EBS vol-ghi  (stays bound to mysql-2)

Startup: mysql-0 Running -> mysql-1 starts -> mysql-2 starts
Shutdown: mysql-2 terminates -> mysql-1 -> mysql-0 (reverse)

Use for: PostgreSQL, MySQL, Cassandra, Kafka, ZooKeeper, Elasticsearch

NetworkPolicy — Zero-Trust Between Pods

yaml — default-deny + targeted allow
# Default deny ALL traffic in namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}          # all pods
  policyTypes: [Ingress, Egress]   # no rules = deny all
---
# Allow API pods to receive from ingress-nginx only
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-ingress
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes: [Ingress]
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: ingress-nginx
      podSelector:
        matchLabels:
          app.kubernetes.io/name: ingress-nginx
    ports:
    - port: 8080
---
# Allow API pods egress to DB + DNS only
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-egress
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes: [Egress]
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: postgres
    ports:
    - port: 5432
  - to:   # always allow DNS
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
    ports:
    - port: 53
      protocol: UDP

Access modes:
  RWO  ReadWriteOnce    -- one node (EBS, local disk)
  ROX  ReadOnlyMany     -- many nodes read-only
  RWX  ReadWriteMany    -- many nodes read+write (NFS, EFS)
  RWOP ReadWriteOncePod -- exactly one pod

emptyDir: ephemeral, deleted when pod dies. Scheduling — How Pods Land on Nodes

Scheduler pipeline — Filter → Score → Bind
New Pod (phase: Pending) enters scheduler queue
     |
     v
FILTER PHASE  (eliminate ineligible nodes)
+----------------------------------------------------------+
|  NodeResourcesFit    enough CPU/memory headroom?         |
|  NodeAffinity        node labels match required terms?   |
|  TaintToleration     pod tolerates all node taints?      |
|  NodeUnschedulable   node cordoned by admin?             |
|  VolumeBinding       PVCs can bind on this node?         |
|  PodTopologySpread   spreading constraints satisfied?    |
|  InterPodAffinity    co-location rules met?              |
|                                                          |
|  Result: {node1, node3, node7}  (feasible set)          |
+----------------------------------------------------------+
     |
     v
SCORE PHASE  (rank feasible nodes 0-100)
+----------------------------------------------------------+
|  LeastAllocated      prefer nodes with more free CPU/mem  |
|  BalancedAllocation  balance CPU vs memory utilisation    |
|  ImageLocality       prefer nodes with image already cached|
|  NodeAffinity        bonus score for preferred affinity   |
|  InterPodAffinity    bonus for preferred co-location      |
|  TaintToleration     bonus for preferred tolerations      |
|                                                           |
|  Result: node3 wins (score: 94/100)                      |
+----------------------------------------------------------+
     |
     v
BIND: write pod.spec.nodeName = "node3" to etcd
  node3 kubelet watches -> pulls image -> starts containers
yaml — taints, tolerations, affinity, topology spread
# Taint GPU node: only GPU workloads land here
# kubectl taint nodes gpu-node-1 gpu=true:NoSchedule

spec:
  tolerations:
  - key: "gpu"
    operator: Equal
    value: "true"
    effect: NoSchedule          # tolerate the taint

  affinity:
    nodeAffinity:
      # HARD: must match (pod won't schedule if fails)
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/arch
            operator: In
            values: [amd64]
      # SOFT: prefer but not required
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        preference:
          matchExpressions:
          - key: node.kubernetes.io/instance-type
            operator: In
            values: [p3.8xlarge, p4d.24xlarge]

    podAntiAffinity:             # spread across zones
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchLabels: {app: myapp}
        topologyKey: topology.kubernetes.io/zone

  # PodTopologySpreadConstraints (better than anti-affinity for spreading)
  topologySpreadConstraints:
  - maxSkew: 1                   # max imbalance between zones
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule
    labelSelector:
      matchLabels: {app: myapp}
QoS classes — determines eviction order under pressure
Guaranteed  (last to be evicted)
  requests.cpu == limits.cpu AND requests.memory == limits.memory
  Exact amount reserved. CPU throttled at limit. OOMKilled at limit.

Burstable   (second to be evicted)
  requests < limits (or only requests set)
  Can burst above requests when capacity available.
  Throttled/OOMKilled at limits.

BestEffort  (first to be evicted -- NEVER use in production)
  No requests or limits set.
  Can consume any available capacity.
  First evicted when node is under memory pressure.

Node memory pressure eviction order:
  BestEffort -> Burstable (most over-request first) -> Guaranteed

Security — RBAC, Pod Security, Supply Chain

RBAC object model
Subject                  Binding                  Role / ClusterRole
+------------------+     +-----------------+      +----------------------+
| User             |     | subjects:       |      | rules:               |
| Group            |---->|   name: dev-svc |----->| - apiGroups: [""]    |
| ServiceAccount   |     | roleRef:        |      |   resources: [pods]  |
+------------------+     |   kind: Role    |      |   verbs: [get, list] |
                         |   name: pod-rdr |      +----------------------+
                         +-----------------+      (namespaced = Role)
                                                  (cluster-wide = ClusterRole)

Binding types:
  RoleBinding        binds Role to subject IN ONE NAMESPACE
  ClusterRoleBinding binds ClusterRole to subject CLUSTER-WIDE

Evaluation logic (PURE ALLOW logic -- no deny rules in RBAC):
  1. Check all applicable RoleBindings + ClusterRoleBindings
  2. Any binding grants the permission? -> ALLOW
  3. No match? -> DENY (implicit -- default)
  Note: There is no explicit Deny in RBAC. Use NetworkPolicy for that.

Common mistake: using ClusterRoleBinding for namespace-scoped work
  Better: RoleBinding + ClusterRole (bind cluster role to one namespace)

Operators and Custom Resources

An Operator encodes operational knowledge in code. It extends Kubernetes with custom resource types (CRDs) and custom controllers that watch those resources and act on them. A PostgreSQL Operator watches for PostgresCluster objects and handles provisioning, failover, backups, and upgrades automatically.

Operator pattern — CRD + controller reconciliation loop
User applies:
  apiVersion: postgres-operator.crunchydata.com/v1beta1
  kind: PostgresCluster
  metadata: {name: prod-db}
  spec:
    instances: [{replicas: 3, dataVolumeClaimSpec: {10Gi}}]
    backups: {pgbackrest: {repos: [{schedules: {full: "0 1 * * 0"}}]}}
          |
          | stored in etcd as custom resource
          v
PostgreSQL Operator Controller (pod in cluster)
+--------------------------------------------------+
|  Watch loop (uses controller-runtime library):   |
|                                                  |
|  1. Receive event: PostgresCluster created       |
|  2. Read spec (3 replicas, 10Gi storage, backup) |
|  3. Observe actual: no StatefulSet, no Services  |
|  4. Calculate diff -> create resources:          |
|     - StatefulSet: postgres-primary (1 pod)      |
|     - StatefulSet: postgres-replica (2 pods)     |
|     - Service: postgres-primary (read/write)     |
|     - Service: postgres-replica (read-only)      |
|     - PVCs: 3 × 10Gi EBS volumes                |
|     - Secret: postgres credentials (generated)  |
|     - CronJob: weekly full backup                |
|  5. Patroni manages leader election              |
|  6. On failure: auto-failover, promote replica   |
+--------------------------------------------------+
          |
          v
Running: primary (write) + 2 replicas (read) + scheduled backups
Status written back to PostgresCluster.status (observable)

Writing a Simple Operator in Go

go — minimal operator reconcile loop (controller-runtime)
// controller-runtime reconciler pattern
type MyAppReconciler struct {
  client.Client
  Scheme *runtime.Scheme
}

// Reconcile is called whenever a MyApp object changes
// (or on error retry, or periodic requeue)
func (r *MyAppReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
  log := log.FromContext(ctx)

  // 1. Fetch the custom resource
  var app myv1.MyApp
  if err := r.Get(ctx, req.NamespacedName, &app); err != nil {
    return ctrl.Result{}, client.IgnoreNotFound(err)
  }

  // 2. Build the desired Deployment
  desired := &appsv1.Deployment{
    ObjectMeta: metav1.ObjectMeta{
      Name:      app.Name,
      Namespace: app.Namespace,
    },
    Spec: buildDeploymentSpec(app.Spec),
  }
  // Set ownership: deleting MyApp cascades to Deployment
  ctrl.SetControllerReference(&app, desired, r.Scheme)

  // 3. Create or Update (server-side apply)
  var existing appsv1.Deployment
  if err := r.Get(ctx, req.NamespacedName, &existing); err != nil {
    if apierrors.IsNotFound(err) {
      return ctrl.Result{}, r.Create(ctx, desired)
    }
    return ctrl.Result{}, err
  }
  existing.Spec = desired.Spec
  return ctrl.Result{}, r.Update(ctx, &existing)
}

// SetupWithManager: watch MyApp + owned Deployments
func (r *MyAppReconciler) SetupWithManager(mgr ctrl.Manager) error {
  return ctrl.NewControllerManagedBy(mgr).
    For(&myv1.MyApp{}).
    Owns(&appsv1.Deployment{}).  // requeue if owned Deployment changes
    Complete(r)
}

HPA and KEDA — Autoscaling

HPA control loop + KEDA event-driven scaling
HPA Controller (runs every 15 seconds)
  1. Query metrics-server: current CPU = 80% across 3 pods
  2. Calculate desired replicas:
     desired = ceil(current * (current_metric / target_metric))
     desired = ceil(3 * 80/50) = ceil(4.8) = 5 pods
  3. Scale Deployment to 5 (within minReplicas/maxReplicas)
  4. Scale-up cooldown: 0s (immediate)
     Scale-down cooldown: 5 min (prevents thrashing)

KEDA (Kubernetes Event-Driven Autoscaler):
  Scale on: Kafka consumer lag, SQS queue depth,
            Prometheus query, Datadog metric, HTTP RPS
  Scale to ZERO when no messages (saves cost for batch workloads)

  Example ScaledObject:
    triggers:
    - type: kafka
      metadata:
        topic: orders
        consumerGroup: order-processor
        lagThreshold: "100"     # scale up if lag > 100 messages
  -> KEDA scales Deployment from 0 to N based on lag depth

Flash Cards

What is etcd and which component is the only one that talks to it?
etcd is the only persistent state store in Kubernetes — all API objects are stored here using Raft consensus. Only the kube-apiserver talks to etcd directly. All other components (controllers, kubelet, scheduler) interact through the apiserver. Loss of etcd quorum makes the cluster read-only at best.
Liveness vs readiness vs startup probe — what does each failure do?
Startup: disables liveness/readiness until it passes — for slow-starting apps. Liveness failure: restarts the container — use for deadlock detection. Readiness failure: removes pod from Service endpoints, does NOT restart — use for "not ready to serve" (warming cache, waiting on DB). Same endpoint for both is a common mistake.
Deployment vs StatefulSet — when do you use each?
Deployment: stateless workloads, interchangeable pods, random names, no stable storage. StatefulSet: stable pod names (mysql-0, mysql-1), stable DNS per pod (mysql-0.mysql.ns.svc...), PVCs that follow the pod, ordered startup/shutdown. Use StatefulSet for databases, Kafka, ZooKeeper — anything needing stable identity.
What are the four K8s networking requirements? How does DNS resolution work?
Every pod gets unique IP; pods reach each other without NAT; nodes reach pods without NAT; pod's self-IP equals what others use to reach it. DNS: CoreDNS resolves service names to ClusterIP. kube-proxy programs iptables/ipvs rules to DNAT ClusterIP to actual pod IPs. Full name: service.namespace.svc.cluster.local.
How does the scheduler choose a node for a pod? Two phases.
Filter: eliminate nodes that can't run the pod (insufficient resources, taint not tolerated, affinity not matched, PVC can't bind). Score: rank remaining 0-100 (prefer least-loaded, image cached, preferred affinity). Bind: write pod.spec.nodeName to etcd. kubelet on that node watches and starts the pod.
RBAC: how does evaluation work? Can you deny access in RBAC?
RBAC is pure-allow: any matching binding grants the permission; no match = implicit deny. There is NO explicit deny in RBAC — you can't write a rule that blocks. To restrict access, don't create the binding. For real network-level deny between pods, use NetworkPolicy. Common mistake: ClusterRoleBinding for namespace-scoped work — use RoleBinding instead.
What is an Operator and how does it differ from a Deployment?
A Deployment runs generic stateless containers and handles basic lifecycle. An Operator = CRD (custom resource type) + custom controller (reconciliation loop). It watches custom resources and manages the full operational lifecycle — provisioning, failover, backups, upgrades, scaling — encoding domain-specific knowledge that a human operator would otherwise perform manually.
Resource requests vs limits — what does each control?
Requests: used by the scheduler for node selection (node must have enough unreserved capacity). Limits: enforced by the runtime. CPU: throttled at limit (never killed). Memory: OOMKilled if container exceeds limit and restarts. QoS: Guaranteed (req=limit), Burstable (req<limit), BestEffort (none). BestEffort is evicted first under memory pressure.
