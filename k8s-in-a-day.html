01

etcd — The Ground Truth of Your Cluster

etcd is the source of truth for all Kubernetes state. Every resource — pods, services, configmaps, secrets, deployments — lives as a key-value entry in etcd. Every component in the control plane watches etcd for changes and reacts to them. Understanding etcd is understanding why Kubernetes behaves the way it does.

etcd uses the Raft consensus algorithm to maintain a consistent, linearizable log across its cluster (typically 3 or 5 nodes). Writes are committed only when a majority of nodes (quorum) acknowledges them. Reads from the leader are linearizable by default — every read sees the most recent committed write. Reads from followers may see stale data unless you use --consistency=l.

Why this matters for production: an etcd cluster that loses quorum (more than half its nodes are unavailable) stops accepting writes — your control plane freezes. No new pods start, no deployments progress, no HPA scaling. The cluster doesn't crash, it just stops responding to changes. This is why etcd nodes should be spread across availability zones.

02

The Kube Scheduler — How Pods Get Placed

The scheduler is a control loop that watches for unscheduled pods (pods with no nodeName) and assigns them to nodes based on a two-phase process: filtering and scoring.

Filtering (predicates): eliminates nodes that can't run the pod. Checks: available CPU/memory, node selector labels, affinity/anti-affinity rules, taints and tolerations, pod topology constraints, volume availability. The result is a set of feasible nodes.

Scoring (priorities): ranks feasible nodes. Factors: resource utilisation balance (spread workloads, don't stack on one node), inter-pod affinity score (place pods close to others they communicate with), image locality (prefer nodes that already have the required image). The highest-scoring node wins.

Custom schedulers and scheduling profiles: you can run multiple schedulers with different configuration profiles, or write a scheduler extender (webhook that the default scheduler calls for custom filtering/scoring). Used for GPU scheduling, topology-aware placement, specialised hardware.

03

CNI Networking — From Pod Creation to Routable IP

When a pod is scheduled, the kubelet invokes the CNI (Container Network Interface) plugin to set up networking. The CNI plugin creates a virtual network interface in the container's network namespace, assigns an IP from the pod CIDR, and sets up routing so the pod can communicate with other pods across nodes.

Flannel (simple overlay): wraps pod traffic in VXLAN packets for cross-node communication. Each node has a subnet (e.g., 10.244.1.0/24). Flannel routes packets to the target node's VXLAN endpoint. Simple but has overhead from encapsulation.

Calico (BGP-based): uses Border Gateway Protocol to advertise pod routes between nodes. No encapsulation overhead — native routing. Supports NetworkPolicy for fine-grained traffic control. Common in on-premise and bare-metal deployments.

Services and kube-proxy: a Service gets a ClusterIP (virtual IP). kube-proxy programs iptables rules (or IPVS) on every node to intercept traffic to the ClusterIP and DNAT it to one of the endpoint pod IPs. iptables rules are re-generated every time an endpoint changes — at large scale (10K+ services, 100K+ endpoints), this causes latency spikes. eBPF-based alternatives (Cilium) replace iptables with efficient kernel programs.

04

Operators and CRDs — Extending Kubernetes

An operator is a controller that watches for custom resources (defined by CRDs) and takes action to reconcile actual state with desired state. This is the same pattern as built-in controllers (the Deployment controller, StatefulSet controller) — just with your custom resources.

The reconciliation loop: watch for changes → compare current state with desired state → take the minimum action needed to close the gap → update status. This must be idempotent — if the reconcile function is called 10 times in a row with no state changes, nothing bad should happen.

Common operator mistakes: not handling the case where the controlled resource was deleted while the operator was down (always reconcile from scratch). Using Update() instead of Patch() for status updates (causes version conflicts). Not setting owner references (leaked resources if the CR is deleted). Not implementing finalizers for resources that need cleanup before deletion.

05

Flash Cards

What happens to your cluster when etcd loses quorum?
The control plane stops accepting writes. No new pods start, no deployments progress, no HPA scaling, no config changes apply. The cluster continues running existing workloads (data plane is unaffected) but is frozen from a control perspective until quorum is restored.
What are the two phases of Kubernetes scheduling?
Filtering: eliminate nodes that can't run the pod (resource constraints, taints, affinity rules, topology constraints). Scoring: rank feasible nodes by resource balance, affinity, image locality. The highest-scoring feasible node gets the pod.
Why does iptables-based kube-proxy struggle at scale?
iptables rules are re-generated from scratch every time any endpoint changes. With 10K+ services and 100K+ endpoints, this regeneration causes latency spikes and high CPU usage. eBPF-based solutions (Cilium) replace iptables with O(1) per-packet decisions using efficient kernel programs.
What must every operator reconcile function handle?
Idempotency (multiple calls with no state change = no action). Missing resources (the controlled resource may have been deleted while the operator was down). Status updates via Patch not Update (avoid version conflicts). Owner references (prevent resource leaks on CR deletion). Finalizers for resources needing cleanup before deletion.
