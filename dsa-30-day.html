The Pattern-First Philosophy

There are roughly 15 algorithmic patterns that cover 90% of problems you'll see in a FAANG-calibre interview. Random LeetCode grinding works against you — you end up with a memory of 200 specific solutions rather than fluency with 15 patterns. When you see a new problem you haven't memorised, you're stuck. When you know the patterns cold, you recognise the shape of the problem and know where to start.

The daily structure is: read the pattern's core idea, understand why it works, then apply it to 3–4 problems in increasing difficulty. On Day 1 you're not doing Two Sum — you're understanding the two-pointer invariant and why it reduces O(n²) to O(n). Two Sum is then trivially obvious.

go — pattern template: two pointers
// Before writing any code: name the pattern out loud.
// "This is two pointers — opposite ends — because the array is sorted
// and I need a pair. I know moving the left pointer increases the sum,
// moving the right decreases it."
//
// The invariant: at every step, the answer (if it exists) is
// within [left, right]. We shrink toward it deterministically.

func twoSum(nums []int, target int) []int {
    left, right := 0, len(nums)-1
    for left < right {
        switch sum := nums[left] + nums[right]; {
        case sum == target: return []int{left + 1, right + 1}
        case sum < target:  left++
        default:            right--
        }
    }
    return nil
}

Week 1 — Arrays, Sliding Window, Two Pointers, Prefix Sum

Linear patterns. These are the building blocks everything else rests on. The goal by end of week: you can recognise within 30 seconds whether a problem wants two pointers, sliding window, or prefix sum — and you can articulate why.

DAY 01 Two Pointers — Opposite Ends Two Pointers ▼
Core Pattern
Start pointers at both ends of a sorted array, move inward based on comparison. The key insight: because the array is sorted, every decision is deterministic. Moving left increases the sum; moving right decreases it. You never need to backtrack. This converts O(n²) brute-force pair search to O(n).
Why It Matters
The pattern generalises far beyond Two Sum. Trapping Rain Water is two pointers tracking left/right maximums. 3Sum is two pointers inside an outer loop with deduplication. Container With Most Water is pure opposite-ends. Any time you have a sorted array and need a pair satisfying a constraint, start here.
Problems
  • #167 Two Sum II — Sorted ArrayEasy
  • #11 Container With Most WaterMed
  • #15 3SumMed
  • #42 Trapping Rain WaterHard
Go tip
For 3Sum: skip duplicates with for i > 0 && nums[i] == nums[i-1] { i++ }. Pre-sort with sort.Ints(nums). Work in-place — no extra allocation needed.
DAY 02 Two Pointers — Fast/Slow (Floyd's Cycle) Two Pointers ▼
Core Pattern
Fast pointer moves 2×, slow moves 1×. If there's a cycle, they must meet — the fast pointer laps the slow one inside the loop. After meeting, reset one pointer to head; advance both at 1× — they meet at the cycle start. This uses O(1) space where a visited set would use O(n).
Why It Matters
Floyd's algorithm is a classic seniority signal — interviewers use it to check if you know O(1) space tricks. The "find duplicate in array" variant (#287) reframes the array as a linked list with the values as next pointers, then applies Floyd's. Once you see that framing, the solution is immediate.
Problems
  • #141 Linked List CycleEasy
  • #142 Linked List Cycle IIMed
  • #287 Find the Duplicate NumberMed
  • #876 Middle of Linked ListEasy
Go tip
type ListNode struct { Val int; Next *ListNode }. Initialise: slow, fast := head, head. Loop condition: fast != nil && fast.Next != nil. Go nil checks are zero-cost — lean on them.
DAY 03 Sliding Window — Fixed Size Sliding Window ▼
Core Pattern
Window of size k slides right. At each step: subtract the outgoing left element, add the incoming right element. The key discipline: never recompute the aggregate from scratch. Maintain a running sum, count, or frequency map. This is what makes it O(n) instead of O(n·k).
Real-World Connection
Fixed-window rate limiting in API gateways is this exact pattern. The window slides over time; the aggregate is the request count. Understanding the algorithm makes the system design obvious.
Problems
  • #643 Maximum Average Subarray IEasy
  • #2461 Max Sum of Distinct Subarrays Length KMed
  • #438 Find All Anagrams in a StringMed
  • #567 Permutation in StringMed
Go tip
For character frequency: use [26]int array instead of map[byte]int for lowercase-only problems — array comparison is a direct memory compare, not a map iteration. Track a matches int counter to avoid reflect.DeepEqual.
DAY 04 Sliding Window — Variable Size Sliding Window ▼
Core Pattern
Expand right when the condition is satisfied, shrink left when violated. Template: left=0; for right := range s { [add right]; for violated { [remove left]; left++ }; update answer }. This greedy expansion and lazy contraction guarantees O(n) — each element is added and removed at most once.
Problems
  • #3 Longest Substring Without Repeating CharactersMed
  • #76 Minimum Window SubstringHard
  • #424 Longest Repeating Character ReplacementMed
  • #904 Fruit Into BasketsMed
Go tip
For ASCII string iteration: for i := 0; i < len(s); i++ with s[i] is faster than range (avoids rune decoding). max(a,b) is a builtin in Go 1.21+ — use it directly.
DAY 05 Prefix Sum + Hash Map Prefix Sum ▼
Core Pattern
prefix[i] = prefix[i-1] + nums[i-1]. Range sum [l,r] in O(1). The power move: combine with a hash map for "subarray sum equals k" — store prefix → count, answer += map[prefix-k]. This transforms O(n²) subarray enumeration to O(n) with a single pass.
Problems
  • #560 Subarray Sum Equals KMed
  • #525 Contiguous ArrayMed
  • #303 Range Sum Query ImmutableEasy
  • #238 Product of Array Except SelfMed
Go tip
Pre-initialise map with zero: seen := map[int]int{0: 1}. This handles subarrays starting at index 0 without a special case. The seed 0:1 represents the empty prefix before any elements.
DAY 06 Intervals — Merge, Insert, Overlap Intervals ▼
Core Pattern
Sort by start time. Merge condition: curr.start ≤ prev.end. For meeting rooms (min rooms needed): sort starts and ends separately, use two pointers — one for earliest start, one for earliest end. When a meeting starts before the earliest end, you need a new room. This is O(n log n) and avoids a heap entirely.
Problems
  • #56 Merge IntervalsMed
  • #57 Insert IntervalMed
  • #253 Meeting Rooms IIMed
  • #435 Non-Overlapping IntervalsMed
Go tip
sort.Slice(intervals, func(i,j int) bool { return intervals[i][0] < intervals[j][0] }). For heap-based meeting rooms, implement container/heap interface (Len, Less, Swap, Push, Pop on a pointer receiver).
Week 1 Checkpoint
Mock Interview — Arrays

  • Verbalize the pattern name and invariant before writing a single line of code
  • Review: prefix sum + hash map combo — most commonly missed pattern in this week
  • Go code standard: no unnecessary allocations, proper nil handling, handles empty input

Week 2 — Stack, Monotonic Stack, Heap, Deque

Order and priority patterns. The theme this week is maintaining a partial ordering on a stream of elements. Monotonic stack and deque are the most underrated patterns — they convert what looks like O(n²) into O(n) by using the ordering itself as a data structure.

DAY 08 Stack — Classic Applications Stack ▼
Core Pattern
LIFO for matching and nesting. Push opening bracket/token, pop on closing, check the top. For calculators: two stacks (nums + ops) or single stack with precedence rules. The key invariant: whatever you've pushed but not popped is "in progress" — not yet matched or resolved.
Problems
  • #20 Valid ParenthesesEasy
  • #155 Min StackMed
  • #150 Evaluate Reverse Polish NotationMed
  • #84 Largest Rectangle in HistogramHard
Go tip
Use slice as stack: push with stack = append(stack, v), pop with v, stack = stack[len(stack)-1], stack[:len(stack)-1]. This is idiomatic Go — no container package needed.
DAY 09 Monotonic Stack — Next Greater/Smaller Monotonic Stack ▼
Core Pattern
Maintain a stack in strictly increasing or decreasing order. When the current element breaks the ordering, pop elements until the order is restored — and record the answer for each popped element at that moment. Each element is pushed and popped at most once: O(n) total. Always store indices, not values — you'll need the distance.
Problems
  • #739 Daily TemperaturesMed
  • #496 Next Greater Element IEasy
  • #503 Next Greater Element II (circular)Med
  • #907 Sum of Subarray MinimumsMed
Go tip
For circular arrays: iterate 2*n using index i % n. When popping: result[stack[top]] = i - stack[top] for distance-based answers. Store index, not value — you need both to answer distance questions.
DAY 10 Heap / Priority Queue — Top K Heap ▼
Core Pattern
Min-heap of size k for top-k largest. Max-heap of size k for top-k smallest. The key insight: a heap of size k never exceeds k elements — pop when it does, and the top is always the kth largest. O(n log k) beats O(n log n) sorting when k is small. For Merge K Sorted Lists: put the head of each list on the heap, pop the min, push its next.
Problems
  • #215 Kth Largest Element in ArrayMed
  • #347 Top K Frequent ElementsMed
  • #23 Merge K Sorted ListsHard
  • #295 Find Median from Data StreamHard
Go tip
container/heap requires 5 methods. Min-heap: Less(i,j int) bool { return h[i] < h[j] }. Use heap.Init, heap.Push, heap.Pop — not methods directly. Push/Pop must use pointer receiver on the heap type.

Week 3 — Binary Search, Trees, Tries

Hierarchical patterns. Binary search extends far beyond sorted arrays — you can binary search on the answer space of any problem with a monotonic predicate. Trees introduce the most important interview concept: the difference between what a DFS function returns locally (to its parent) vs what it reports globally (to a shared variable).

DAY 15 Binary Search — Beyond Sorted Arrays Binary Search ▼
Core Pattern
Template: lo, hi := 0, n-1; for lo <= hi { mid := lo + (hi-lo)/2 }. For leftmost position: use lo <= hi and set hi = mid-1 even when found. Binary search on answer space: define a canAchieve(x int) bool predicate — if it's monotonic (false...false...true...true), binary search finds the boundary in O(log(range)).
Problems
  • #33 Search in Rotated Sorted ArrayMed
  • #153 Find Minimum in Rotated ArrayMed
  • #875 Koko Eating Bananas (BS on answer)Med
  • #410 Split Array Largest SumHard
Go tip
Always: mid := lo + (hi-lo)/2 — never (lo+hi)/2 which overflows for large indices. For answer-space problems: define the predicate as a named function — clean separation between "what are we searching?" and "what's the binary search structure?"
DAY 16–17 Trees — DFS Patterns (Pre/In/Post-order) Tree DFS ▼
Core Pattern
The key distinction for hard tree problems: what does the DFS function return to its parent (local value, like height) vs what does it update globally (like the maximum path sum seen so far). Diameter and maximum path sum both use this — the function returns the best single-arm path upward, but records the best two-arm path into a global variable before returning.
Problems
  • #543 Diameter of Binary TreeEasy
  • #124 Binary Tree Maximum Path SumHard
  • #98 Validate BSTMed
  • #297 Serialize and Deserialize Binary TreeHard
  • #236 Lowest Common AncestorMed
Go tip
Global state in DFS: use a closure. var maxPath int; var dfs func(*TreeNode) int; dfs = func(node *TreeNode) int { ... }; dfs(root). Closures over local variables stay on the stack — no heap allocation, testable, no package-level mutation.
DAY 19–20 Trie — Prefix Tree Trie ▼
Core Pattern
Node = {children [26]*TrieNode, isEnd bool}. Insert: create nodes for each char. Search: traverse, return false if any node is nil. The power: prefix search is identical to search but stops before checking isEnd. Add a count int field for frequency-based autocomplete ranking.
Real-World Connection
Search autocomplete uses a trie (or radix trie for memory efficiency). IP routing tables in network hardware use longest-prefix-match tries. XOR tries let you find maximum XOR pair in O(32n). Understanding the data structure makes the system design discussions much richer.
Problems
  • #208 Implement TrieMed
  • #211 Add and Search WordsMed
  • #212 Word Search II (Trie + DFS)Hard
  • #421 Maximum XOR of Two NumbersMed
Go tip
Use [26]*TrieNode over map[byte]*TrieNode for lowercase-only — array indexing is 5–10× faster than map lookup. Index: c - 'a'. For wildcard search, recurse on all non-nil children when you see '.'.
Week 3 Checkpoint
Mock Interview — Trees & Search
  • Implement Trie from scratch in under 15 minutes — no reference
  • Solve a Hard tree problem cleanly: Binary Tree Maximum Path Sum or Serialize/Deserialize
  • Binary search on answer space: pick an unseen problem, identify the monotonic predicate before coding
  • BFS vs DFS decision: articulate when you'd choose each without pausing to think

Week 4 — Graphs, DP, Backtracking

The deepest patterns. Dynamic programming has exactly five sub-patterns worth mastering — everything else is a variant of one of them. For graphs, the signal that separates senior from mid-level answers is union-find: knowing when to use it over BFS/DFS and being able to implement path compression from memory.

DAY 22–23 Graph — BFS/DFS + Union Find Graph ▼
Core Pattern
BFS for shortest path (unweighted), DFS for connected components and cycle detection. Union-Find: find(x) with path compression, union(x,y) by rank. The key — Union-Find is O(α(n)) practically O(1) per operation, making it better than BFS/DFS for connectivity queries that are answered incrementally as edges are added.
Problems
  • #200 Number of IslandsMed
  • #133 Clone GraphMed
  • #684 Redundant Connection (Union-Find)Med
  • #417 Pacific Atlantic Water FlowMed
Go tip
Iterative path compression: for parent[x] != x { parent[x] = parent[parent[x]]; x = parent[x] } — path halving, one fewer allocation than full path compression with a slice.
DAY 26–27 Dynamic Programming — 5 Core Patterns DP ▼
The 5 Patterns
1D Linear — Climb Stairs, House Robber. Knapsack — 0/1 vs Unbounded. LCS/LIS — string subsequences, longest increasing subsequence. Interval DP — Burst Balloons, Matrix Chain. State Machine — Stock prices with cooldown or transaction limit. Every DP problem is one of these five. Identify the pattern, not the problem title.
How to approach any DP problem
Write the brute force recursion first — don't skip this step. Identify overlapping subproblems (same arguments called multiple times). Add memoisation. Then convert to bottom-up if needed for space. Space optimisation last: if you only need the previous row, collapse to a 1D array.
Problems
  • #198 House Robber (1D Linear)Med
  • #322 Coin Change (Unbounded Knapsack)Med
  • #1143 Longest Common SubsequenceMed
  • #309 Buy/Sell Stock with Cooldown (State Machine)Med
  • #312 Burst Balloons (Interval DP)Hard
  • #72 Edit DistanceMed
Go tip
2D DP: dp := make([][]int, m+1) then for i := range dp { dp[i] = make([]int, n+1) }. For space optimisation, two rows: prev, curr := make([]int,n+1), make([]int,n+1). Swap with prev, curr = curr, prev each iteration.

The Mock Session Protocol

The gap between "I understand this pattern" and "I can solve this under pressure in 20 minutes" is only closed by doing timed sessions. Here's the exact protocol for every weekly mock:

protocol — weekly mock session
// Step 1 (2 min): Read the problem. Say the pattern name out loud.
// "This is sliding window — variable size — because..."
// If you can't name it, you haven't learned the pattern yet.

// Step 2 (3 min): State the invariant and edge cases before coding.
// What is always true at the start of each iteration?
// What breaks if input is empty? Single element? All duplicates?

// Step 3 (15 min): Write the solution. No hints. Talk through it.
// Think of yourself as explaining to a colleague, not solving alone.

// Step 4 (5 min): Analyse complexity. State time AND space.
// "O(n) time because each element is pushed/popped once."
// "O(k) space for the heap of size k."

// Step 5 (5 min): Test with 2 examples. Walk through manually.
// Edge case first. Then the given example.

// Total budget: 30 minutes for medium, 40 for hard.
// If you go over: note where you got stuck and why.
