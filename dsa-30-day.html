Staff-Level Interview Philosophy

Staff-level DSA at Google L7, Meta E7, Apple ICT6, or OpenAI is not harder problems — it's higher standards on the same problems. You are expected to: name the pattern before touching the keyboard, articulate the invariant, handle every edge case unprompted, derive the complexity correctly, and propose optimisations the interviewer hasn't asked for. A Medium solved perfectly is better than a Hard solved messily.

SIGNAL 1 · PATTERN RECOGNITION
Name it in 60 seconds
Say the pattern name out loud before writing a line. "This is sliding window — variable size — because I need the longest subarray satisfying a constraint and the constraint is monotonic as the window grows." If you can't name it, you don't know it.
SIGNAL 2 · INVARIANT FIRST
State what's always true
Every correct algorithm has an invariant — a condition that holds at the start of every iteration. State it before you code. For two pointers on a sorted array: "at every step, the answer (if it exists) is between left and right." If your invariant is wrong, your code will be wrong.
SIGNAL 3 · OPTIMISE UNPROMPTED
Drive the trade-off conversation
After your first solution: "I can reduce space from O(n) to O(1) by using two pointers instead of a map — want me to do that?" or "There's a follow-up where the array is sorted: that changes the approach entirely." Raising the next question yourself is the Staff signal.

Company calibration: Google values elegant solutions with proof of correctness. Meta values speed and correctness under time pressure. OpenAI values depth — understanding why the algorithm works. Apple values clean, production-quality code. Amazon (leadership principles) wants you to think about scale and edge cases. All of them want you to talk continuously while you solve.

go — staff-level pattern template
// Step 1 (before touching keyboard): name the pattern + state the invariant
// "Two pointers, opposite ends, sorted array."
// "Invariant: answer (if exists) is always in [left, right]."
// "Move left when sum too small, right when too large."

func twoSum(nums []int, target int) []int {
    l, r := 0, len(nums)-1
    for l < r {
        switch s := nums[l] + nums[r]; {
        case s == target: return []int{l + 1, r + 1}
        case s < target:  l++
        default:           r--
        }
    }
    return nil
}
// After: "Time O(n), space O(1). Edge cases: empty array handled by l<r."
// Optimisation offer: "If unsorted, use a hash map: O(n) time, O(n) space."
W1

Week 1 — Arrays, Two Pointers, Sliding Window, Prefix Sum

Linear patterns. Staff-level signal: you can derive the O(n) approach from first principles, not from memory. The interviewer wants to see that you understand why moving a pointer in one direction is always correct — the invariant — not just that you've memorised the pattern exists.

DAY 01 Two Pointers — Opposite Ends Two Pointers ▼
Core Pattern
Works on a sorted array. Maintain left and right pointers. At each step, the decision of which pointer to move is deterministic — no guessing. Because the array is sorted, moving left strictly increases the sum; moving right strictly decreases it. This monotone property is what makes the algorithm provably correct and O(n). You never backtrack.
Staff-Level Depth
The generalisation: any time you have a sorted structure and need to find a pair satisfying a constraint, ask "is the decision of which end to move always deterministic?" If yes, two pointers applies. Trapping Rain Water uses two pointers tracking leftMax and rightMax — the insight is that the water at position i is min(leftMax, rightMax) - height[i].

DAY 02 Two Pointers — Fast/Slow (Floyd's Cycle) Two Pointers ▼
Core Pattern
Fast pointer moves 2×, slow moves 1×. If a cycle exists, fast will lap slow inside it — they must meet. After meeting, reset one to head, advance both 1× — they meet at the cycle entry. The proof: at meeting point, distance from entry to meeting = distance from head to entry (mod cycle length). This O(1) space replaces an O(n) visited set.
Staff-Level Depth
The "find duplicate in array" variant (#287) is a Staff signal: frame the array as a linked list where arr[i] is the next pointer. Values in [1,n] means every node points somewhere valid — forming a functional graph with exactly one cycle. Then apply Floyd's. This pattern of "reframe the problem as a different data structure" is common at Google and OpenAI.

DAY 03 Sliding Window — Fixed Size Sliding Window ▼
Core Pattern
Window of size k slides right. Subtract outgoing element, add incoming — never recompute the aggregate from scratch. The invariant: the window always contains exactly k elements and the aggregate is current. This converts O(n·k) to O(n).

DAY 04 Sliding Window — Variable Size Sliding Window ▼
Core Pattern
Expand right when condition is met, shrink left when violated. Template: left=0; for right := range s { add(right); for violated() { remove(left); left++ }; updateAns() }. Correctness: each element is added once and removed at most once — O(n) total.
Staff-Level Depth
The key question: "is the condition monotone with window size?" If adding an element can only make a valid window invalid (never fix an invalid one), the variable sliding window applies. Non-monotone conditions require a different approach. This reasoning, stated explicitly, is the Staff signal.

DAY 05 Prefix Sum + Hash Map Prefix Sum ▼
Core Pattern
prefix[i] = prefix[i-1] + nums[i-1]. Range sum [l,r] in O(1). Power move: combine with a hash map for "subarray sum equals k" — store prefix → count, answer += map[prefix-k]. Single pass, O(n). Pre-seed: map[0] = 1 handles subarrays starting at index 0.
Staff-Level Depth
The generalisation of prefix sum is "any associative operation over a range can be answered in O(1) after O(n) preprocessing." XOR prefix sums find subarrays with XOR = k. Prefix products answer range product queries. 2D prefix sums answer rectangle sum queries. Knowing these extensions is expected at Staff level.

DAY 07 Hashing — Maps, Sets, Design Hashing ▼
Core Pattern
Hash maps trade space for O(1) average-case lookup. The key discipline: define your hash key precisely — what information must the key capture? For anagram grouping: sorted string. For isomorphic strings: the position of first occurrence. Getting the key design right is the hard part; the rest is mechanical.
Staff-Level Depth
Hash map design questions test depth: implement LRU Cache (doubly linked list + map), LFU Cache (three maps: key→val, key→freq, freq→LRU list). At Staff level you're expected to design the data structure, not just use a built-in one. LFU is a Google L6+ signal problem.

Week 2 — Stack, Monotonic Stack, Heap, Deque

Order and priority patterns. The theme: maintaining a partial ordering on a data stream. Monotonic stack converts what looks like O(n²) into O(n) by using the order property to discard elements that can never be the answer. This is the most underrated pattern cluster in Staff-level interviews.

DAY 08 Stack — Classic Applications Stack ▼
Core Pattern
LIFO for matching and nesting. Push on open, pop on close, check the top for validity. The invariant: everything on the stack is "in progress — not yet resolved." For calculators: two stacks (values + operators) or postfix evaluation with one stack. Largest Rectangle in Histogram is the hardest stack problem — understand it deeply.

DAY 09 Monotonic Stack — Next Greater, Span Monotonic Stack ▼
Core Pattern
Maintain a stack in strictly increasing or decreasing order. When the current element breaks the order, pop and record the answer for each popped element — the current element is their answer. Each element is pushed/popped at most once: O(n) total. Always store indices, not values. The invariant: everything on the stack hasn't found its "next greater/smaller" yet.
Staff-Level Depth
Sum of Subarray Minimums (#907) is the Staff signal. The approach: for each element, find how many subarrays have it as the minimum, using prev smaller and next smaller element arrays (both computed with monotonic stacks). The contribution = element × left_count × right_count. This two-stack technique extends to maximum subarray products and similar problems.

DAY 10 Heap / Priority Queue — Top-K and Streaming Heap ▼
Core Pattern
Min-heap of size k for top-k largest. When heap exceeds k, pop — the top is always the kth largest. O(n log k) beats O(n log n) sort when k ≪ n. For Merge K Sorted Lists: heap of (value, list-index, node-pointer). Pop minimum, push its next node. O(n log k) where n is total elements, k is number of lists.
Staff-Level Depth
Find Median from Data Stream (#295) is the classic Staff problem. Two heaps: max-heap for the lower half, min-heap for the upper half. Invariant: max-heap size = min-heap size ± 1. Median is the top of the larger heap (or average of both tops). Rebalancing after each insert: O(log n). This "two heaps for median" pattern extends to k-th quantile tracking.

DAY 11 Deque — Sliding Window Maximum + Monotonic Queue Deque ▼
Core Pattern
A monotonic deque maintains a decreasing (for max) or increasing (for min) sequence of indices. Add to the back: pop the back while the back is smaller than the new element (it can never be the maximum of any future window). Remove from the front: pop front if it's outside the current window. Each element enters/leaves at most once: O(n) total.

Week 3 — Binary Search, Trees, Graphs, BFS/DFS

Hierarchical patterns. The single most important concept this week: the difference between what a DFS function returns to its parent (a local value, like height) vs what it updates globally (like the maximum path sum). Getting this distinction clear unlocks every hard tree problem.

DAY 15 Binary Search — Answer Space Binary Search ▼
Core Pattern
Template: lo, hi := 0, n-1; for lo <= hi { mid := lo + (hi-lo)/2; if pred(mid) { ans = mid; hi = mid-1 } else { lo = mid+1 } }. Binary search on answer space: define a feasible(x) bool predicate. If it's monotone (false...true), binary search finds the first true. Works on anything with monotone ordering — not just sorted arrays.
Staff-Level Depth
The Staff signal: applying binary search to a problem that doesn't look like a search problem. "What's the minimum speed such that Koko can eat all bananas?" Binary search on speed, check feasibility in O(n). "What's the minimum capacity such that we can ship packages in D days?" Same pattern. At Staff level you derive this independently in under 2 minutes.

DAY 16–17 Tree DFS — Local vs Global, Hard Problems Tree DFS ▼
Core Pattern
The key insight for hard tree problems: the DFS function returns one value to its parent (what's useful going up: height, best path from this node to a leaf) but updates a global variable with a different, better answer (best path passing through this node, using both subtrees). Diameter and Max Path Sum both follow this pattern.
Staff-Level Depth
Binary Tree Cameras (#968) and Recover Binary Search Tree (#99) are Staff-level signals. Cameras requires a 3-state DFS (covered, camera-here, not-covered) with greedy reasoning. Recover BST requires finding the two misplaced nodes in an in-order traversal without extra space — Morris traversal for O(1) space.

DAY 18 Tree BFS — Level Order, Right View, Zigzag Tree BFS ▼
Core Pattern
Use a queue. Process level by level: outer loop = levels, inner loop = len(queue) at the start of the level. The key: snapshot the queue length at the start of each level — nodes added during processing are for the next level. This level-snapshotting trick is used by 80% of BFS tree problems.

Core Pattern

The invariant: every path from root to an end-marked node spells a valid word. XOR trie: 32 children per node (bits 0/1), finds maximum XOR pair in O(32n).

DAY 20–21 Graph — BFS/DFS, Topological Sort, Dijkstra Graph ▼
Core Pattern
BFS = shortest path (unweighted). DFS = connectivity, cycle detection, topological order. Topo sort (Kahn's): in-degree array, queue of zero-in-degree nodes, add neighbours and decrement. Dijkstra: min-heap of (dist, node), relax edges. Template: dist[src]=0; heap push; while heap { pop min; for neighbour: relax }.
Staff-Level Depth
Alien Dictionary (#269) and Minimum Cost to Connect All Points (#1584 — Prim's/Kruskal's) are Staff signals. Alien Dictionary: extract ordering constraints from word pairs, build a DAG, topo sort — if cycle exists, return "". Word Ladder II: BFS to find shortest path length, then DFS to enumerate all shortest paths. Both test graph construction, not just graph traversal.

Week 4 — Union Find, DP, Backtracking

The deepest patterns. Dynamic programming has five sub-patterns — every DP problem is one of them. Union-Find separates Staff from mid-level on graph connectivity: knowing when to use it over BFS/DFS and being able to implement path compression + union by rank from memory in under 5 minutes.

DAY 22 Union-Find — Connectivity, Path Compression Union-Find ▼
Core Pattern
Union-Find answers "are x and y connected?" in near-O(1) with path compression + union by rank. find(x): follow parent pointers to root, compress path on the way back. union(x,y): attach smaller-rank tree to larger-rank root. After both optimisations: O(α(n)) per operation where α is the inverse Ackermann function — effectively O(1).
Staff-Level Depth
The Staff signal: knowing when Union-Find beats BFS/DFS. BFS/DFS is O(V+E) per query. Union-Find is O(α(n)) per query after O(n) preprocessing. Use Union-Find when: edges are added incrementally, you need dynamic connectivity, or you're checking connectivity many times. Number of Provinces can use BFS; Redundant Connection should use Union-Find (detect cycle as you add edges).

DAY 23–25 Dynamic Programming — 5 Core Sub-Patterns DP ▼
The Framework — Apply in Order
1. Write the brute force recursion first. Never skip this. 2. Identify overlapping subproblems — same arguments called repeatedly. 3. Add memoisation (top-down). 4. Convert to bottom-up if space matters. 5. Optimise space: if you only need the previous row, collapse to 1D.
The 5 Sub-Patterns
1D Linear: current state depends on 1–3 previous states. House Robber, Climb Stairs, Decode Ways. Knapsack (0/1 + Unbounded): include/exclude decision for each item. Partition Equal Subset Sum, Coin Change. LCS/LIS family: 2D DP on two sequences or the sequence with itself. Edit Distance, Longest Palindromic Subsequence. Interval DP: dp[i][j] = best answer for subproblem on [i,j]. Burst Balloons, Matrix Chain. State Machine: model states explicitly as DP dimensions. Stock with cooldown/fee/k transactions.

DAY 26–27 Backtracking — Systematic Exhaustive Search Backtracking ▼
Core Pattern
Template: func bt(start int, path []int) { if valid(path) { append to result }; for i := start; i < n; i++ { path = append(path, candidates[i]); bt(i+1, path); path = path[:len(path)-1] } }. The key decision: pass i or i+1 as the start for the recursive call — the former allows repetition, the latter doesn't.
Staff-Level Depth
The Staff signal is pruning — not just writing a correct backtracking solution but making it fast. For Combination Sum: sort first, then break the inner loop when candidates[i] > remaining. For N-Queens: use three boolean arrays (rows, diagonals, anti-diagonals) for O(1) conflict check instead of O(n) linear scan. Articulating the pruning strategy and its effect on the worst-case is expected.

DAY 28 Advanced Graph — Bellman-Ford, Floyd-Warshall, SCC Advanced Graph ▼
Core Pattern
Bellman-Ford: relax all edges V-1 times. O(VE). Detects negative cycles (if any edge still relaxes after V-1 passes). Floyd-Warshall: all-pairs shortest paths. O(V³). dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j]) for all k. Tarjan's SCC: single DFS pass, discovery + low-link values to find strongly connected components.

The Mock Session Protocol

The gap between "I understand this pattern" and "I can solve this under pressure in 20 minutes" is only closed by doing timed sessions. Here's the exact protocol for every weekly mock:

protocol — weekly mock session
// Step 1 (2 min): Read the problem. Say the pattern name out loud.
// "This is sliding window — variable size — because..."
// If you can't name it, you haven't learned the pattern yet.

// Step 2 (3 min): State the invariant and edge cases before coding.
// What is always true at the start of each iteration?
// What breaks if input is empty? Single element? All duplicates?

// Step 3 (15 min): Write the solution. No hints. Talk through it.
// Think of yourself as explaining to a colleague, not solving alone.

// Step 4 (5 min): Analyse complexity. State time AND space.
// "O(n) time because each element is pushed/popped once."
// "O(k) space for the heap of size k."

// Step 5 (5 min): Test with 2 examples. Walk through manually.
// Edge case first. Then the given example.

// Total budget: 30 minutes for medium, 40 for hard.
// If you go over: note where you got stuck and why.
